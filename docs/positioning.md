# 산란 포지셔닝 기준

이 문서는 산란을 기존 국내 CMS와 비교할 때 어떤 기준으로 설명하고, 어떤 기대는 명시적으로 낮춰야 하는지 정리한다.

산란은 그누보드, 라이믹스, 워드프레스처럼 오래 운영된 범용 CMS 생태계를 정면으로 대체하려는 프로젝트가 아니다. 산란의 1차 기준은 저가형 PHP 웹호스팅에서도 배포할 수 있고, 요청 흐름과 모듈 경계를 파일에서 직접 따라갈 수 있는 회원 중심 모듈형 웹 솔루션 베이스다.

## 경쟁 범위

산란이 실제로 비교되는 범위:

- PHP와 MySQL 또는 MySQL 호환 DB로 배포하는 소규모 회원 사이트
- 회원, 관리자, 콘텐츠, 커뮤니티, 퀴즈, 설문, 리액션, 자산 흐름을 같은 코드베이스에서 운영하려는 사이트
- Composer, 상시 worker, 별도 queue daemon, SPA 빌드 파이프라인 없이 운영하고 싶은 환경
- 모듈 경계와 요청 흐름을 명시적 파일 include로 확인하고 싶은 개발자

산란이 현재 정면 경쟁 대상으로 삼지 않는 범위:

- 검증된 대형 커뮤니티 CMS 생태계 전체
- 풍부한 스킨, 플러그인, 제작자 시장이 중요한 사이트
- headless API-first 또는 모바일 앱 연동을 핵심으로 하는 서비스
- 상시 worker, Redis, 전문 queue, 검색 엔진, CDN edge logic을 전제로 하는 고부하 서비스
- 외부 결제, 정산, 실명 인증, 세무/회계 증빙까지 포함한 상거래 플랫폼

## 차별점

산란의 차별점은 기능 수보다 다음 운영 감각에 있다.

- 절차형 PHP와 명시적 `paths.php` 기반 요청 흐름
- 코어가 도메인 정책을 흡수하지 않고 모듈이 자기 테이블과 화면을 소유하는 구조
- 회원 인증도 기본 제공 모듈로 취급하는 경계
- 공유호스팅을 배제하지 않는 배포 보호 기준
- 개인정보 export/cleanup, 읽기 참조, URL 임베드 캐시, 자산 원장처럼 교차 모듈 계약을 문서화하는 습관
- 관리자 화면과 smoke 기준을 기능 구현과 함께 갱신하는 운영 중심 개발 방식

이 차별점은 대형 생태계보다 뛰어나다는 뜻이 아니다. 산란은 생태계 규모와 외부 검증 이력에서는 불리하다. 대신 작은 사이트와 소규모 운영자가 코드를 직접 읽고 고치는 환경에서는 구조의 투명성을 더 중요하게 둔다.

## 낮춰야 할 기대

다음 내용은 README, 릴리스 노트, 운영자 문서에서 과장하지 않는다.

- 선택 번들 모듈이 많다는 사실만으로 운영 성숙도를 주장하지 않는다.
- 자산, 쿠폰, 유료 접근권 흐름은 자동 점검과 smoke 기록 없이 신뢰 대상으로 표현하지 않는다.
- CKEditor HTML 저장과 sanitizer는 보안 검증 대상이지 완성된 보안 보증으로 표현하지 않는다.
- 이메일 큐, 예약 발행, 만료, 배치 작업은 공유호스팅 환경에서 실시간 처리를 보장한다고 설명하지 않는다.
- 1.0 전에는 라이믹스나 그누보드의 생태계, 장기 보안 패치 이력, 대형 운영 사례와 같은 급의 신뢰를 주장하지 않는다.

## 비교 문구 기준

산란을 설명할 때 권장하는 문구:

```text
저가형 PHP 웹호스팅에서도 운영 가능성을 확인하는, 요청 흐름이 파일에 드러나는 회원 중심 모듈형 웹 솔루션 베이스.
```

피해야 할 문구:

```text
그누보드/라이믹스 대체 CMS
한국형 올인원 CMS
포인트, 쿠폰, 커뮤니티, LMS, 개인정보까지 모두 완성된 플랫폼
```

## 개선 우선순위

외부 평가에서 약점으로 보이는 부분은 다음 순서로 줄인다.

1. 검증 증거를 현재 상태 문서로 묶는다.
2. 자산/쿠폰/유료 접근권 흐름의 reconciliation과 멱등성 검증을 강화한다.
3. HTML sanitizer의 허용 범위와 공격 payload 회귀 검사를 공개한다.
4. 공유호스팅 queue/cron 한계를 관리자 화면과 운영 문서에 명확히 노출한다.
5. 모듈 계약 구현 매트릭스와 누락 감지 도구를 유지한다.
