# 1.0 전 구현 스냅샷

이 문서는 1.0 배포 전까지 저장소 안에서 현재 구현의 핵심 표면을 빠르게 확인하기 위한 임시 스냅샷이다. DB 명세, 관리자 화면별 필드 가이드, 온보딩 문서도 현재는 저장소 `docs/`를 기준으로 관리하며, GitHub Wiki를 운영 문서의 정본으로 사용하지 않는다. 기능과 특장점 요약은 [산란 특장점 소개](operator-feature-list.md)를 기준으로 하고, 이 문서는 요청 경로, DB 테이블, 모듈 표면을 대조하는 기술 보조 자료로 사용한다.

## 런타임 기준

| 항목 | 현재 값 |
| --- | --- |
| 본체 버전 | `0.2.0` |
| 모듈 계약 버전 | `2.0` |
| PHP 기준 | `8.1` 이상 |
| DB 기준 | MySQL 또는 MySQL 호환 DB, `pdo_mysql` |
| 요청 흐름 | `index.php`가 `core/request-bootstrap.php`의 요청 준비 함수를 순서대로 호출한 뒤 활성 모듈의 `paths.php`를 읽고 action 파일을 명시적으로 include |
| 기본 DB prefix | `sr_` |

상세 컬럼과 인덱스는 설치 SQL과 저장소 `docs/`의 현재 문서를 정본으로 본다. 이 문서는 1.0 전 구현 표면을 빠르게 확인하기 위한 보조 문서다. 회원가입 동의 증적은 `sr_member_consents`에 policy document key/version/title/body hash snapshot을 함께 저장한다. 회원 환경설정의 가입 약관 동의 섹션에서 이용약관, 개인정보 수집 및 이용 동의, 선택 마케팅 수신 동의에 사용할 policy document를 고르며, 필수 두 문서는 게시 버전이 있어야 저장할 수 있다. 이용약관과 개인정보처리방침 version 작성 화면은 표준 문안 자동 채우기를 제공하고 버튼 옆에 해당 기준의 최근 개정일 뱃지를 표시하고 공식 공고를 새 탭으로 연결한다. 이용약관 문안은 공정거래위원회 전자상거래 표준약관 구조를, 개인정보처리방침 문안은 개인정보보호위원회 작성지침/표준안을 기준으로 하며 사이트 이름, 기준 URL, 상호, 대표자명, 사업자등록번호, 통신판매업 신고번호, 사업장 주소, 고객센터, 개인정보보호책임자, 호스팅 제공자 같은 사이트 설정 사업자 정보를 반영한다. 정책 문서 새 version은 별도 version key 컬럼 없이 DB version ID와 본문 hash를 기준으로 식별하며 운영자 화면에는 내부 식별자를 표시하지 않는다. 정책 문서 새 version은 `append_previous_versions` 옵션으로 저장 본문을 변경하지 않고 공개 렌더링된 약관 내용 끝에 이전 게시/보관 version의 일자 링크만 자동으로 붙일 수 있다. 일자는 `2026년 7월 2일` 형식으로 표시하고 링크는 해당 이전 version 공개 화면으로 연결하며, 현재 약관 내용에는 이전 본문 전문과 버전 키를 붙이지 않는다. 동의 snapshot의 body hash는 실제 렌더링된 약관 내용 기준으로 남긴다. 회원 관리자 수정 화면은 이메일 인증 상태를 저장할 수 있고, 인증 완료로 저장하면 현재 인증 시각이 없는 계정에 완료 시각을 기록한다. 정책 문서 version을 게시 상태로 만들면 `sr_policy_document_mail_jobs`와 활성 회원 기준 delivery가 생성되며, 공유호스팅 친화적으로 관리자 화면에서 batch 단위로 발송한다. 미래 시행일이 있는 게시 version은 안내메일로 먼저 고지하고 해당 시각부터 공개 약관 기준에 적용한다. 같은 문서의 새 안내메일 작업이 생성될 때 기존 미완료 작업은 취소 상태로 닫고, 관리자 변경 안내메일 목록의 작업 상태는 설정 언어의 라벨로 표시한다. 안내메일 batch는 오래 머문 처리 중 대상을 포함한 delivery를 먼저 claim한 뒤 해당 row만 발송하고, 관리자는 실패 delivery를 재대기하거나 남은 발송을 취소할 수 있다. 새 설치 화면의 선택 모듈 전체 선택에는 `member_oauth`, `member_oauth_providers`, `reaction`이 포함되고, 선택 모듈이 다른 선택/기반 모듈을 요구하면 설치기가 필요한 모듈을 재귀적으로 자동 포함한다. `member_oauth`는 state/nonce/PKCE 원문을 DB에 저장하지 않고 hash로 보관하며, callback까지 필요한 PKCE verifier는 세션 임시 저장소에서 1회만 회수한다. provider 계약의 authorization/token/userinfo endpoint와 profile claim 설정을 바탕으로 관리자 저장 client id/secret/scope override를 적용해 SDK 없는 generic OAuth2/OIDC callback을 처리한다. 사용을 켠 provider는 Client ID가 필요하고, 저장된 client secret이 없으면 새 secret 입력도 필수다. scope override는 항목 단위로 저장 입력을 받고 authorization 요청 시 provider별 delimiter로 직렬화한다. provider 계약의 기본 scope는 삭제할 수 없고 저장 시 누락되어도 다시 보존하며, 운영자가 추가한 scope는 저장된 목록이 authorization 요청에 우선 반영된다. 기본 프로필 동기화 rule의 scope 선택값은 `email_scope`, `display_name_scope` 같은 provider별 scope 메타데이터를 우선 따른다. OAuth 로그인/연결 때 provider profile snapshot을 갱신하고, 검증된 이메일과 이름은 회원 기본 필드에, 운영자가 claim path로 매핑한 그 외 값은 정의된 선택 프로필 항목에만 동기화한다. OAuth 선택 프로필 동기화는 매핑된 field key만 부분 갱신하고, provider가 빈 값·배열 값·선택지 밖 값을 보내거나 현재 정의되지 않은 항목이면 기존 저장값을 삭제하지 않고 건너뛴다. 관리자 프로필 동기화 설정은 provider 계약의 기본 claim path를 선택 목록으로 제공하고, 후보에 없는 claim은 직접 입력으로 저장할 수 있다. secret은 저장 후 화면에 원문을 표시하지 않고, mock provider callback, 기존 계정 로그인, 계정 연결/해제, OAuth 신규 가입 completion 동의 gate를 모듈 내부에서 처리한다. 회원 로그인 완료 지점은 비밀번호 로그인, 가입 직후 자동 로그인, OAuth 기존 계정 로그인, OAuth completion 자동 로그인 모두 `sr_member_login_or_start_mfa()`를 거쳐 `sr_member_login()`으로 들어간다. `/login/mfa` GET/POST route는 등록되어 있고, challenge 상태는 `sr_account_id`와 분리된 세션 값으로만 저장된다. 2차 인증 저장소는 TOTP의 `sr_member_mfa_factors`와 `sr_member_mfa_recovery_codes`, 이메일 코드 challenge의 세션 hash가 함께 담당하며, 운영자가 허용한 활성 provider가 있으면 로그인 MFA challenge로 진입한다. `/mypage/security`는 비밀번호가 있는 계정에서 현재 비밀번호를 확인한 뒤 pending TOTP factor를 만들고, 등록용 QR 이미지, 수동 입력 secret, `otpauth://` URI를 1회 flash로 보여주며, 첫 TOTP 코드를 검증해 factor를 active로 전환한다. 활성화 트랜잭션은 백업 코드를 `sr_hmac_hash()` 기반 hash로 저장하고 원문 코드는 1회 flash로 보여준다. `/login/mfa` POST는 이메일 코드, app key 기반 secret-at-rest 암호문을 복호화한 TOTP 코드, 또는 미사용 백업 코드를 검증한 뒤 `sr_member_login()`으로 로그인 세션을 만든다. 개인정보 사본 제공은 factor 상태/표시 메타데이터와 recovery code 상태별 개수만 포함하고 secret ciphertext, fingerprint, code hash는 내보내지 않는다. 탈퇴와 관리자 withdrawn/anonymized 전환은 회원 소유 MFA row를 직접 삭제한다. 계정 보안 화면은 백업 코드 재발급과 2차 인증 해제를 현재 비밀번호 또는 OAuth-only 계정의 현재 TOTP/백업 코드 재확인 뒤 처리하며, 재발급은 기존 미사용 백업 코드를 폐기하고 해제는 활성/pending factor와 남은 미사용 백업 코드를 닫는다.

로그인 MFA 운영자 정책은 `mfa_login_mode`의 `required`, `optional`, `disabled` 세 값으로 저장하며, 새 설치 기본값은 `disabled`다. 새 설치의 허용 후보는 이메일 인증 코드와 인증 앱 OTP지만 정책이 꺼져 있어 로그인 2차 인증 challenge는 시작하지 않는다. `optional`은 회원이 허용 provider factor를 직접 등록했거나 이메일 코드처럼 계정에 바로 적용할 수 있는 provider가 있는 경우에만 challenge를 시작하고, `required`는 허용 provider가 있는 회원에게 로그인 MFA를 강제하며 등록형 factor 해제를 운영자 정책으로 막는다. `required`에서 인증 앱 OTP 같은 등록형 provider가 허용되어 있는데 회원에게 활성 factor가 없으면 로그인 직후와 보호 화면 진입 시 `/mypage/security`로 이동시켜 등록을 완료하게 한다. `disabled`는 로그인 challenge와 신규 등록을 중지한다. 로그인 MFA provider는 `member-mfa-providers.php` 계약으로 확장하며, 기본 회원 모듈은 이메일 코드와 TOTP provider를 제공한다.

`payment_ledger`는 주문, 콘텐츠 접근, 커뮤니티 열람처럼 도메인별 구매 대상 자체를 소유하지 않고 공통 결제 증빙만 저장한다. `sr_payment_records`는 account, subject module/type/id, 결제 상태, 결제 전 payable amount와 실제 settlement amount/currency를 보존하고, `sr_payment_record_items`는 쿠폰 사용, 포인트/적립금/예치금 거래, 외부 PG 승인, 접근권 부여 같은 구성 item을 참조한다. 접근권 item reference에는 신규 raw account id를 넣지 않고, 탈퇴/익명화 cleanup은 과거 reference와 item snapshot 문자열의 정확한 account segment도 익명화하며 저장소 오류를 member cleanup rollback으로 전파한다. 커머스 모듈을 추가할 때도 주문/주문상품/배송/환불 정책은 커머스가 소유하고, 주문 결제가 확정되는 같은 트랜잭션 안에서 `payment_ledger` record와 item만 남긴다. 이를 위해 결제 대상 도메인 모듈은 `payment-ledger-targets.php` 계약을 제공하고, `payment_ledger`는 해당 계약을 소비해 subject module/type을 저장 전에 검증한다. 같은 `dedupe_key`의 재호출은 기존 record id만 반환하고 item을 추가하지 않으며, account/subject/amount/currency 같은 생성 증빙 값이나 비익명 record의 item 묶음이 다르면 실패한다. lifecycle key는 저장 전 형식을 검증해 잘못된 값을 기본값으로 흡수하지 않는다. 도메인 환불이 원장 환불 또는 접근권 회수를 완료하면 실제로 처리된 연결 item만 `reversed`로 표시하고, 모든 reversible item이 닫힌 record는 `refunded`로 전환한다. 콘텐츠 결제 내역(`/admin/content/payments`)은 유료 콘텐츠 열람과 파일 다운로드 결제 단위를 함께 보여 주고, 커뮤니티 결제 내역(`/admin/community/payments`)은 유료 게시글 열람과 첨부 다운로드 결제 단위를 함께 보여 준다. 두 화면은 결제 유형, settlement kind, 자산 allocation, 쿠폰 redemption, 환불 계약 version, legacy 여부와 처리 가능 여부를 표시한다. 콘텐츠 파일 다운로드와 커뮤니티 첨부 다운로드의 관리자 수동 환불은 연결된 자산 원장 거래를 복원하고 접근권 item을 함께 닫는다. 쿠폰 접근권 환불은 쿠폰 사용 item과 접근권 item을 닫되, 별도 자산 환급 없이 자산 거래 item을 대신 닫지 않는다. 동시 생성 unique 충돌은 같은 값이면 기존 record로 흡수하고, 탈퇴/익명화 후 같은 dedupe replay가 늦게 도착해도 account 연결을 복원하지 않는다. `payment_ledger`가 활성화되어 있는데 helper나 테이블이 준비되지 않은 부분 설치 상태라면 콘텐츠/커뮤니티 유료 접근 처리는 결제 기록 없이 성공시키지 않고 실패한다.

`content`, `community`가 활성 상태이면 관리자 모듈 상태 변경에서 `payment_ledger` 비활성화를 차단한다. `community`는 보상 회수 실패 callback과 공통 미회수 큐 때문에 `asset_ledger`도 기반 의존성으로 선언하며, 활성 커뮤니티나 활성 자산 모듈이 있으면 `asset_ledger` 비활성화를 차단한다. `coupon`은 유료 발급본의 자산 차감 참조를 자기 claim log와 issue snapshot에 보존하고, 콘텐츠/커뮤니티 결제 record의 coupon item으로 참조될 수 있지만 자체 결제 subject record를 만들지는 않는다.

현재 포인트, 적립금, 예치금은 각 모듈의 balance/transaction 테이블을 사용하고, 공통 원자적 갱신은 숨김 기반 공식 선택 모듈 `asset_ledger`의 helper로 줄인다. 새 설치에서는 `point`, `reward`, `deposit`, `community` 선택 시 설치기가 `asset_ledger`를 필요한 기반 모듈로 자동 포함하고 함께 설치됨을 안내한다. 기존 설치에서 세 자산 모듈 또는 커뮤니티를 설치하거나 활성화할 때도 `asset_ledger`는 자동 설치/활성화되며, 활성 자산 모듈이나 활성 커뮤니티 모듈이 있는 동안 비활성화할 수 없다. `asset_ledger`는 `/admin/assets/recovery-failures` 공통 보상 미회수 큐와 `sr_asset_recovery_failures`, `sr_asset_recovery_reversal_links`를 소유한다. 이 큐는 정상 지급 로그를 저장하지 않고 실패한 회수 작업만 기록하며, 지급 로그는 각 자산/도메인 모듈의 읽기 전용 화면에 남긴다. 포인트와 적립금은 유효기간 설정이 1 이상이면 새 `grant` 거래에 `expires_at`과 `expires_remaining`을 기록하고, 사용/차감 시 먼저 만료되는 지급분부터 소진하며 각각 `sr_point_expiration_consumptions`, `sr_reward_expiration_consumptions`에 소비 거래와 원 지급 거래의 연결을 남긴다. 포인트 환불은 환불 건마다 참조 원거래 유효기간을 우선할지, 환불 시점부터 유효기간을 다시 계산할지 선택할 수 있고, 사용 거래 환불의 기본값은 소비 매핑의 원 지급 유효기간을 우선한다. 포인트/적립금/예치금 원장 환불은 참조된 차감 원거래 복원에만 사용하며, 참조 없는 환불, 양수 원거래 환불, 원거래 잔여 환불 가능액 초과 환불은 서버에서 거부한다. 포인트의 기한이 지난 지급분은 다음 포인트 거래 전 또는 `.tools/bin/expire-points.php` 운영 스크립트 실행 시 `transaction_type=expire` 음수 거래로 차감되고, 적립금의 기한이 지난 지급분은 다음 적립금 거래 전 또는 회원 적립금 화면 조회 시 같은 방식으로 차감된다. 관리자 수동 조정은 1,000,000 초과 시 별도 승인자 확인을 요구하고, 자산별 1회 10,000,000과 관리자별 일일 10,000,000 절대금액 상한을 서버에서 검증한다. 적립금은 운영자 직접 회수를 `transaction_type=reclaim` 음수 거래로 별도 기록하며, 회수 대상 원거래를 `reference_type=reclaim`, `reference_id=reward_transaction:{id}`로 연결하고 같은 대상의 누적 회수액이 원거래 금액을 넘지 못하게 검증한다. 회수는 거래 내역 화면의 대상 내역 기준으로만 처리하고, 회수 거래 자체는 환불 대상으로 삼지 않는다. 쿠폰은 통합 금액 원장에 합치지 않고 `sr_coupon_definitions`, `sr_coupon_issues`, `sr_coupon_redemptions`로 정의, 발급 상태, 사용 이력을 따로 기록한다. 쿠폰 정의는 `access`, `fixed_discount`, `percent_discount` 유형을 저장하며 정액/정률 할인값은 정의 row에 보존한다. 정액 할인 정의는 현재 관리자 UI에서 원 단위 금액만 입력받고 저장 기본 통화는 KRW이며, 다중 통화별 금액 또는 환율 snapshot 적용은 주문/결제/가격 정책을 소유한 소비 도메인 계약에서 정한다. 운영자는 `/admin/coupons/campaigns`에서 무료 공개 발급 캠페인을 만들고 수정할 수 있으며, 공개 `/coupons` 쿠폰존과 `/coupons?campaign={campaign_key}` 단건 화면은 활성 공개 캠페인의 발급 기간, 재고, 회원당 제한을 검증한 뒤 같은 발급 의도 토큰을 `sr_coupon_claim_logs.dedupe_hash`로 수렴시켜 `sr_coupon_issues` 발급본을 만든다. 발급본은 claim 유형, campaign/log 연결, 명목 가격, 자산 reference, claim snapshot을 보존할 수 있으며, claim log는 자산 차감형 발급의 audit용 asset reference 사본을 `payment_reference_*`가 아니라 `asset_reference_*`에 보존한다. 무료 발급도 `settlement_kind=free` 스냅샷을 남긴다. 쿠폰 모듈은 `url-embed-targets.php`로 `/coupons?campaign=...` 내부 URL 임베드를 제공하며, 본문 임베드는 캠페인 상태와 남은 수량을 서버 값으로 렌더링하고 실제 발급은 같은 쿠폰존 POST로 보낸다. 팝업레이어는 `coupon_claim_campaign_key`로 공개 무료 쿠폰 발급 캠페인 CTA를 연결할 수 있으며, 팝업 닫기 쿠키와 발급 상태를 분리하고 렌더링 시 쿠폰 캠페인 상태와 재고를 서버에서 다시 판정한다. 발급 캠페인 목록은 상태, 발급 유형, 공개 여부, 캠페인·쿠폰 검색어로 필터링한 캠페인 row만 조회하고, 최근 claim log는 `/admin/coupons/campaigns/logs`의 별도 읽기 전용 화면에서 회원, 상태, 발급 표면, 캠페인·쿠폰 검색어로 필터링해 확인한다. 발급 로그가 생긴 캠페인의 key와 연결 쿠폰 변경, 이미 점유된 수보다 낮은 발급 한도 변경은 계속 차단한다. lazy 가용량 계산에서는 이미 제외됐지만 아직 `expired`로 물질화되지 않은 예약 행을 `만료(미정리)`로 표시한다. 현재 런타임 연동은 콘텐츠 유료 열람, 콘텐츠 파일 다운로드, 커뮤니티 유료 게시글 열람, 커뮤니티 첨부 다운로드에서 선택 쿠폰을 먼저 평가한다. `access` 쿠폰이나 전액 할인 쿠폰이 성공하면 자산 차감 없이 접근권을 만들고, `fixed_discount` 또는 `percent_discount`가 일부만 덮으면 같은 작업 트랜잭션에서 쿠폰 사용 row와 남은 결제액의 포인트/적립금/예치금 차감을 함께 확정한다. 콘텐츠와 커뮤니티의 `multi_asset_payment_enabled` 설정은 기본 허용이며, 끈 상태에서는 관리자 유료 열람/다운로드 설정 저장 시 여러 포인트/금액 항목 선택을 거부하고 public 결제 실행에서도 두 개 이상 allocation이 필요한 결제를 실패시킨 뒤 쿠폰 사용 row, 사용 횟수, 자산 차감, payment-unit row를 같은 트랜잭션에서 rollback한다. 콘텐츠 유료 열람/파일 다운로드/완료 버튼 차감과 커뮤니티 글쓰기/댓글/쪽지/유료 열람/첨부 다운로드 차감은 가격·정책 row의 settlement currency 기준 금액을 먼저 산정한 뒤 `member-assets.php`의 `purchase_power`와 `deduction_order`로 자산 단위 차감량을 배분한다. 도메인 차감 로그는 `amount`를 자산 단위로 유지하고 `settlement_amount`, `settlement_currency`, `purchase_power_snapshot_json`, `settlement_kind`, `snapshot_schema_version`, `rounding_policy_version`을 별도 저장하며, 다운로드 조회와 개인정보 export는 저장된 settlement snapshot을 요약한다. 결제 확정 시 콘텐츠/커뮤니티는 같은 작업 단위에서 `payment_ledger` record를 만들고 coupon redemption, 자산 transaction, 도메인 접근 로그, 접근권 부여 item을 연결한다. 소비 모듈은 `coupon-targets.php` capability와 pricing resolver를 제공할 수 있고, 콘텐츠와 커뮤니티 게시글 target은 현재 가격과 기존 접근권 상태를 조회하는 계약을 제공한다. 쿠폰 사용 helper는 같은 dedupe를 먼저 흡수하고, 다른 dedupe라도 이미 접근권이 있으면 사용 row와 사용 횟수를 남기지 않는 `already_entitled` no-op으로 끝낸다. pricing resolver가 성공하면 쿠폰 사용 기록에는 금액, 통화 또는 자산 단위, 정책 요약, 가격 산정 시각, target type/id, 쿠폰 유형, 할인액, 남은 결제액을 감사 스냅샷으로 저장한다. 환전 모듈은 포인트/적립금/예치금 고정 6방향 정책 row마다 출금 기준값과 입금 기준값을 직접 저장하고, 기준값과 반올림으로 계산되는 최소 허용 환전량보다 작은 최소 환전량은 거부하며, 상태·최소/최대 금액·반올림·수수료도 각 방향 정책 row가 소유한다. 환전 UI에서 포인트/적립금/예치금 항목명과 단위는 각 환경설정의 표시명과 단위를 사용한다. 정책 동기화는 기존 방향별 기준값과 조건을 보존하며, 비표준 정책 row는 로그의 정책 참조만 비운 뒤 제거한다. 전역 `exchange_enabled` 설정이 꺼져 있으면 회원 환전 후보, 예상 금액 계산, 실행 helper를 모두 막되 기존 로그 조회와 완료 묶음 정정은 유지한다. 환전 실행은 비율, 반올림, 수수료 스냅샷을 로그에 남기고, 독립 실행 환전에서 deadlock/lock wait timeout 계열 오류를 짧게 재시도하며, 완료 환전 묶음은 `/admin/asset-exchange/logs`의 관리자 정정 action이 `sr_asset_exchange_correct_completed_group()` helper를 호출해 반대 원장 거래와 감사 로그를 남긴다. 콘텐츠/커뮤니티 복합 자산 처리도 상위 작업 단위에서 retryable DB 잠금 오류를 재시도한다. 콘텐츠/커뮤니티 유료 열람과 다운로드에서 선택 자산 조합이 정확 충당에 실패하더라도 활성 환전 정책으로 같은 조합 안의 잔액을 맞출 수 있으면, 확인 모달이 환전 후 결제를 별도로 확인하고 POST에서 환전 실행 후 settlement 계획을 다시 계산한다.

쿠폰 사용기간 정책은 #390 기준으로 `sr_coupon_definitions.validity_policy`, `validity_days`, `valid_from`, `valid_until`과 발급본 `sr_coupon_issues.starts_at`, `expires_at`에 저장한다. 현재 런타임은 최신 스키마를 기준으로 실패하며 쿠폰 정의/지급 컬럼이 없는 부분 마이그레이션 상태를 정상 경로로 폴백하지 않는다. 미래 시작 발급본은 회원 보유 목록에는 표시하지만 실제 사용 후보와 사용 SQL, 회원 요약 usable count에서는 제외한다.

#396 denomination 계약에 따라 자산 잔액은 통화 금액이 아니라 asset-unit 정수다. 통화 의미는 `purchase_power`와 currency-literal에만 있으며, `currency_min_unit`은 decimal exponent가 아닌 정수 settlement minimum unit이다. 사이트 기본 통화 변경은 기존 settlement/refund/payment/tax snapshot을 record-time 통화로 보존하고 소급 변환하지 않는다.

쿠폰 정의 상태는 `active`, `issue_stopped`, `disabled`로 나뉜다. `active`는 신규 지급과 기존 지급분 사용을 모두 허용하고, `issue_stopped`는 신규 지급과 공개 발급 캠페인을 막되 이미 지급된 활성 쿠폰 사용은 허용하며, `disabled`는 신규 지급과 기존 지급분 사용을 모두 막는다. 발급/사용 이력 또는 운영 참조는 상태 변경을 차단하지 않고 참조 현황과 감사 로그로 남긴다. `/admin/coupons/settings`는 지급, 사용, 사용 환불, 발급 환불, 지급 상태 변경, 사용 중지 회수 안내 알림의 사용 여부와 회원 대상 채널을 케이스별로 저장한다. 사용 중지 회수 안내는 내부적으로 `coupon` 모듈의 `issue.definition_disabled` 이벤트를 사용하며, 알림 모듈이 비활성화되었거나 템플릿이 없으면 상태 변경 자체는 실패하지 않는다.

정액/정률 할인 쿠폰의 일부 할인 + 자산 차감 결제는 현재 사용 생성만 지원한다. 쿠폰 사용과 남은 자산 차감을 함께 되돌리는 취소 계약이 생기기 전까지 할인 쿠폰 정의는 환급 가능 정책으로 저장하지 않는다.

쿠폰 환경설정은 공개 `/coupons` 목록 화면과 사이트 메뉴 링크 후보에 표시할 쿠폰존 명칭을 `coupon_zone_label` 모듈 설정으로 저장한다. 값이 비어 있으면 기본 명칭 `쿠폰존`으로 정규화한다.

콘텐츠/커뮤니티 유료 접근에서 선택된 `access` 쿠폰이나 전액 할인 쿠폰이 성공하면 자산 settlement 계산을 섞지 않고 쿠폰 redemption 가격 snapshot과 접근권만 남긴다. 일부 할인 쿠폰은 할인액과 남은 결제액을 redemption snapshot에 남기고, 남은 금액만 자산 settlement snapshot과 원장 거래로 기록한다. 기존 `legacy 1:1 assumed` 또는 `legacy_unknown` 콘텐츠/커뮤니티 차감 로그는 업데이트에서 삭제하고, 이후 조회와 개인정보 export는 신규 settlement snapshot row만 요약한다.

쿠폰 관리자 대상 검색은 검색 결과에 사용처 capability와 현재 가격 또는 가격 조회 불가 사유를 함께 표시해 운영자가 금액성 사용 가능 여부를 저장 전에 확인할 수 있게 한다.

읽기 참조 계약은 `core/helpers/read-references.php`가 공통 로드/정규화를 맡고, 대상별 파일 `coupon-references.php`, `banner-references.php`, `popup-layer-references.php`, `member-group-references.php`, `site-setting-references.php`로 분리한다. 관리자 화면은 공통 참조 현황 버튼/모달로 쿠폰 정의, 배너, 팝업레이어, 회원 그룹, 사이트명 참조 row와 관리자 이동 URL을 표시한다. 쿠폰 정의 상태 변경은 발급/사용 이력과 운영 참조를 차단 조건으로 삼지 않고, 참조 현황과 감사 로그로 운영자가 영향을 확인하게 한다. 배너/팝업레이어 삭제와 enabled 상태 해제, 회원 그룹 enabled 상태 해제, 사이트명 변경처럼 운영 영향이 큰 POST는 최신 참조 row와 계약 오류를 다시 확인하고, 참조나 계약 오류가 있으면 소비 모듈 정책을 자동 수정하지 않고 저장을 중단한다.

`coupon-targets.php`의 `target_type`은 쿠폰, 배너, 팝업레이어가 함께 쓰는 전역 식별자이므로 모듈 간 중복 선언하지 않는다. 쿠폰, 배너, 팝업레이어의 런타임 소비자는 이미 등록된 `target_type`을 덮어쓰지 않고 첫 유효 계약을 유지한다.

모듈 간 본문 연결은 전용 관리 모듈이나 marker가 아니라 사용자가 본문에 단독으로 붙여 넣은 URL을 공개 렌더링 시점에 해석하는 방식으로 처리한다. 공통 helper `core/helpers/url-embed.php`는 standalone URL 또는 URL 텍스트 링크를 감지하고, YouTube, X, Instagram 같은 외부 URL과 활성 모듈의 `url-embed-targets.php` 계약이 해석하는 내부 URL을 안전한 공개 HTML로 바꾼다. 내부 임베드는 `sr-content-embed`, `sr-community-embed`, `sr-quiz-embed`, `sr-survey-embed`, `sr-coupon-embed` 같은 전용 custom tag와 대상 모듈의 전용 `assets/embed.css`를 사용하며, fragment sanitizer는 현재 계약의 `target_module` 네임스페이스 tag/속성을 허용한다. 계약 제공 모듈은 자기 모듈 key와 같은 `target_module`만 선언할 수 있고, 공통 helper는 다른 모듈 target을 선언한 항목을 등록하지 않는다. 임베드 카드는 제목 링크와 별도로 사이트 base URL을 붙인 전체 canonical URL 텍스트 링크를 함께 보여준다. 설문 URL 임베드는 직접 공개 화면과 같은 활성/기간 조건으로 렌더링하고, 로그인·회원 그룹 참여 제한은 링크 대상 설문 화면에서 처리한다. 외부 URL 임베드는 공통 `sr-url-embed` custom tag와 `/assets/url-embed.css`를 사용한다. 공개 fragment cache 파일은 `storage/cache/embeds/{module_key}` 아래에 저장하며, 각 대상 모듈은 자기 관리자 임베드 캐시 화면에서 파일 수, 용량, 미리보기 목록을 조회하고 조건별 정리를 수행한다. 저장 HTML에는 완성 카드 HTML, 전용 marker, hidden 참조 목록을 남기지 않는다. 콘텐츠 CKEditor 본문 이미지는 다운로드 파일 모델과 DB 참조 테이블 없이 콘텐츠 모듈의 로컬 저장 경로 `storage/content/body/{content_id}`가 소유하며, 저장된 HTML에서 살아남은 콘텐츠 프록시 URL만 서버가 보존 대상으로 인정한다.

커뮤니티 게시판은 `sr_community_categories`와 `sr_community_posts.category_id`로 게시판 내부 카테고리를 선택적으로 지원한다. 카테고리 key는 게시판 안에서 유일하며 공개 URL `category` 파라미터로 사용된다. 상태는 `enabled`/`disabled`만 사용하고, 비활성 카테고리는 신규 선택과 공개 필터 노출에서 제외하되 기존 게시글 표시는 텍스트로 보존한다. 게시판 설정 `category_enabled`가 꺼진 경우 공개 목록 필터와 작성/수정 카테고리 선택을 표시하지 않고 POST 값도 저장하지 않는다. `category_required`가 켜진 경우 `category_enabled`도 자동으로 켜지며 공개 작성/수정 POST에서 서버가 카테고리 선택을 강제한다. 커뮤니티 환경설정은 새 게시판의 게시글 본문 최소/최대 길이 기본값을 제공하고, 게시판 설정은 같은 항목을 개별 게시판에서 재정의한다. 게시글 본문 저장 컬럼은 `MEDIUMTEXT`이며, 입력 수집은 DB 저장 안전 바이트 한도 안에서만 기술적으로 거부하고 실제 글자수 정책은 운영 설정의 최소/최대 길이 검증으로 적용한다. 관리자 게시글/댓글 목록의 게시글 확인 링크는 `preview=admin`으로 열려 조회수 증가와 유료 열람 처리를 건너뛴다. 게시판별 본인확인은 `identity_verification_enabled`, `identity_verification_purpose`, `identity_verification_required_actions`, `identity_verification_max_age_days` 설정으로 저장한다. `identity_verification_required_actions`는 `enter`, `read`, `write`, `comment`, `download`를 선택할 수 있고, 공개 게시판 접근, 글 읽기, 글쓰기, 댓글 쓰기, 첨부 다운로드 action에서 서버가 각각 본인확인 통과 여부를 검사한다. 성인 본인확인 목적은 본인확인 환경설정의 생년월일 사용이 켜져 있을 때만 저장할 수 있다. 커뮤니티는 provider 원문 응답이나 민감 식별값을 저장하지 않고 `identity_verification` helper 판정만 소비한다. 콘텐츠, 퀴즈, 설문은 각 환경설정에서 공개 열람/참여 본인확인과 성인 본인확인을 전역 접근 정책으로 저장할 수 있고, 관리자 미리보기를 제외한 공개 접근에서 이를 검사한다. 게시글, 댓글, 첨부 업로드 개인정보 수집 및 이용동의는 게시판/그룹/전역 설정의 `privacy_consent_document_key`가 가리키는 policy document published version을 서버에서 다시 조회하고, `sr_community_submission_consents`에 policy document key/version/title/body hash snapshot을 저장한다.

커뮤니티 게시글 묶음 feed cache helper는 `커뮤니티 피드 노출`이 켜진 everyone-discoverable 공개 게시판 baseline을 다룬다. 게시판에서 이 설정을 끄면 해당 게시판의 글, 댓글, 시리즈는 커뮤니티 메인, 사이드바, 최근글/인기글 같은 게시판 목록 밖 피드 후보와 feed cache 입력 board set에서 제외되지만 게시판 목록과 게시글 직접 접근 정책은 바뀌지 않는다. 게시글 테이블은 `summary_feed_candidate`를 가진다. 신규 글은 작성 시점의 커뮤니티 피드 노출 설정을 복사하고, 관리자 게시판 설정에서 커뮤니티 피드 노출 값을 바꾸면 해당 게시판의 기존 글 후보값을 일괄 동기화한다. 최신글/인기글 live query는 게시판 설정 테이블 join 대신 이 게시글 후보값을 우선 사용한다. `context_hash` 입력에는 viewer tier/class 차원을 넣지 않고, 회원 전용/그룹별 discoverable 후보는 후속 viewer-class cache 모델이 필요할 때 별도 구현한다. 커뮤니티 피드 노출 설정 기준이 반영된 cache policy version을 사용해 이전 기준으로 만들어진 snapshot을 재사용하지 않는다. 홈 최신글/인기글 조회와 측정 하니스는 같은 feed query helper를 사용해 fixture/EXPLAIN 증거와 운영 쿼리가 어긋나지 않게 한다. 일반 피드는 변경 이벤트로 갱신하고, 인기글처럼 조회수 기반 자연 변동이 있는 피드는 코드에 고정한 feed key 정책으로 주기 재계산한다. 로그인 사용자의 홈 화면에 회원 전용 게시판이 섞여 있어도 public baseline 파일 캐시는 별도로 예열한다. `/admin/community/feed-cache` 점검 화면은 파일 영속 저장소 상태, 현재 유효한 feed context, 코드 갱신 정책을 read-only로 보여 주며, 갱신 대기 파일은 요약 숫자로만 표시한다. `storage/cache/community-feed`는 피드 snapshot JSON 파일 저장소다. 카드 snapshot helper는 `author_account_id`, 공개 댓글 수, 공개 excerpt, 썸네일 source marker를 저장하고 작성자 label은 렌더 시 resolve한다. `author_label_snapshot`, `author_public_name_snapshot`, 렌더된 thumbnail URL, 본문 전체, CSRF token, 계정별 권한 결과, 유료 접근권 상태를 snapshot JSON에 넣지 않는다. 공개 baseline cache hit는 후보 재선정과 게시글 row hydrate를 건너뛰고 snapshot으로 카드 row를 구성하며, 익명/public 최신댓글도 같은 파일 snapshot으로 재사용한다. 기존 DB 피드 캐시 테이블은 사용하지 않고 업데이트에서 제거한다.

콘텐츠와 커뮤니티 시리즈는 각 모듈이 소유한다. 콘텐츠는 `sr_content_series`와 `sr_content_series_items`, 커뮤니티는 `sr_community_series`와 `sr_community_series_items`를 사용한다. 시리즈 상태는 운영 대기/노출/숨김/보관/삭제를 구분하되 공개 렌더링은 `active` 상태와 `active` 항목만 사용한다. 공개 범위는 `public`, `member`, `private`이며, 커뮤니티 private 시리즈는 소유자만 볼 수 있고 콘텐츠 private 시리즈는 공개 출력에서 제외한다. 콘텐츠 시리즈는 `/admin/content/series`에서 만들고 콘텐츠 편집 화면에서 회차로 연결한다. 콘텐츠 그룹은 목록 페이지와 메뉴 후보를 위한 운영 묶음이고, 콘텐츠 시리즈는 회차 표시와 이전/다음 이동을 위한 읽기 흐름이다. 한 콘텐츠는 콘텐츠 그룹에 속하면서 동시에 하나의 시리즈 회차로 연결될 수 있으며 두 정렬 기준은 서로 영향을 주지 않는다. 커뮤니티 시리즈는 회원이 `/community/series`에서 만들거나 글 작성/수정 중 새로 만들 수 있고, 관리자는 `/admin/community/series`에서 상태와 공개 범위를 조정한다. 공개 커뮤니티 글은 본문 시작 전에 현재 글이 포함된 시리즈 목록을 렌더링하고, 콘텐츠는 본문 다음에 시리즈 내비게이션을 렌더링한다. 커뮤니티 시리즈 상세는 `/community/series?id={series_id}`에서 열람 가능한 시리즈 글만 모아 보여준다. 커뮤니티 시리즈 스크랩은 `sr_community_series_scraps`에 게시글 스크랩과 분리해 저장하며, `/community/scraps`에서 게시글 스크랩과 별도 섹션으로 표시하고 시리즈 상세로 연결한다. hidden, archived, deleted 상태가 되거나 게시판 읽기 권한이 사라진 시리즈는 목록에서 열람 불가 항목으로 남기되 해제할 수 있다.

운영 항목 복사는 콘텐츠, 커뮤니티 게시판, 배너, 팝업레이어 모듈이 각자 소유한다. 콘텐츠 복사는 기본적으로 시리즈 연결을 복사하지 않지만, 시리즈 복사 옵션을 선택하면 원본 시리즈에 사본 항목을 섞지 않고 새 시리즈 사본과 새 항목을 만든다. 커뮤니티 게시판 복사는 관리자 화면에서 `전체`, `설정`, `게시글+댓글`, `첨부파일`, `시리즈` 복사 범위를 체크박스 토글로 선택한다. 새 게시판 key 기본값은 원본 key에 `_copy`를 붙이고, 이미 존재하거나 아직 게시판 생성 전인 복사 job이 예약한 key와 겹치면 `_2`, `_3`처럼 다음 후보로 자동 보정한다. 설정만 선택하면 동기 action으로 복사본 게시판을 만들고, 게시글+댓글을 포함하면 `sr_community_board_copy_jobs`와 `sr_community_board_copy_job_maps`를 사용하는 `/admin/community/board-copy-jobs` 작업 화면에서 prepare, board, posts, comments, attachments, series, verify, complete, cleanup 총 9개 정의 stage 중 현재 단계를 페이지 타이틀과 작업 상세에 표시하며 버튼 반복으로 처리한다. 복사 생성 화면의 validation 실패는 같은 복사 화면으로 돌아와 입력값과 오류를 다시 보여주고, 게시판 목록과 복사 생성 화면은 `/admin/community/board-copy-jobs` 작업 목록 링크를 제공한다. 작업 목록은 진행 중, 실패, 정리 필요 상태를 완료 작업보다 먼저 보여주고 상태를 관리자 상태 뱃지로 표시한다. `/admin/community/board-copy-jobs`는 관리자 메뉴에서 커뮤니티 게시판 관리 항목을 활성 상태로 표시하며, `/admin/operations`의 게시판 복사 지연/실패 신호는 이 작업 화면으로 이동하는 링크를 제공한다. 첨부파일과 시리즈는 복사 범위에서 선택한 경우에만 대상 map과 stage를 처리한다. 게시판 복사 부하 등급은 선택한 범위의 게시글, 댓글, 첨부, 시리즈 항목, 첨부 총량 기준으로 산정하며, 게시글 1개처럼 작은 복사는 `낮음`으로 표시한다. 배치 복사 대상 게시판은 완료 후에도 `disabled`로 남는다.

회원 콘텐츠 소유권은 해시가 아니라 내부 계정 ID로 저장한다. 커뮤니티 게시글/댓글 작성자와 커뮤니티 시리즈 소유자는 편집, 삭제, private 시리즈 열람 같은 권한 판정에 직접 쓰이므로 탈퇴 cleanup에서 null로 바꾸지 않고 회원 계정 행 자체를 익명화한다. 콘텐츠/커뮤니티 시리즈의 `created_by`, `updated_by`, `moderated_by`, 시리즈 항목 `created_by`처럼 nullable 운영 메타데이터는 privacy cleanup에서 null 처리한다.

콘텐츠 예약 발행은 `status = scheduled`와 미래 `published_at`으로 표현한다. 공개/관리자 조회 시점에 예약 시각이 지난 항목은 `published`로 전환되고 `content.scheduled_published` 감사 로그를 남긴다. 예약 설정과 예약 해제는 관리자 저장 시 각각 `content.scheduled`, `content.schedule_cleared` 감사 로그로 추적한다. 관리자 미리보기 권한이 있으면 `draft`와 `scheduled` 콘텐츠 공개 URL을 열람할 수 있지만, 유료 열람 차감, 다운로드, 완료 버튼 처리는 실행하지 않는다. 관리자 목록/수정 화면의 사용자 화면 미리보기는 `preview=admin`으로 열려 공개 콘텐츠도 조회수를 올리지 않는다. 콘텐츠 댓글은 `sr_content_comments`가 소유하고 공개 콘텐츠 하단에서 작성/표시된다. 커뮤니티 게시글/댓글과 콘텐츠 댓글은 작성자 계정 ID와 작성 당시 공개 이름 snapshot을 함께 저장해 닉네임 변경 후에도 작성 당시 표시명을 보존한다. #392 재검토 결과 댓글 공통화는 공유 저장소나 공통 `comment-targets.php` 계약으로 올리지 않고, 정책 판정 뒤의 공개 read shaping/thread helper가 실제로 필요해질 때만 좁은 공통 helper 후보로 남긴다. 공개 작성자 이름 메뉴는 쪽지 보내기와 팔로우/팔로우 끊기를 제공하고, 커뮤니티 화면에서는 같은 작성자의 게시글 목록으로 이동할 수 있다. 팔로우 관계는 `sr_member_follows`에 저장하며, 팔로우한 회원이 공개 커뮤니티 게시글이나 공개 콘텐츠를 새로 등록하면 알림 모듈의 `community/followed_author.post_created`, `content/followed_author.content_created` 이벤트로 사이트 알림을 만든다. 커뮤니티/콘텐츠/퀴즈/설문 댓글 textarea는 로그인 회원에게 `/member/mention-search` 기반 멘션 후보 자동완성을 제공하며, 후보 선택 시 `@공개이름#prefix`를 본문에 삽입한다. 서버는 `@공개이름#prefix`가 현재 공개 이름과 public account hash prefix로 단일 활성 회원에 일치할 때만 확정 멘션으로 처리하고, 모호한 `@공개이름`은 알림 대상으로 확정하지 않는다. 비밀 댓글은 제3자에게 맥락을 노출하지 않도록 멘션 알림을 만들지 않는다. 댓글 알림 처리는 알림 모듈 비활성화나 템플릿 누락 시 no-op이다. 댓글 작성 감사 로그 metadata에는 작성자 알림 생성 여부를 남기고, 댓글 작성/수정 감사 로그에는 멘션 후보 수, 실제 멘션 알림 생성 수, 멘션 대상 공개 해시 목록을 남긴다. 콘텐츠 시리즈는 유료 열람 회차가 있으면 공개 시리즈 내비게이션에 완독 예상 금액을 자산별로 표시하며, 로그인 회원에게 회원 그룹 정책 적용 후 금액이 달라지면 원가와 회원가를 함께 표시한다. 로그인 회원이 이미 접근권을 보유한 회차는 남은 금액 합계에서 제외한다.

감사 로그 조회 화면 `/admin/audit-logs`는 이벤트 유형, 대상 유형, 대상 식별값, 처리자 계정 ID, 처리자 유형, IP, 처리 결과, 날짜 범위 필터를 제공한다. metadata JSON은 민감 키와 민감 문자열을 마스킹한 뒤 상세 모달에 표시하고, 모달에는 원본 이벤트/대상 식별자, 처리자, 결과, IP, user agent를 함께 보여 운영 조사 맥락을 확인할 수 있게 한다. 별도 로그 export는 아직 제공하지 않으며, 보존/삭제는 `/admin/retention`의 감사 로그 보관일 정책을 따른다.

공통 리액션은 반응/추천을 커뮤니티 전용 기능으로 두지 않고 공식 선택 모듈 `reaction`으로 구현한다. 1차 target allowlist는 `content/content`, `content/comment`, `community/post`, `community/comment`, `quiz/quiz_set`, `quiz/comment`, `survey/survey_form`, `survey/comment`이며, 대상 모듈은 `reaction-targets.php` 계약으로 단건/batch resolve, target 상태, viewer별 공개 열람 가능 여부, 반응 가능 여부, target owner/author, 알림 생성 가능 여부, 적용 preset/key를 제공한다. 계약 제공 모듈은 자기 모듈 key와 같은 `target_module`만 선언할 수 있고, `reaction`은 다른 모듈 target을 선언한 계약 항목을 무시한다. 1차 공개 리액션은 target/account 기준 단일 선택이며 unique 기준은 `(account_id, target_module, target_type, target_id)`이고, `reaction_key`는 같은 row의 현재 선택값으로 저장한다. 작성자 본인은 자신의 글/댓글/콘텐츠/퀴즈/설문 target에 리액션할 수 없고, legacy 자기 target row가 있으면 취소 또는 관리자 정리만 허용한다. 원장의 권위 소스는 `sr_reaction_records`이며 정의와 preset은 `sr_reaction_definitions`, `sr_reaction_presets`, `sr_reaction_preset_items`로 분리한다. 운영자는 리액션 종류를 추가, 수정, 정렬, 사용 중지할 수 있고, 아이콘은 이모지, Material icon key, 직접 업로드한 JPG/PNG/WebP 이미지 중 선택한다. 전체 definition 수는 제한하지 않되 preset별 공개 노출 key 수는 기본 6개, hard safety cap 12개로 둔다. 콘텐츠와 커뮤니티는 환경설정 preset을 기본값으로 쓰고 개별 콘텐츠/게시판 설정이 있으면 그 값을 우선한다. 퀴즈/설문은 환경설정 < 개별 설정 순으로 preset을 상속한다. 사용 중지 key는 신규 적용/변경 write를 막고, 기존 레코드는 보관/공개 숨김/관리자·통계 유지/삭제/병합 중 운영자가 선택한다. 삭제/병합은 영향 수 계산, 확인 문구, CSRF, 권한, 감사 로그를 요구하는 위험 작업이다. 익명 write는 허용하지 않고 탈퇴/익명화 cleanup에서는 계정의 reaction record를 삭제한다. target 작성자/소유자 알림은 notification 모듈이 활성화된 경우 no-op 가능 계약으로 처리하며, actor와 recipient가 같거나 target이 write 불가능/알림 제외 상태이면 알림을 만들지 않는다. 집계 캐시, 추천/랭킹/SEO 반영, 커뮤니티 시리즈 target, 보상 연결은 후속 범위다. 현재 사용자 반응 표면인 커뮤니티 게시글/시리즈 스크랩과 콘텐츠 완료 버튼은 유지하며, reaction은 이 기능들을 대신하지 않는다.

마일스톤 18 설문 모듈은 `survey` 서비스 모듈로 구현했다. 현재 반영 범위는 `/survey` 공개 목록, `/survey/{survey_key}` 응답 제출, 로그인/익명/동의/응답 제한, 설문 연구 메타데이터와 개인정보 안내 스냅샷, 단일/복수 선택·짧은 답변·긴 답변·숫자·별점·척도 문항, 설문 공개 레이아웃 후보와 환경설정의 레이아웃/스킨 기본값, `/admin/surveys` CRUD, `/admin/surveys/comments` 댓글 관리, `/admin/surveys/responses` 품질 관리, `/admin/surveys/reward-logs` 리워드 로그, `/admin/surveys/statistics` 통계, `/admin/surveys/export` CSV, `/admin/surveys/settings` 기본값, `/survey/ui-kit`, `comments_enabled` 기반 완료 화면 댓글 작성/수정/삭제와 비밀 댓글, 완료 화면 설문/댓글 리액션, 개인정보 export/cleanup 계약, 쿠폰 정의 읽기 참조 계약, sitemap 계약, `ledger_asset`/`coupon` 응답 보상 지급, 댓글 멘션 알림이다. 선택형 통계는 `question_key`/`choice_key`와 응답 ID 기준으로 집계해 같은 응답의 선택지를 한 번만 계산하고, 개인정보 사본은 응답 스냅샷, 답변 행, 작성 댓글을 함께 제공한다. 퀴즈와 설문 보상 grant 및 콘텐츠/커뮤니티/쿠폰 접근권 중복 방지는 `.tools/bin/check-reward-abuse-standards.php`로 통합 점검하며, 설문 전용 회귀 기준은 `.tools/bin/check-survey-consistency.php`와 `.tools/bin/check-survey-response-runtime.php`로 확인한다.

설문 공개 본문도 환경설정의 기본 `skin_key`와 개별 설문의 선택값으로 결정한다. 개별 설문의 `skin_key`가 비어 있으면 환경설정 기본값을 상속하고, 값이 있으면 허용 목록으로 검증된 개별값이 우선한다. `sr_survey_skin_options()`의 내장 목록과 `home`, `view`, `complete` 필수 view 목록으로 include 경로를 매핑한다. 스킨 또는 특정 view가 없으면 view 단위로 `modules/survey/skins/basic/*.php`에 fallback하고 운영 로그에 fallback 맥락을 남긴다. 설문 스킨은 `survey-skin-*` class hook을 유지하며, 완료 화면에서는 보상 지급 결과 안내 surface를 유지해야 한다.

마일스톤 34 모듈 공개 테마 v1은 외부 패키지를 사용하지 않고 `core/views/theme/{theme_key}`와 각 모듈의 `modules/{module}/theme/{theme_key}` 디렉터리를 공개 화면 DOM과 theme asset의 기준으로 삼는다. 모듈 레이아웃 shell은 선택된 layout provider의 `modules/{module}/theme/{theme_key}/layout.php`가 담당하고 없으면 같은 provider의 `theme/basic/layout.php`로 fallback한다. 번들 `content.basic`, `community.basic`, `quiz.basic`, `survey.basic`은 네 공개 모듈의 필수 화면 target 전체를 지원해 각 모듈 환경설정에서 다른 모듈 레이아웃도 선택할 수 있다. `basic`은 배포판 기본 view theme이고, 기존 `default` 설정값은 호환 alias로 `basic`에 매핑된다. `sample`은 DOM/색상 차이를 확인하는 샘플이며 CSS는 `/assets/theme/sample.css` 또는 `/modules/{module}/theme/sample/assets/theme.css`에서 로드한다. 모듈 공개 화면의 reset/UI kit/UI-KIT preview/layout/module stylesheet는 선택된 view theme의 `theme/{theme_key}/assets/`를 우선한다. 콘텐츠는 `home/group/content/ui-kit`, 커뮤니티는 `home/group/list/post/form/search/ui-kit`, 퀴즈는 `home/view/result/ui-kit`, 설문은 `home/view/complete/ui-kit` view를 모듈별로 요구하고, 관리자 화면에는 적용하지 않는다. 스킨과 테마는 별도 레이어이며, theme view가 없을 때만 기존 community/quiz/survey skin view가 fallback으로 사용된다. 상세 계약은 [공개 모듈 테마 기준](public-module-themes.md)을 따른다.

## 검증 기준

현재 정적 기준은 다음 명령으로 확인한다.

```sh
find . -name '*.php' -not -path './.git/*' -print0 | xargs -0 -n 1 php -l
php .tools/bin/check.php
```

HTTP와 브라우저 검증은 [스모크 테스트 기준](smoke-test.md)을 따른다. 1.0 릴리스 후보에서는 정적 점검 통과만으로 완료 판정하지 않고, 새 설치와 관리자 주요 흐름을 브라우저에서 확인한 기록을 남긴다.
