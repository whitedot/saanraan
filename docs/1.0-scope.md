# 1.0 범위 잠금 기준

이 문서는 산란 1.0 전까지 새 기능 욕심보다 현재 구현의 설치, 운영, 검증 가능성을 우선하기 위한 범위 기준이다.

## 목표

1.0은 산란이 다음 상태임을 확인하는 릴리스다.

- 새 DB에서 설치할 수 있다.
- 기본 모듈과 선택 번들 모듈을 관리자 화면에서 설치, 활성화, 업데이트할 수 있다.
- 회원, 관리자, 개인정보, 콘텐츠, 커뮤니티, 사이트 운영 보조, 회원 자산 흐름이 500 오류 없이 기본 스모크를 통과한다.
- 코어는 작은 실행 기반으로 남고, 도메인 정책은 모듈이 소유한다.
- 배포 보호, DB 접근, 보안 contract, 모듈 업데이트 기준을 문서와 구현이 함께 만족한다.

## 1.0 포함 범위

1.0 안정화 대상은 현재 저장소에 포함된 본체와 번들 모듈이다.

| 분류 | 대상 |
| --- | --- |
| 본체 | `index.php`, `core/`, 공통 asset, 설치/업데이트 helper |
| 시스템 | `admin`, `asset_ledger` |
| 회원 | `member`, `point`, `reward`, `deposit`, `asset_exchange`, `coupon` |
| 사이트 | `site_menu`, `logo_manager`, `banner`, `popup_layer`, `seo` |
| 서비스 | `content`, `community`, `quiz`, `survey`, `embed_manager` |
| 운영 | `notification`, `privacy` |
| 플러그인 | `ckeditor` |

각 모듈은 1.0 전까지 다음 기준을 우선한다.

- 설치 SQL과 update SQL 재실행 위험 점검
- 관리자 GET/POST action의 로그인, 권한, CSRF contract 점검
- 공개 route의 500 방지와 권한 우회 방지
- 개인정보 export/cleanup 계약이 있는 모듈의 탈퇴/사본 제공 흐름 점검
- 관리자 화면 필수 입력의 label, browser validation, server validation 정렬
- HTTP 스모크와 필요한 브라우저 수동 점검

## 1.0 제외 범위

다음 문서는 계획으로 유지하고, 1.0 안정화 범위에는 새 구현을 넣지 않는다.

- [본인확인 플러그인 계획](plans/identity-verification-plugin-plan.md)
- [회원 마이그레이션 계획](plans/member-migration-plan.md)
- [결제 플러그인 계획](plans/payment-plugin-plan.md)

위 기능이 1.0 전에 꼭 필요해지면 다음 조건을 먼저 만족해야 한다.

- 기존 1.0 안정화 항목을 지연시키지 않는 별도 모듈 또는 플러그인으로 격리한다.
- 새 DB 테이블과 관리자 화면, 보안/개인정보 영향, 배포 절차를 문서화한다.
- `php -l`, `.tools/bin/check.php`, 관련 HTTP/브라우저 스모크 점검을 통과한다.
- 계획 문서의 완료된 부분은 유지 문서나 모듈 README로 옮기고, 남은 계획과 섞어 두지 않는다.

## 1.0 전 보완 항목

1.0 전에는 다음 항목을 새 기능보다 우선 처리한다.

- 금액형 쿠폰은 쿠폰 모듈 단독 선행 구현 대상이 아니라 실제 적용 도메인과 함께 정의할 계약임을 문서와 관리자 화면에서 구분
- Wiki 구현 명세 정리 전까지 저장소 [1.0 전 구현 스냅샷](implementation-snapshot.md) 최신화
- README와 스모크 테스트 문서의 검사 상태 최신화
- 릴리스 zip 포함/제외 기준과 배포 보호 기준 재확인
- 새 설치, 선택 모듈 설치, 업데이트 화면, 관리자 권한 흐름의 브라우저 수동 점검 기록

## 완료 판정

1.0 릴리스 후보는 다음 조건을 만족해야 한다.

- 전체 PHP 문법 검사가 통과한다.
- `.tools/bin/check.php`가 통과한다.
- 로컬 또는 스테이징 HTTP 스모크 점검 결과가 기록되어 있다.
- 브라우저 수동 점검 결과가 기록되어 있다.
- Wiki 정리 이슈에 남길 항목과 저장소에 유지할 기준 문서가 구분되어 있다.
- 새 기능 계획 문서가 구현 완료 항목과 미구현 항목을 섞어 설명하지 않는다.
