# 관리자 UI 작성 기준

관리자 화면은 G5 Codex 계열의 관리자 UI 톤을 기준으로 맞춘다. public/admin 양쪽에서 쓰는 이름이 같은 기본 UI class라도 런타임 CSS 파일은 분리한다. 관리자 반복 UI 원형은 admin 모듈의 `modules/admin/assets/common.css`, 공개 반복 UI 원형은 `assets/ui-kit.css`가 각각 소유하고, 관리자 shell과 관리자 전용 배치는 `modules/admin/assets/admin.css`로 제한한다. 공개 UI-KIT 미리보기 배치는 `ui-kit-layout.css`에 두고, 모듈 도메인 고유 보정은 각 모듈의 `admin.stylesheets`로 분리한다.

대부분의 실용적인 운영 기능은 여러 모듈의 상태, 설정, 조치 항목을 한 화면이나 한 작업 흐름 안에 배치해야 하는 필요가 생긴다. 이런 관리자 화면은 통합 CRUD가 아니라 운영 보조판으로 설계하고, 각 정보의 제공 모듈과 상세 관리자 화면을 드러낸다. 도메인 상태 판단과 저장 정책은 소유 모듈의 명시적 계약이나 endpoint에 두며, 자세한 기준은 [모듈을 가로지르는 운영 흐름 가이드](cross-module-operations-guide.md)를 따른다.

환경설정 화면은 운영자가 결정을 내리는 단위로 `section.card`를 나눈다. 새 레코드에 복사되는 기본값은 `작성 기본값` 또는 `레벨 기본값`, 공개 레이아웃과 메뉴 연결은 `공개 화면 구성`, 접근 조건은 `접근 정책`, 포인트/금액/유료 열람/재결제 기준은 `자산/과금` 또는 `이용/과금 기준`, 대량 재계산과 삭제처럼 되돌리기 어려운 작업은 `위험 작업`으로 분리한다. 한 카드에 표시 구조, 접근 정책, 과금 기준을 섞지 않고, 저장 key는 호환을 위해 유지한다.

관리자 폼 행의 라벨과 첫 컨트롤은 기본적으로 같은 높이 중심선에 맞춘다. 스위치, 셀렉트, 정적 값, 회원 그룹 배지 목록, 자산/과금 설정처럼 컨트롤 높이가 다른 항목도 첫 줄 기준이 흔들리지 않아야 하며, 긴 설명이나 다중 행 보조 정보는 첫 컨트롤 아래 보조 영역으로 흐르게 둔다.

공개 레이아웃의 메뉴 연결은 고정 위치명이 아니라 슬롯명으로 표시한다. 예를 들어 `layout_primary_menu_key`와 `site_menus.primary`는 label을 `주 메뉴 슬롯`으로 두고, 여러 메뉴 슬롯을 지원하는 모듈의 `layout_secondary_menu_key`, `layout_tertiary_menu_key`, `layout_quaternary_menu_key`, `layout_quinary_menu_key`는 `보조 메뉴 슬롯`, `추가 메뉴 슬롯 1`, `추가 메뉴 슬롯 2`, `추가 메뉴 슬롯 3`처럼 표시한다. 선택지는 `사용 안 함`, 모듈이 제공하는 내장 메뉴, 사이트 메뉴 순서로 둔다. 콘텐츠는 내장 `콘텐츠 그룹`, 커뮤니티는 주 메뉴 슬롯에서만 내장 `게시판 그룹`을 제공한다. 실제 출력 위치가 레이아웃마다 달라질 수 있음을 도움말에 적고, 사용 안 함 선택 시 모듈 fallback이 표시되는 화면은 fallback 기준도 같은 도움말에 적는다.

긴 관리자 폼에서 섹션 이동 탭이 필요하면 전역 class `sticky-tabs anchor-tabs tab-nav-justified`를 사용한다. 탭 관련 호환 alias는 두지 않고 접두 없는 탭 class를 기준으로 붙인다.

관리자 z-index 계층은 레이아웃 shell을 `--admin-z-layout: 1000`, sticky 레이아웃을 `1010`, 레이아웃 popover를 `1100`, 모달/오버레이를 `--admin-z-overlay: 2000`, toast를 `2100` 기준으로 둔다. 모달은 항상 사이드바, 상단바, sticky 탭 위에 떠야 한다.

신규 설치에서 사이트 메뉴 모듈을 선택하면 기본 `header` 메뉴는 홈과 설치된 서비스 모듈의 메인 페이지로 구성한다. 로그인/회원가입은 기본 header 항목이 아니라 회원 영역 또는 메뉴 후보에서 운영자가 직접 추가하는 링크로 다룬다.

사이트 메뉴 관리자 화면은 초안 편집과 공개 반영을 분리한다. 메뉴/항목 추가, 수정, 삭제, 순서 저장은 초안에만 저장하고, 운영자가 `공개 반영`을 눌렀을 때 현재 초안을 공개 메뉴 테이블로 교체한다. 사이트 메뉴 항목 삭제 시 하위 항목이 있으면 하위 항목도 함께 삭제된다는 확인을 받은 뒤 초안에서 삭제하고, 서버 POST에서도 같은 확인값을 다시 검증한다.

SEO 설정의 사이트맵 영역은 확인 링크와 URL 복사 버튼을 함께 제공한다. robots 설정은 다른 설정 카드와 같은 폼 흐름을 사용하고, `robots.txt` 파일 확인 링크는 카드 헤더 오른쪽 보조 버튼으로 둔다. robots 미리보기는 박스형 코드 블록이 아니라 줄바꿈을 유지하는 보조 meta 텍스트로 표시한다.

데이터량에 따라 부하가 달라지는 작업은 `낮음`, `주의`, `높음`, `매우 높음` 등급으로 안내한다. 등급 판단에는 대상 레코드 수, 관련 테이블 수, 파일 저장소 작업 수, 외부 저장소 호출, 긴 트랜잭션 가능성, 실패 시 롤백/재시도 가능성을 함께 반영하되, 작업별 운영자 체감 기준이 더 정확하면 해당 모듈의 구체 기준을 우선한다. 예를 들어 게시판 복사는 여러 테이블을 만진다는 사실만으로 `주의`를 올리지 않고, 선택한 복사 범위의 게시글·댓글·첨부·시리즈·첨부 총량을 기준으로 산정한다. 실행 전 화면 또는 모달에는 처리 대상 수, 영향을 받는 데이터, DB/파일/외부 저장소 부하 성격, 권장 실행 시점, 중단/실패 시 상태, 취소/되돌리기 가능 여부, 감사 로그나 작업 목록 기록 위치를 표시한다. `높음` 이상 작업은 확인 문구 입력, dry-run/사전 계산, 배치 실행, 진행률 표시 중 하나 이상을 적용하고, 확인 문구나 hidden 확인값은 서버 POST에서 다시 검증한다. 작업 완료/실패/부분 완료는 화면 피드백과 감사 로그 metadata에 대상 수, 성공/실패 수, 배치 여부, 부하 등급, 확인 검증 결과를 남긴다.

고부하 관리자 작업은 모듈별 작업 테이블과 공통 상태 contract를 기본으로 설계한다. 신규 작업은 `즉시 제한형` 또는 `작업 테이블형` 중 하나로 분류하고, 작업 ID 없이 브라우저가 cursor를 이어 보내는 배치형은 기존 구현의 전환/호환 패턴으로만 둔다. 고위험 작업은 기본적으로 작업 테이블형을 사용하지만, 소량·건수 제한·서버 재검증·조건부 상태 전이·대상 단위 멱등성으로 충분히 통제되는 경우에만 즉시 제한형을 유지할 수 있다. 작업 테이블형의 `lock_token`은 fencing token으로 보아 늦게 끝난 이전 요청의 쓰기를 거부해야 하며, 도메인 쓰기와 map/cursor/count 갱신은 같은 commit에 포함한다. 재시도 가능한 batch는 대상 단위 완료 마커, map row, dedupe key, 또는 `status = :expected_from` 조건부 상태 전이를 가져야 한다. 검색 결과 전체 선택은 고위험·금전성 작업에서는 explicit ID snapshot 고정을 기본으로 하고, 그룹 재평가나 레벨 재계산처럼 멱등적인 작업에만 서버가 정규화한 query snapshot 재계산을 허용한다. query snapshot 모드는 batch 전 drift를 계산하고 절대 50건 이상 또는 10% 이상 변동하면 재확인 필요 상태로 멈춘다. 진행 모달은 서버 상태 조회 결과를 표시하고, 진행 변화는 `aria-live` 영역으로 안내하며 단계 전환 후 focus를 제목 또는 첫 오류/진행 상태로 옮긴다.

`/admin/storage-cache`는 `썸네일 파일 캐시` 제목으로 공개 이미지 썸네일 파일 캐시를 다루는 파일 스캔형 운영 화면이다. 페이지 제목 아래에는 공개 이미지에서 만든 작은 이미지 파일이며 정리해도 원본 파일과 게시글이 삭제되지 않는다는 제약을 짧게 둔다. 기간 필터는 파일 수정 시각 기준으로 집계하고, 모듈 필터는 현재 기간 조건에서 발견된 캐시 모듈을 select로 제공한다. 필터 영역은 초기화 버튼 없이 필요한 너비만 차지하는 형태로 표시하고 조회 버튼은 모듈 select 바로 다음에 둔다. 파일 수, 총 용량, 가장 오래된 파일, 가장 최근 파일 요약은 `일자별 생성 내역` 목록 상단에 일반 뱃지 스타일로 표시한다. 정리 작업은 위험 작업으로 화면 맨 아래에 두고, sticky 액션의 `정리` 버튼이 확인 모달을 열며 현재 조회 조건, 삭제 권한, CSRF, 확인 문구, 감사 로그를 모두 요구한다. 이 화면은 원본 파일이나 모듈 데이터를 변경하지 않고 `storage/cache/thumbnails` 아래의 생성 파일명 패턴과 일치하는 이미지 캐시만 대상으로 한다.

`/admin/community/feed-cache`는 `피드 캐시` 제목으로 커뮤니티 메인과 게시판 목록 밖 피드의 공개 baseline 최신글/인기글, 익명/public 메인의 최신댓글 조회 계약을 read-only로 점검하는 화면이다. 현재 파일 영속 저장소 상태, 현재 유효한 feed context, 코드에 고정된 갱신 정책을 보여 준다. 갱신 대기 파일은 이전 조건으로 만든 값이므로 표에 섞지 않고 요약 숫자로만 표시한다. baseline은 게시판 관리의 `커뮤니티 피드 노출`이 켜진 게시판만 대상으로 하며, 이 설정이 꺼진 게시판의 글, 댓글, 시리즈는 커뮤니티 메인, 사이드바, 최근글/인기글 같은 게시판 목록 밖 피드 후보에서 제외된다. 게시글, 댓글, 게시판 공개/권한/유료 열람 조건 변경 시 기존 저장값이 삭제되고 다음 조회에서 다시 계산된다는 점을 운영자 표현으로 안내한다. 게시판별 상세 기준은 게시판 관리 화면의 책임으로 둔다.
로그인 사용자의 홈 화면에 회원 전용 게시판이 섞여 있어도 public baseline 파일 캐시는 별도로 예열될 수 있으며, 이 경우 관리 화면에는 public baseline 컨텍스트만 표시된다.

- `assets/reset.css`: 공개 런타임에서 재사용할 토큰, 아이콘 fallback, reset/base, 접근성 helper, 역할 기반 타이포그래피를 둔다.
- `modules/admin/assets/tokens.css`: 관리자 런타임에서 재사용할 `--color-*`, `--spacing`, 타이포그래피, 반경, 그림자 토큰을 둔다. 공개 토큰 파일과 공유하지 않는다.
- `modules/admin/assets/common.css`: 관리자 런타임의 reset/base와 `btn`, `card`, `table`, `badge`, `form-*`, `dropdown-*`, `modal-*`, `tab-*`, icon sizing 같은 반복 UI 원형을 둔다.
- `assets/ui-kit.css`: 공개 kit profile에서만 쓰는 반복 UI 원형과 public UI 조합을 둔다. 관리자 런타임과 파일을 공유하지 않고, 콘텐츠/커뮤니티처럼 서비스 단위의 화면은 모듈 복제 CSS를 명시 호출한다.
- `assets/ui-kit-layout.css`: 원본 `/ui-kit` 미리보기의 컨테이너, 샘플 배치, 미리보기 전용 helper만 둔다.
- `assets/layout.css`: 기본 공통 공개 layout shell만 둔다.
- `assets/module.css`: 초기 공개 화면의 본문 스타일만 둔다.
- `assets/admin-ui.css`: `.admin-ui-scope` 안의 반복 가능한 관리자 작업 조합만 둔다.
- `modules/admin/assets/admin.css`: 관리자 shell, 사이드바, 상단바, 관리자 콘텐츠 폭, 목록/폼 배치 같은 admin 모듈의 실제 화면 구조를 둔다. 공용 UI 원형이나 모듈 도메인 class를 이 파일에 다시 넣지 않는다.
- `modules/{module_key}/assets/*.css`: 모듈 관리자 본문에서만 쓰는 도메인 고유 스타일은 `admin.css`에 둔다. 공개 화면 스타일을 가진 모듈은 서비스 화면 스타일을 `module.css`, 레이아웃 shell을 `layout.css`, 스킨 화면 요소를 `skin.css`에 둔다. 콘텐츠/커뮤니티/퀴즈/설문처럼 공개 foundation, UI 원형, UI-KIT 미리보기를 독립 검증해야 하는 모듈은 `admin.css`, `reset.css`, `layout.css`, `module.css`, `ui-kit.css`, `ui-kit-layout.css`를 기본으로 둔다. 공개 레이아웃을 제공하는 모듈의 공개 CSS는 `public-*`, `public-ui-*`, `sr-public-*`가 아니라 모듈 namespace class를 사용한다. 모듈 CSS는 `btn`, `card-*`, `form-*`, `filtering-*`, `modal-content`처럼 UI-KIT이 소유한 class를 재정의하지 않고, 배치 조정은 모듈 전용 wrapper나 도메인 class로 표현한다.

공개 화면 런타임은 style profile로 CSS 호출 범위를 고른다. 기본 공개 profile은 `Pretendard`, `assets/reset.css`를 먼저 호출하고, Google Material Symbols는 reset 안의 번들 폰트 선언으로 제공한다. `kit` profile은 그 뒤에 `assets/ui-kit.css`와 layout context의 `stylesheets`를 호출하며, 공통 공개 layout이 기본으로 사용한다. 회원, 알림, 포인트/자산, 개인정보처럼 자기 public layout을 갖지 않는 account/utility 화면은 사이트 설정의 공개 layout이 제공하는 공통 UI kit primitive를 그대로 쓰고, layout context에는 필요한 스킨 stylesheet만 추가한다. `/ui-kit`은 초기/기본 공개 페이지와 public layout 런타임을 확인하는 원본 UI-KIT로 보존하고, 미리보기 보조 CSS는 `assets/ui-kit-layout.css`만 호출한다. `minimal` profile은 reset 뒤에 layout context의 `stylesheets`만 호출한다. `module` profile은 루트 reset을 자동 호출하지 않으므로 모듈 helper가 자기 `assets/reset.css`를 먼저 추가해야 한다. 공통 공개 layout은 `assets/layout.css`와 `assets/public-layout.js`를 사용하고, 초기 공개 화면은 `assets/module.css`를 추가한다. 별도 공개 theme stylesheet는 사용하지 않는다. 콘텐츠/커뮤니티/퀴즈/설문 공개 화면은 각 모듈의 `reset.css`, `ui-kit.css`, 선택된 layout provider의 레이아웃 stylesheet, `module.css` 순서로 호출하고, 모듈 UI-KIT 화면은 `ui-kit-layout.css`를 추가한다. 레이아웃을 제공하는 모듈은 layout shell 동작을 `assets/layout.js`, 화면 모듈 동작을 `assets/module.js`로 분리한다. 스킨을 사용하는 페이지와 요소는 `skin.css`를 화면 조합에 맞게 추가한다. `install` profile은 reset과 설치 전용 `assets/install.css`만 사용한다. 일반 공개 런타임과 `/ui-kit` 공개 UI-KIT 모두 관리자 런타임 CSS인 `modules/admin/assets/common.css`, `assets/admin-ui.css`, `modules/admin/assets/admin.css`를 직접 호출하지 않는다.

공개 footer의 색상 모드 선택은 `assets/reset.css`와 각 모듈 reset이 제공하는 `data-color-scheme` 기반 기본 토큰을 사용하고, 방문자별 선택값은 `assets/common-ui.js`가 `localStorage`에 저장해 같은 브라우저에서 유지한다. 공통 public layout은 `public-*` class를 사용할 수 있지만, 콘텐츠/커뮤니티/퀴즈/설문처럼 public layout을 제공하는 모듈은 footer 색상 모드 컨트롤도 각각 `content-*`, `community-*`, `quiz-*`, `survey-*` namespace class로 표현한다.

공개 reset과 모듈 reset은 문서 `body` 배경색을 직접 칠하지 않는다. 사용자 화면의 기본 문서 배경은 공통 또는 모듈 public layout shell이 `--color-body-bg`로 담당하고, 개별 화면은 운영자나 사용자 요청으로 명시된 경우 또는 UI-KIT 컴포넌트가 이미 소유한 상태/표면 표현인 경우를 제외하고 임의의 배경색을 새로 지정하지 않는다. 다크모드에서도 같은 원칙을 적용하며, 별도 배경 레이어를 늘리는 대신 기존 텍스트/경계/상태 토큰으로 시인성을 맞춘다.

관리자/공개 런타임 CSS 호출은 PHP helper가 실제 파일의 `filemtime()` 값을 `?v=` query string으로 붙여 캐시를 갱신한다.

관리자 런타임 CSS 호출 순서는 `Pretendard`, `modules/admin/assets/tokens.css`, `modules/admin/assets/common.css`, `assets/admin-ui.css`, `modules/admin/assets/admin.css`, 현재 화면에 필요한 모듈의 `admin.stylesheets` 선언 순서다. Google Material Symbols는 `modules/admin/assets/common.css` 안의 번들 폰트 선언으로 제공한다. 일반 관리자 하위 페이지에서는 관리자 메뉴/경로 기준으로 현재 화면을 소유한 모듈만 포함하고, `/admin` 대시보드에서는 실제 대시보드 섹션을 제공하는 모듈만 포함한다. 관리자 view가 다른 모듈의 명시적 UI 컴포넌트를 함께 렌더링해야 하는 좁은 예외에서는 `sr_admin_stylesheet_tag()`의 추가 모듈 인자로 필요한 모듈 키를 직접 넘긴다. 모듈 stylesheet는 `/modules/{module_key}/assets/*.css` 안쪽 파일만 허용하고 외부 URL, `..` 경로, 모듈 폴더 밖 파일은 무시한다. 공개 모듈 stylesheet는 화면을 렌더링하는 view/layout context에서 필요한 파일만 명시한다.

프로젝트 기본 아이콘셋은 Google Material Symbols Outlined다. 아이콘 폰트는 외부 Google Fonts stylesheet를 호출하지 않고 public/admin 런타임 CSS의 번들 `@font-face` 선언으로 제공한다. 공용 helper는 `sr_icon()`이며 기존 `sr_material_icon_html()`은 호환 래퍼로 유지한다. `.sr-icon[data-sr-material-icon]`은 `sr_icon_bootstrap_script()`가 폰트 준비 완료 class를 붙이기 전까지 투명하게 유지하므로, `home`, `settings` 같은 Material ligature 글자가 로딩 중 잠깐 보이지 않는다. 관리자 메뉴 허용 심볼 이름은 `settings`, `admin-mode`, `users`, `user`, `content`, `stats`, `home`, `folder`, `image`, `layers`, `search`, `menu-list`, `bell`, `shield`, `coins`, `point`, `database`, `savings`, `payments`, `wallet`, `gift`, `ticket`, `message-circle`, `service`, `quiz`, `poll`, `emoji_emotions`, `integration_instructions`다. `/admin/settings`의 관리자 화면 섹션에서 아이콘 키별 표시 방식을 `Material` 또는 `이미지`로 선택할 수 있고, 기본 키 외 사용자 키를 추가할 수 있다. 기본 키는 제거할 수 없고 초기화만 가능하며, 추가한 키만 제거할 수 있다. Material 방식은 키가 가리킬 Material Symbols 이름을 저장하고, 이미지 방식은 JPG, PNG, GIF, WebP 파일을 storage에 업로드한 뒤 관리자 전용 `/admin/icon-image` 엔드포인트로 렌더링한다. 이 엔드포인트는 관리자 설정 조회 권한을 확인하고, S3 저장소에서는 공개 URL 대신 짧은 signed URL만 발급한다. 아이콘 설정 저장이 실패하면 해당 요청에서 새로 업로드한 아이콘 이미지는 삭제하고, 저장 성공 후 더 이상 참조하지 않는 이전 아이콘 이미지는 정리한다. 이미지 삭제가 실패하면 관리자 오류와 서버 오류 로그로 확인할 수 있게 한다.

Material Symbols는 메뉴, 툴바, 본문 액션처럼 페이지에서 독립 아이콘을 표시할 때 사용한다. 체크박스 체크 표시, 드롭다운 caret, 스위치 thumb처럼 컴포넌트 내부 상태나 조작 힌트는 해당 컴포넌트 CSS가 소유하며 Material 아이콘으로 대체하지 않는다. 드롭다운 caret처럼 여러 위치에서 재사용하는 방향 화살표는 `sr_ui_arrow_icon_html()` helper로 출력한다.

드롭다운, 오버레이/모달, 탭처럼 관리자와 공개 화면에서 함께 쓸 수 있는 기본 상호작용은 `assets/common-ui.js`에 둔다. 관리자 UI-KIT과 public 런타임 미리보기도 이 파일을 호출해 같은 동작 원형을 확인한다.

관리자 디자인 책임은 admin 모듈에 둔다. 각 모듈의 관리자 view는 본문 마크업과 도메인 출력만 맡고, 관리자 shell, 사이드바, 상단바, 공통 관리자 asset, 관리자 콘텐츠 컨테이너는 admin skin이 맡는다. 현재 관리자 skin은 `admin_skin_key`로 선택하며, 등록된 key가 없거나 파일이 없으면 `basic`으로 fallback한다.

관리자 shell은 화면을 편하게 그리기 위한 목적으로 view 출력 뒤 `DOMDocument`로 HTML을 다시 해석하거나 class를 주입하지 않는다. 폼 행, 카드 헤더, 테이블 wrapper, 체크박스 보조 텍스트처럼 의미가 있는 구조는 view가 최종 마크업으로 직접 출력해야 한다.

렌더 후 DOM 처리가 필요한 경우는 별도 예외로 다룬다. 예를 들어 보안 정화, 외부 HTML의 제한적 변환, 편집기 콘텐츠 처리처럼 입력 자체가 HTML이고 변환 목적이 명확한 경우에는 호출 위치와 책임 모듈을 드러내고, 허용 범위와 테스트를 함께 둔다. 단순히 관리자 화면의 반복 마크업을 줄이거나 class를 자동 보정하기 위한 후처리는 사용하지 않는다.

관리자 화면에서 `방금 전`, `5분 전`처럼 축약한 일시는 PHP helper가 `<time class="sr-time-tooltip" data-sr-time-tooltip>` 마크업을 직접 출력해야 한다. 정확한 일시는 `datetime`, `data-sr-time-tooltip-label`, `aria-label`에 함께 남기고, JavaScript로 렌더 후 class나 tooltip 속성을 주입하지 않는다.

CSS class는 범위를 드러내는 접두어를 사용한다.

- 관리자 반복 UI는 `btn`, `card`, `table`, `badge`, `form-*`처럼 `modules/admin/assets/common.css`에 직접 둔다. 공개 kit profile의 같은 이름 class는 `assets/ui-kit.css`에서 따로 소유한다. 공개 minimal profile을 쓰는 모듈은 자기 wrapper 안에서 필요한 컨트롤 스타일을 모듈 CSS로 소유한다.
- 관리자 shell과 관리자 전용 배치는 `admin-*` 접두어를 사용하고 `modules/admin/assets/admin.css`에 둔다.
- 모듈별 관리자 본문에서 도메인 고유 스타일이 필요하면 `{module_key}-admin-*` 또는 `sr-{module_key}-admin-*` 형식을 사용하고, 해당 CSS는 모듈의 `assets/` 아래 stylesheet에서 소유한다.
- 관리자 view는 전역 `body`, `a`, `.container`, `.btn` 같은 넓은 선택자를 직접 재정의하지 않는다.
- 목록 상단의 검색/필터 영역은 입력과 버튼의 작업 흐름으로 취급하고, 별도 카드처럼 보이는 배경, 테두리, 큰 안쪽 여백을 기본으로 두지 않는다.
- 검색/필터 영역의 label은 `0.75rem` 수준, input/select/textarea/button 텍스트는 `0.875rem` 수준을 기준으로 둔다.
- 탭처럼 공통 CSS에 이미 정의된 반복 UI는 `tab-nav-*`, `tab-trigger-*` 같은 기존 시맨틱 클래스를 먼저 사용한다. 탭 전환 화면은 초기 HTML에서 `role="tablist"`, `role="tab"`, `role="tabpanel"`, `aria-controls`, `aria-labelledby`, `aria-selected`를 직접 출력하고, 동작은 공통 `data-tab` 마크업 기준으로 붙인다. 알림은 관리자와 공개 UI-KIT 모두 `alert alert-{status}` 또는 `alert alert-solid alert-{status}`를 기준 class로 사용한다. `alert` 자체가 테두리, 얕은 그림자, 상태별 배경/텍스트 대비를 소유하므로 모듈 CSS에서 상태색이나 alert 표면을 다시 만들지 않는다. 토스트는 기존 관리자 메시지 클래스인 `admin-flash-message-*`에 `alert` 상태 class와 `data-admin-toast` 동작 속성을 더해 사용하고, 위치와 닫기 버튼 배치는 `data-admin-toast-*` 속성 선택자로 처리한다.
- 상태 배지나 유형 코드가 있는 목록 하단의 짧은 설명은 `card-description-block`, `card-description-title`, `card-description-list`를 사용한다. 마크업은 `dl`을 유지하되 `dt`/`dd`가 한 줄 설명처럼 읽히게 하며, 배경은 목록 테이블 헤더와 같은 `var(--color-default-100)`, 글자 크기는 `0.875rem` 기준으로 둔다.
- 긴 등록/수정 폼의 섹션 이동 보조는 UI kit 탭 class를 시각 표현으로 재사용하되, 콘텐츠를 숨기는 탭 패널 전환으로 만들지 않는다. wrapper는 `sticky-tabs anchor-tabs tab-nav-justified`, 링크는 `tab-trigger-underline-justified`, 각 대상 섹션은 안정적인 `id`와 `data-admin-section-anchor`를 사용한다. 이 경우 `role="tablist"`나 `role="tab"`을 쓰지 않고 일반 앵커 내비게이션으로 둔다. 첫 렌더에서는 첫 링크에 `active`와 `aria-current="location"`을 줄 수 있고, 이후 `modules/admin/assets/admin-shell.js`가 현재 스크롤 위치에 맞춰 `active`와 `aria-current="location"`을 갱신한다.
- UI-KIT 조회 화면의 배치와 예시 상태 표시처럼 실제 컴포넌트 원형이 아닌 표현은 `ui-kit-*` 접두어로만 둔다. `ui-bg-*`, `ui-text-*`, `ui-grid`, `ui-flex`, `ui-gap-*` 같은 Tailwind식 범용 utility 표현은 관리자/공개 UI-KIT 샘플에 사용하지 않는다.
- 공통 UI를 변경하거나 새 관리자 화면에서 UI 조합을 확인할 때는 `/admin/ui-kit` 관리자 조회 화면과 `/ui-kit` public 런타임 UI-KIT에서 런타임별 결과를 확인한다. `/ui-kit`은 초기/기본 공개 페이지용 원본 UI-KIT이므로 모듈 화면 기준으로 덮어쓰지 않는다. `/admin/ui-kit` 상단에는 초기 UI Kit과 설치된 모듈별 사용자 화면 UI Kit 링크를 함께 제공한다. 모듈이 별도 UI-KIT 조회 화면을 제공하면 해당 모듈의 실제 화면 조합은 모듈 UI-KIT도 함께 확인한다. 예를 들어 콘텐츠 모듈의 공개 타이포그래피와 콘텐츠 화면 조합은 `/content/ui-kit`, 커뮤니티 모듈의 게시판/게시글 화면 조합은 `/community/ui-kit`, 퀴즈 모듈의 목록/응시/결과 화면 조합은 `/quiz/ui-kit`, 설문 모듈의 목록/응답/완료 화면 조합은 `/survey/ui-kit`에서 현재 공개 레이아웃 설정을 적용해 확인한다.

## 관리자 안내 문구

관리자 화면의 안내 문구는 운영자가 저장 전 판단해야 하는 결론을 먼저 보여주고, 긴 배경 설명은 필요할 때만 열어볼 수 있게 나눈다. 새 기준을 만들 때는 이 문서의 폼, 목록, 모달, 토스트 기준을 함께 갱신하고, 실제 화면이 이미 사용하는 `form-help`, `$adminPageSubtitle`, `sr_admin_feedback_toasts()`, `alert`, `admin-notice` 흐름에 맞춘다.

- 짧고 즉시 필요한 정보는 컨트롤 아래 `p.form-help`, `small.form-help`, `span.form-help`에 둔다. 입력 형식, 허용 문자, 단위, 비워 두었을 때의 기본값, 저장 후 반영 시점, 공개 화면 노출 여부, 조건부 필수 판단처럼 저장 직전에 알아야 하는 정보가 여기에 속한다.
- 선택지 설명, 우선순위, 예외, 보안/개인정보 기준, 고부하 작업의 영향 범위처럼 한두 문장으로 충분하지 않은 내용은 항목명 앞의 `?` 도움말로 분리한다. 단, `?` 도움말을 열지 않아도 기본 입력과 저장 판단은 가능해야 한다.
- 같은 내용을 힌트와 `?` 도움말에 반복하지 않는다. 힌트는 결론, 도움말은 배경과 예외를 맡는다.
- 페이지 제목 아래 `$adminPageSubtitle`는 제목만 읽어도 알 수 있는 목적 설명에는 쓰지 않는다. 삭제되지 않음, 읽기 전용, 통계 제외 기준, 최대 단계 수, 마지막 소유자 보호처럼 화면에 들어가기 전 알아야 하는 제약과 예외가 있을 때만 짧게 둔다. 개별 필드 입력법은 페이지 안내가 아니라 해당 필드의 `form-help`나 도움말로 보낸다.
- placeholder는 안내 문구의 대체물이 아니다. 검색 단서나 짧은 입력 예시만 placeholder에 두고, 저장 조건이나 제한은 보이는 힌트 또는 도움말에 둔다.
- 한 모듈의 화면에서 다른 모듈의 기능을 비유로 들어 설명하지 않는다. 내부 재사용 구조가 있더라도 운영자 문구에는 노출하지 않고, 현재 화면의 도메인 언어로 직접 설명한다. 실제 연동 관계가 있으면 “선택한 대상”, “연동 대상”, “참조 대상”처럼 현재 작업의 관계를 설명한다.
- `helper`, `route`, `contract`, SQL, 테이블명, 내부 파일 경로 같은 구현 용어는 운영자가 직접 조치해야 하는 경우가 아니면 화면 문구에 쓰지 않는다. 필요한 경우에도 쉬운 설명을 먼저 쓰고 기술 용어는 보조로만 둔다.
- 이메일 원문, 토큰, 공개 해시가 아닌 내부 ID, IP/User-Agent 원문, 파일 시스템 경로, 외부 저장소 signed URL처럼 민감하거나 운영자 판단에 필요하지 않은 값은 토스트, alert, 도움말에 노출하지 않는다. 자세한 원인은 서버 로그나 감사 로그에 남기고, 화면에는 결과와 다음 행동을 안내한다.

실행 결과와 상태 메시지는 성격에 따라 위치를 나눈다.

- 저장, 삭제, 적용 같은 POST 결과는 PRG 뒤 `sr_admin_feedback_toasts($notice, $errors)`로 짧게 표시한다. 성공 메시지는 완료된 작업 단위를 구체적으로 말하고, 실패 메시지는 가능한 경우 수정할 행동을 함께 말한다.
- 필드 validation 오류는 필드 근처 오류가 우선이며, 서버 검증 실패 toast는 전체 실패 요약을 맡는다. 두 문구가 서로 다른 조건을 말하지 않아야 한다.
- 조건 미충족, 차단 사유, 고위험 작업 경고처럼 화면에 계속 남아야 하는 상태성 피드백은 `alert` 또는 `admin-notice`를 유지할 수 있다.
- 단순 사용법이나 운영 기준이 본문 흐름을 끊고 있다면 우선 삭제한다. 저장 판단에 필요한 짧은 조건은 `form-help`, 길거나 예외가 많은 내용은 `?` 도움말, 페이지 전체에 걸친 제약만 `$adminPageSubtitle`로 옮긴다.

버튼, 배지, 탭, 드롭다운, 모달처럼 UI-KIT 샘플과 kit profile 실제 화면이 함께 쓰는 컴포넌트는 기본, hover, focus-visible, disabled 상태를 관리자에서는 `modules/admin/assets/common.css`, 공개 kit에서는 `assets/ui-kit.css`가 각각 소유한다. `btn-solid-*`, `btn-outline-*`, `btn-soft-*`, `btn-ghost-*`, `btn-gradient-*`, `btn-icon`, `badge`, `badge-list`, `tab-trigger-*`, `dropdown-*`, `modal-*` 계열을 kit profile에 새로 쓰면 `/admin/ui-kit`과 `/ui-kit`에서 상태별 표현을 함께 확인한다. 콘텐츠/커뮤니티 같은 minimal profile 화면에서는 이 class에 의존하기보다 모듈 CSS가 자기 화면 범위 안에서 필요한 스타일을 제공한다.

UI-KIT 샘플은 프로젝트 공용 아이콘 helper에 의존한다. 실제 아이콘 자리에는 Google Material Symbols를 출력하는 `sr_icon()`을 우선 사용하고, 기존 `sr_material_icon_html()` 호출은 같은 출력으로 이어지는 호환 래퍼로 본다. 샘플 설명이나 임시 보조 표식처럼 아이콘 자체가 목적이 아닌 위치에만 텍스트 보조 라벨을 둔다.

## 화면 내 이동 링크

사이드 메뉴에 이미 등록된 관리자 화면 이동 링크는 본문 상단에 다시 만들지 않는다. 예를 들어 회원 그룹의 그룹 목록과 회원 그룹 규칙처럼 사이드 메뉴가 1차 탐색을 제공하는 화면은 각 view 안에서 `admin-local-nav` 버튼 묶음을 반복하지 않는다.

`admin-local-nav`는 사이드 메뉴에 없는 같은 화면 안의 하위 흐름이나 필터성 이동에만 사용한다. 상태 요약 링크처럼 현재 목록의 조건을 빠르게 바꾸는 경우가 여기에 해당한다. 잔액/조정/거래 내역, 알림 목록/이메일 발송 작업처럼 독립 운영 화면으로 자주 오가는 흐름은 사이드 메뉴 항목으로 둔다. 새 항목 추가, 수정, 목록 복귀처럼 현재 작업의 직접 액션은 카드 헤더나 폼 액션 영역에 둔다.

관리자 헤더의 운영 알림 드롭다운은 회원 사이트 알림과 다른 흐름이다. 열린 운영 알림 수와 최근 항목만 요약하고, 상세 필터와 확인/처리/보관 상태 변경은 `/admin/admin-notifications` 화면에서 처리한다. 드롭다운 항목 본문을 누르면 해당 알림을 읽음 처리한 뒤 바로가기로 이동하고, 항목의 읽음 버튼을 누르면 이동하지 않고 현재 화면에서 읽음 처리만 하며 해당 항목은 드롭다운에서 즉시 사라져야 한다. 운영 알림 바로가기는 가능한 경우 알림의 `target_type`과 `target_id`를 해당 관리자 목록의 단건 필터로 변환해, 예를 들어 커뮤니티 신고 알림은 `/admin/community/reports?report_id=ID`처럼 관련 건만 보이게 한다.

포인트처럼 운영자가 표시명을 바꿀 수 있는 회원 자산 모듈은 사이드 메뉴의 모듈명과 잔액/거래 내역 항목에도 같은 표시명을 사용한다.

사이드 메뉴와 같은 전체 화면 이동 버튼을 본문 상단에 반복하지는 않지만, 현재 데이터 컨텍스트를 유지하는 액션 링크는 본문에 둘 수 있다. 예를 들어 회원 자산 화면에서 특정 회원을 조회한 뒤 같은 `account_identifier`로 잔액, 조정, 거래 내역을 오가는 버튼이나 잔액 목록 행의 `조정`, `거래 내역` 버튼은 허용한다. 회원 그룹 수동 배정과 배정 이력처럼 특정 그룹을 대상으로 하는 작업은 별도 사이드 메뉴 화면으로 두지 않고 그룹 목록 행의 관리 모달에서 처리한다.

## 삭제 상태 목록

글, 댓글, 제출본처럼 공개 화면에서 삭제 또는 숨김 처리된 운영 대상이 관리자 목록에 남는 경우, 삭제 상태는 별도 배지와 상태 필터로 식별할 수 있어야 한다. 삭제 상태 행을 일반 공개 행과 같은 모양으로 섞지 말고, 공개 링크는 대상 모듈의 정책에 따라 미리보기나 관리자 전용 조회로 제한한다.

관리자 화면의 `삭제` 버튼이 실제 물리 삭제가 아니라 `status = deleted`, `deleted_at`, 숨김 상태 전환이라면 확인문과 도움말에서 공개 화면 제외와 운영 보존 여부를 구분한다. 물리 삭제 또는 상위 도메인 삭제로 하위 글/댓글/파일을 같이 지우는 작업은 처리 대상 수, 감사 로그, 개인정보 export/cleanup, 첨부/본문 이미지 정리 실패 재시도, 자식 댓글 처리 기준을 소유 모듈 문서와 화면 안내에 함께 둔다.

## 텍스트 색상과 크기

관리자 화면의 기본 글자색은 `html`, `body`, `.admin-content`에서 상속되는 `--color-body-color`를 기준으로 한다. 기본 글자 크기는 `--text-body`인 `16px`를 기준으로 한다. 일반 본문, 테이블 셀, 테이블 헤더, 폼 라벨, 필터 라벨, 카드 제목에는 별도 회색 색상이나 작은 글자 크기를 반복해서 지정하지 않는다.

타이포그래피는 primitive 크기 토큰(`--text-xs`부터 `--text-2xl`) 위에 역할 기반 alias를 얇게 둔다. 런타임에서 반복되는 제목과 본문 위계는 가능하면 `.type-display`, `.type-display-fluid`, `.type-page-title`, `.type-page-title-fluid`, `.type-section-title`, `.type-card-title`, `.type-body`, `.type-small`, `.type-meta`, `.type-caption`, `.type-code` 또는 같은 역할 토큰을 사용한다. 관리자 런타임은 `modules/admin/assets/common.css`, 공개 런타임은 `assets/reset.css`에 역할 class를 가지고, 실제 값은 공개는 `assets/reset.css`, 관리자는 `modules/admin/assets/tokens.css`의 `--type-*-size`, `--type-*-line-height`, `--text-strong`, `--text-muted`, `--text-subtle`에서 가져온다.

관리자 실제 화면의 구조 ID나 화면 고유 클래스는 배치, 간격, 경계선, 상태 예외를 맡고, 텍스트 크기와 줄 높이는 가능한 한 `type-*` 역할 클래스로 표시한다. 예를 들어 현재 페이지 제목은 `#container_title.type-page-title`, 페이지 보조문구는 `#container_subtitle.type-small`로 출력한다. `#container_title`과 `#container_subtitle` 스타일은 제목 아래 선, 여백, 색상 예외처럼 shell에 묶인 표현만 담당한다. `form-help`, `admin-summary-meta`, `admin-list-summary`, `admin-dashboard-meta` 같은 재사용 `admin-*` 클래스와 `filtering-label` 같은 UI kit 컴포넌트 클래스는 UI kit 토큰을 소비하는 컴포넌트 규칙으로 유지한다. 이때 색상은 `--text-muted/subtle`, `--color-*`, 배경은 `--color-default-*`나 `color-mix()` 기반 토큰 조합, 포커스 링은 `--color-primary-ring`처럼 light/dark 값을 가진 토큰, 타이포는 `--type-meta-*` 또는 `--type-caption-*` 같은 역할 토큰을 기준으로 한다. 설정 화면처럼 페이지 제목을 더 작게 쓰는 관리자 전용 예외는 `--admin-page-title-compact-*`처럼 admin 레이어의 명시 토큰으로 둔다.

UI-KIT 미리보기 layout CSS는 토큰 표와 샘플의 배치만 담당한다. 실제 관리자/공개 화면에서 쓰는 제목, 본문, 도움말, 메타, 캡션 값은 `admin.css`, `module.css`, `ui-kit.css`, 화면별 stylesheet 같은 런타임 스타일시트가 역할 토큰을 소비해 정의한다.

역할별 기준은 다음과 같다. `Display`는 공개 hero나 UI-KIT 미리보기처럼 매우 큰 강조가 필요한 위치에만 사용한다. 유동형 hero 제목처럼 화면 폭에 따라 커지는 값은 직접 최대값을 쓰지 말고 `--type-display-max-size` 같은 역할 토큰을 함께 사용한다. `Page title`은 화면이나 콘텐츠의 주 제목, `Section title`은 카드 묶음이나 설정 구역 제목, `Card title`은 카드 내부 제목에 사용한다. 공개 화면의 유동형 페이지 제목 최대값은 `--type-page-title-fluid-max-size`를 사용한다. `Body`는 기본 본문, `Small`은 보조 설명, `Meta`는 날짜·상태·식별자 같은 낮은 우선순위 정보, `Caption`은 표·이미지·예시의 캡션, `Code`는 key, 토큰명, 짧은 식별자에 사용한다. 관리자 폼의 `form-help`와 hint 계열 도움말은 더 조밀한 폼 흐름을 위해 `--text-xs`인 12px 기준을 사용한다.

`--font-weight-medium`은 `500`, `--font-weight-semibold`는 `600`, `--font-weight-bold`는 `700`으로 구분한다. 과거처럼 medium과 semibold가 같은 값이면 카드 제목, 라벨, 강조 텍스트의 위계가 UI-KIT에서 구분되지 않으므로 새 기준에서는 분리한다.

`/admin/ui-kit`와 `/ui-kit`의 Typography 섹션은 한글, 영문, 숫자, 긴 식별자, 테이블 내부 텍스트를 함께 보여준다. 공통 타이포그래피를 바꿀 때는 두 화면에서 관리자 조밀 화면 기준과 공개 콘텐츠 기준을 같이 확인한다. 특정 모듈이 별도 UI-KIT를 제공하면 모듈 CSS가 실제로 소비하는 역할 토큰도 그 화면에서 확인한다.

옅은 색상은 의미가 분명한 보조 텍스트에만 사용한다. 예를 들어 `hint-text`, `muted-text`, `form-help`, `card-subtitle`, 빈 상태 안내, 보조 메타 정보처럼 본문보다 우선순위가 낮은 텍스트가 여기에 해당한다. 성공, 경고, 위험, 링크, 활성 탭처럼 상태나 상호작용을 나타내는 색상은 예외로 유지한다.

`var(--text-xs)`, `var(--text-sm)`, `.8125rem`처럼 기본보다 작은 크기는 화면 밀도나 정보 계층이 분명한 경우에만 사용한다. 도움말, 힌트, 보조 메타, 상태 배지, 사이드바 보조 메뉴, 테이블 안의 부가 정보가 여기에 해당한다. 새 관리자 화면의 본문과 입력 라벨은 먼저 16px 상속을 기준으로 확인한다.

관리자 런타임의 기본 입력 필드는 class 없는 `input`, `select`, `textarea`가 아니라 `.form-input`, `.form-select`, `.form-textarea`, `.form-checkbox`, `.form-radio` 시맨틱 클래스를 사용한다. 이 클래스들은 다른 요소의 자식 선택자가 아니라 `admin.css`의 직접 선택자로 스타일을 소유한다. class 없는 필드 선택자는 기존 화면 호환을 위한 fallback으로만 남긴다.

## 사이드바

사이드바는 모듈의 `admin.category_label`을 라벨로만 표시하고, 실제 조작 가능한 메뉴는 `모듈 그룹 > 메뉴 항목`의 2단계 구조를 기본으로 유지한다. 단, 하위 메뉴 항목이 하나뿐인 모듈 그룹은 그룹 자체를 해당 항목 링크로 렌더링해 바로 화면을 연다. 라벨은 시스템, 회원, 사이트, 커뮤니티, 운영처럼 메뉴를 구분하는 시각적 구획일 뿐이며 접기/펼치기 버튼이나 링크로 사용하지 않는다. 루트 경로에서 열 초기화면 후보 선택은 사이트 설정의 `화면` 섹션에서 다룬다.

시스템 분류의 관리자 기본 메뉴 순서는 대시보드, 환경설정, 모듈·플러그인 관리, 업데이트, 데이터 정리, 관리자 메뉴, 관리자 권한, 관리자 작업 로그 순서로 둔다.

관리자 권한 화면은 필터 검색 목록이 아니라 이미 소유자 또는 메뉴 권한이 부여된 회원 목록을 기본으로 보여준다. 목록은 공개 해시와 이메일을 한 셀에 줄바꿈으로 표시하고, 해당 셀의 정렬은 이메일 기준으로 처리한다. 목록의 현재 권한 칸은 UI kit의 `badge-list` 패턴으로 소유자 또는 대표 메뉴를 먼저 보여주고, 조회/수정/삭제 개수는 보조 요약으로 표시한다. `/admin` 진입 권한은 소유자이거나 메뉴별 관리자 권한을 하나 이상 가진 상태를 뜻하고, 실제 하위 메뉴 접근은 각 메뉴의 개별 조회/수정/삭제 권한으로 다시 제한한다. 목록 조회와 상세 조회는 `view`, 새로 만들기/수정 폼 진입과 저장·상태 변경은 `edit`, 명시적 삭제는 `delete` 권한으로 보호한다. 소유자가 아닌 제한 관리자가 `/admin`에 접근하면 대시보드를 보여주지 않고 첫 번째 허용 메뉴로 이동시키며, 관리자 대시보드와 UI-KIT은 소유자 전용으로 둔다. 관리자 UI-KIT 상단에는 초기 UI Kit과 설치된 콘텐츠/커뮤니티/퀴즈/설문 모듈의 사용자 화면 UI Kit 링크만 제공하고, 콘텐츠/커뮤니티/퀴즈/설문 UI Kit은 `/content/ui-kit`, `/community/ui-kit`, `/quiz/ui-kit`, `/survey/ui-kit` 사용자 화면에서 현재 공개 레이아웃 설정을 적용해 확인한다. 해당 링크는 각 모듈의 사이드바 서브메뉴에 노출하지 않는다. 새 권한 부여는 `권한 추가` 모달에서 시작하고, 회원 검색은 별도 스택 모달로 열어 선택 후 권한 추가 모달로 돌아온다. 전체 및 목록 관리 셀의 `권한 추가` 버튼은 보조 아웃라인 텍스트 버튼으로 표시한다. 목록 관리 셀의 변경/회수 액션은 아이콘 버튼으로 제공하고, 목록 관리 셀의 `권한 추가` 버튼은 해당 회원을 미리 지정한 상태로 권한 추가 모달을 연다. 권한 추가와 권한 변경 모달은 별도 패널 카드 구조가 아니라 일반 관리자 모달처럼 `form-row`와 `form-field` 흐름으로 정돈한다. 신규 권한은 1차 분류, 2차 메뉴, 조회/수정/삭제 권한을 선택한 현재 상태를 저장 시점에 적용하며, 이미 메뉴 권한을 가진 회원에게는 기존 메뉴 권한을 유지한 채 새 선택 권한을 더한다. 수정이나 삭제 권한은 조회 권한을 전제로 하므로 프론트에서는 수정/삭제 선택 시 조회를 자동 선택하고, 서버 저장 시에도 조회 권한을 함께 정규화한다. 권한 변경 모달은 대상 회원 요약, 소유자 권한, 대체 메뉴 권한 선택기, 선택된 메뉴 권한 영역을 분리하고, 기존에 존재하는 메뉴별 행에서 조회/수정/삭제 체크박스와 제거를 조정하거나 선택기로 새 메뉴 권한 행을 추가한다. 기존 소유자의 소유자 권한을 해제해 제한 관리자로 전환하는 저장 요청은 하나 이상의 대체 메뉴 권한이 있어야 하며, 모든 관리자 권한을 비우는 작업은 `권한 회수` 버튼으로 `revoke_permission` 요청을 보낸다. 1차 분류나 2차 메뉴에서 `전체`를 선택하면 해당 범위의 모든 메뉴에 같은 작업 권한을 적용한다. 새 관리자 권한은 정상 상태 회원에게만 부여할 수 있고, 비활성 회원의 기존 권한 회수와 축소는 허용한다. 소유자는 모든 관리자 권한을 가진 상태이므로 메뉴별 권한을 함께 저장하지 않으며, 기존 소유자에게 메뉴별 권한을 추가하려는 요청은 서버에서 거부한다. 소유자가 2명 이상이면 목록에서 소유자 행에도 `권한 회수` 버튼을 표시할 수 있지만, 저장 시점에는 소유자 행을 잠근 뒤 다시 확인해 마지막 소유자나 마지막 활성 소유자가 사라지는 동시 실행을 서버에서 막는다. 권한 추가와 권한 변경 모달의 항목 도움말은 부모 모달을 유지한 채 위에 스택 모달로 연다. 권한 추가 요청은 소유자 권한 또는 하나 이상의 메뉴 권한이 있어야 하며, 프론트 검증과 서버 POST 검증이 모두 같은 조건을 막아야 한다. 목록의 소유자가 아닌 회원은 행 단위 `권한 회수` 버튼으로 메뉴 권한을 즉시 비울 수 있다.

관리자 사이드바 브랜드는 로고 매니저의 `관리자 사이드바 로고`가 있으면 확장 상태에서 사이트명 텍스트 없이 로고만 표시한다. 접힌 상태에서는 활성 `public.app_icon` 앱아이콘이 있으면 앱아이콘을 우선 표시하고, 앱아이콘이 없을 때 관리자 사이드바 로고를 원본 비율 `contain`으로 표시하며, 별도 검정 박스 배경을 입히지 않는다. 관리자 사이드바 로고가 없을 때만 앱아이콘 후보와 사이트명 텍스트를 함께 표시하고, 해당 자산도 없으면 검정 사각형 안에 사이트명 첫 글자를 표시한다. 공개 기본/콘텐츠/커뮤니티/퀴즈/설문 레이아웃의 브랜드 로고는 `사용자화면 로고(PC)`와 `사용자화면 로고(모바일)`을 분리해 쓰되, 한쪽만 등록되어 있으면 다른 화면에서도 같은 로고를 fallback으로 사용한다. 모바일 폭에서는 모바일 로고가 PC 로고보다 우선이고, 모바일 로고가 없을 때 PC 로고를 사용한다. 공개 헤더 전용 로고가 둘 다 없고 활성 `public.app_icon` 로고의 `사용자 화면 심볼로 사용`이 켜져 있으면 앱아이콘 심볼과 사이트명을 브랜드 fallback으로 함께 표시하며, 해당 로고의 링크 URL이 있으면 브랜드 링크에도 사용한다.

데스크톱 사이드바 접기/펼치기 버튼은 상단바 왼쪽에서 breadcrumb 바로 앞에 둔다. 모바일 메뉴 열기 버튼도 같은 영역에서 breadcrumb 앞에 두지만, 데스크톱 폭에서는 숨긴다.

접힌 사이드바에서 하위 메뉴가 없는 메인 메뉴는 메뉴명 커스텀 툴팁을 제공한다. 하위 메뉴가 있는 메인 메뉴는 툴팁 대신 팝업 서브메뉴 목록 맨 위에 메인 메뉴명을 굵은 텍스트로 표시해 현재 목록의 맥락을 드러낸다.

접힌 사이드바의 현재 활성 메인 메뉴는 아이콘만 보이더라도 위치를 인지할 수 있도록 회색 계열 배경과 라운드 처리를 적용한다.

저장된 접힘 상태를 페이지 진입 시 복원할 때는 초기 복원 중 transition을 비활성화해 콘텐츠 영역이 오른쪽에서 왼쪽으로 자라나는 것처럼 보이지 않게 한다. 사용자가 접기/펼치기 버튼을 직접 누를 때의 전환 효과는 유지한다.

`/admin/menu`에서 저장한 순서, 숨김 설정, 모듈 그룹 아이콘 설정은 기본 선언보다 우선한다. 이 화면은 회원별 권한으로 위임하지 않고 소유자만 접근할 수 있게 둔다. 다만 번들 메뉴 구조를 정리하면서 생긴 과거 기본 순서값은 사용자 커스텀 순서로 보지 않고 새 기본 순서를 따른다. 시스템 분류, 관리자 모듈 그룹, 관리자 메뉴 항목은 운영자가 관리자 메뉴 설정 화면에 다시 접근할 수 있어야 하므로 숨김 처리할 수 없다.

관리자 메뉴 표시 설정 화면은 페이지 제목 아래에 드래그, 위아래 이동, 접기/펼치기, 저장 방식을 짧게 안내한다. 표 상단 카드 헤더에는 제목 문구 대신 좌측에 보조 아웃라인 토글 버튼 하나를 두고, 접힌 항목이 있으면 `전체 펼치기`, 없으면 `전체 접기`로 동작하게 한다. 분류, 모듈 그룹, 항목은 같은 저장 표에서 다루되 대상 칸에서 들여쓰기와 보조 경로로 위계를 드러낸다. 모듈 그룹 행은 허용된 기본 심볼과 `/admin/settings` 공용 아이콘 설정에서 추가한 아이콘 키 중 하나를 선택해 사이드바 아이콘을 오버라이드할 수 있고, 빈 값은 모듈 선언 또는 분류 기본 아이콘을 사용한다. 순서 드래그는 같은 범위와 같은 부모 안에서만 가능하게 유지하고, 이동 가능한 위치에만 고스트 라인을 표시한다. 이동 버튼도 드래그 가능한 같은 범위 안에서만 위아래로 움직이며, 분류나 모듈 그룹을 움직이면 하위 행도 같은 블록으로 함께 이동한다. 하위 항목 접기/펼치기는 긴 목록 편집을 돕는 브라우저별 화면 상태이며, 각 행의 접기/펼치기 버튼도 보조 아웃라인 버튼으로 표시한다. 전체 펼치기/접기 버튼은 이 화면 상태를 한 번에 바꾼다. 저장 데이터나 하위 행의 포함 관계는 바꾸지 않는다. 저장 후 다시 펼칠 때도 부모와 자식 행이 분리되지 않아야 한다. 저장/초기화 POST 결과는 새로고침 때 같은 토스트나 저장 작업이 반복되지 않도록 flash 결과와 redirect로 한 번만 표시한다. 저장/초기화 액션 영역은 긴 메뉴 목록에서도 접근 가능하도록 화면 하단에 sticky로 유지한다. 기본값 초기화는 메뉴 표시 순서, 숨김 설정, 모듈 그룹 아이콘 설정이 모두 초기화된다는 안내에 동의한 경우에만 처리한다.

관리자 UI-KIT은 일반 메뉴 트리에 넣지 않고 사이드바 메뉴 목록 아래쪽의 보조 링크로 노출한다. UI-KIT은 실제 업무 메뉴가 아니라 관리자 UI 점검 화면이므로 메뉴 순서/숨김 오버라이드 대상과 분리한다.

관리자 메뉴 아이콘의 기본 표현 선택권은 각 모듈에 둔다. 모듈은 `module.php`의 `admin.icon`에 관리자 shell이 제공하는 심볼 이름을 선언하거나, 자기 모듈의 `assets/` 아래에 둔 `jpg`, `jpeg`, `png`, `gif`, `webp` 파일을 선언할 수 있다. 관리자 shell은 선언을 그대로 믿지 않고 허용된 심볼과 모듈 내부 자산 경로만 렌더링하며, 선언이 없거나 유효하지 않으면 카테고리 기본 아이콘으로 fallback한다. 운영자가 `/admin/menu`에서 모듈 그룹 아이콘을 선택하면 해당 공용 아이콘 키 오버라이드가 모듈 기본 아이콘보다 우선한다.

관리자 메뉴 심볼 이름 목록과 Material Symbols 매핑은 `modules/admin/helpers/icons.php`의 공통 계약이 소유한다. admin skin은 심볼 이름을 직접 해석하지 않고 helper 매핑을 통해 Material 아이콘 이름으로 변환한다. 기본 표현은 Material Symbols다.

모듈 메뉴에 허용되는 기본 심볼 이름은 `settings`, `admin-mode`, `users`, `user`, `content`, `stats`, `home`, `folder`, `image`, `layers`, `search`, `menu-list`, `bell`, `shield`, `coins`, `wallet`, `gift`, `message-circle`, `service`, `quiz`, `poll`, `emoji_emotions`, `integration_instructions`이다. 툴바 전용 심볼은 같은 helper 계약에 있더라도 `module_menu`가 꺼져 있으면 모듈 선언에서 사용할 수 없다. 운영자가 `/admin/settings`에서 추가한 공용 아이콘 키는 모듈 선언 기본값이 아니라 `/admin/menu`의 운영자 오버라이드 선택지로 사용한다.

아이콘은 메뉴 텍스트를 보조하는 장식 요소로 취급한다. 접근 가능한 이름은 메뉴 label이 맡고, 이미지 아이콘도 `alt=""`와 `aria-hidden="true"`로 출력한다.

리액션 관리 모듈은 성격이 다른 작업을 한 화면에 몰아넣지 않고 `리액션 정의`, `Preset 관리`, `레코드 점검`을 같은 모듈 그룹의 별도 하위 페이지로 둔다. 정의 화면은 reaction key, 표시명, 아이콘 유형, 색상, 사용 중지 key 레코드 처리 정책을 다루고, 아이콘 유형은 이모지, Material icon key, 직접 업로드한 JPG/PNG/WebP 이미지 중 선택하게 한다. 이미지 업로드 저장이 실패하거나 정의 저장 검증이 실패하면 새로 업로드한 이미지를 남기지 않아야 한다. preset 화면은 공개 버튼 묶음과 표시 개수를 다루며, 레코드 점검 화면은 회원/대상/key 기준 조회와 대상 상태 확인만 담당한다.

모듈 전역 설정은 가능하면 목록 화면 안에 배치하지 않고 독립 설정 화면으로 둔다. 이때 독립 도메인 관리 화면을 가진 모듈은 `admin-menu.php`에 `목록`, `설정`처럼 같은 모듈 그룹 안의 별도 항목을 선언하고, 설정 저장 POST는 설정 화면의 action이 소유하게 한다. CKEditor처럼 다른 화면의 입력 경험을 보강하는 플러그인은 도메인 정책을 자기 설정에 두지 않고, 필요한 경우 `플러그인` 같은 별도 분류에서 런타임 자산 설정 화면만 제공한다.

## 사이트/모듈 설정 화면

사이트 설정 화면은 사이트 이름, 시간대, locale, 운영 상태, 회원 전용 모드, 사이트 기본 메타, 사업자 정보, 공개 화면 레이아웃, 초기화면, 관리자 UI 색상 모드, 관리자 스킨처럼 운영자가 의미를 이해하고 직접 변경할 수 있는 전용 필드만 다룬다. 시간대와 locale은 자유 입력이 아니라 사용 가능한 값 중에서 선택하는 select로 제공하고, 지원 언어 목록은 체크박스 토글로 표시하며 최소 한 개 이상 선택하도록 프론트엔드와 서버에서 함께 검증한다. 기본 언어로 선택된 locale은 지원 언어 목록에서 항상 체크하고 해제할 수 없게 표시하며, 서버 저장 시에도 기본 언어를 지원 목록에 다시 포함한다. 기본 언어는 번역 fallback, 메일/알림 문구, 신규 회원 locale의 기준이므로 필수값으로 둔다. 설치 시 선택한 기본 통화는 읽기 전용으로 표시하고 일반 설정 POST로 저장하지 않는다. 이 값은 신규 가격/정책 row의 기본값과 통화가 빠진 자산 구매력 계약의 fallback 기준이며 기존 가격, 로그, 구매력 snapshot을 변환하는 스위치가 아니라는 도움말을 함께 둔다. 기본 통화는 설치 초기에 잘못 선택한 경우를 복구할 수 있도록 owner에게만 사이트 설정 sticky submit 영역 좌측의 secondary outline `통화 변경` 버튼을 보여주고, `/admin/settings/currency` 별도 페이지에서 모달로 호출해 변경한다. 별도 페이지에는 문장형 안내와 가격/정산 통화 row 및 자산 구매력 영향 요약을 기본 표시하고, 모달 form에는 새 통화, 변경 사유, 현재/대상 통화 확인 문구, 현재 비밀번호 입력만 둔다. 서버는 owner 권한, CSRF, 현재 비밀번호, reauth throttle, 확인 문구, 감사 로그 metadata를 다시 검증한다. 사이트 기본 메타는 `site.title_suffix`, `site.meta_description`, `site.og_image`로 저장하며 화면별 메타가 비어 있을 때만 공개 출력에서 fallback으로 사용한다. 사업자 정보는 `site.business_info_items` JSON 설정으로 저장하며, 상호, 대표자명, 사업자등록번호, 사업장 주소, 고객센터 전화번호, 개인정보보호책임자, 개인정보보호책임자 이메일 기본 행과 운영자가 추가한 임의 항목명을 함께 다룬다. 통신판매신고업번호처럼 필요한 특수 항목은 기본 행이 아니라 운영자가 추가 항목으로 입력한다. 서버는 항목명/값 길이와 중복 항목명을 검증하고, 범용 site setting key/value 편집 화면으로 확장하지 않는다. 회원 전용 모드는 운영 상태와 같은 사이트 운영 섹션에 두고, 켜면 비로그인 방문자가 공개 사이트 화면보다 로그인 화면을 먼저 본다는 도움말을 둔다. 공개 레이아웃과 초기화면은 `화면` 섹션에 두되, 회원 전용 모드가 켜져 있으면 초기화면보다 로그인 화면이 우선한다는 안내를 함께 표시한다. 관리자 UI 색상 모드와 관리자 스킨은 `관리자 화면` 섹션에 둔다. 팝업레이어 본문 에디터처럼 특정 모듈의 입력 경험은 해당 모듈 환경설정에서 관리한다. 화면 상단의 다크 모드 전환 버튼은 admin 모듈 설정의 `admin_color_scheme` 값을 라이트/다크로 즉시 저장하는 빠른 전환이며, 같은 브라우저에서는 공개 화면 색상 모드와 같은 `sr_public_color_scheme` 저장값을 사용해 서로 호환된다. `sr_site_settings`는 내부 저장소로 유지하지만, 임의 key/value를 추가·삭제하는 범용 관리자 화면은 제공하지 않는다.

코어나 모듈이 사이트 범위 설정을 추가로 필요로 하면 먼저 명확한 전용 관리자 화면, 설정 schema, 업데이트 스크립트, 또는 소유 모듈의 설정 화면을 만든다. 기능 플래그나 외부 연동 옵션처럼 운영자가 판단해야 하는 값도 의미, 허용 값, 검증 규칙이 드러나는 전용 UI로 승격한다. 범용 key/value UI로 전용 화면 책임을 우회하지 않는다.

전용 UI로 승격할 기준:

- 설정 의미를 설명해야 한다.
- 값 형식이 복잡하다.
- 실수하면 운영 장애가 난다.
- 여러 필드가 함께 저장되어야 한다.
- 모듈 도메인 정책과 연결된다.

전용 UI가 있는 모듈 전역 설정은 목록/검색/행 관리 화면에 섞지 않고 별도 설정 화면과 저장 action으로 분리한다. 예를 들어 배너 기본 스킨은 `/admin/banners/settings`, 팝업레이어 기본 스킨은 `/admin/popup-layers/settings`가 소유한다.

공유 대상 삭제, 비활성화, key 변경, 사이트명 변경처럼 다른 모듈 정책에 영향을 줄 수 있는 작업은 읽기 참조 계약 결과를 운영자에게 보여줄 수 있어야 한다. 참조 현황 버튼과 모달은 `sr_admin_read_reference_button_html()`과 `sr_admin_read_reference_modal_html()`을 사용해 행 액션이나 관련 설정 필드 근처에 둔다. 설정 입력의 기본 폭을 줄이지 않아야 하므로 사이트명처럼 긴 입력은 참조 버튼을 input group 안에 넣지 않고, 같은 줄의 별도 액션으로 둔다. 링크는 내부 상대 관리자 URL만 사용하고, 서버 POST는 화면 표시나 JavaScript 확인과 별개로 최신 참조 결과와 계약 오류를 다시 확인한다. 참조가 있거나 계약 오류가 있으면 소비 모듈 값을 자동으로 바꾸지 않고 저장을 중단하거나 소비 모듈 관리자 화면으로 이동하게 한다.

## 페이지 타이틀

관리자 페이지의 주 타이틀은 admin skin이 출력하는 `#container_title` 하나를 기준으로 스타일링한다. 각 모듈 view는 `$adminPageTitle` 값만 지정하고, 페이지 최상단 `h1`을 별도로 반복하지 않는다.

관리자 목록은 `전체보기` 같은 별도 초기화 버튼을 타이틀 옆이나 본문 상단에 두지 않는다. 해당 목록의 첫 진입 상태로 돌아가는 동작은 필요한 경우 `$adminPageTitleUrl`로 페이지 타이틀 자체에 항상 연결한다.

## 폼 화면

등록, 수정, 설정처럼 화면의 주 작업이 폼인 페이지는 실제 view 마크업에서 다음 구조를 사용한다.

```php
<form method="post" action="..." class="admin-form ui-form-theme">
    <?php echo sr_csrf_field(); ?>

    <section class="card">
        <h2>기본 정보</h2>
        <div class="form-row">
            <label class="form-label" for="example_title">이름</label>
            <div class="form-field">
                <input id="example_title" type="text" name="title" required>
            </div>
        </div>
    </section>

    <div class="form-sticky-actions form-actions form-actions-split">
        <a href="..." class="btn btn-surface-default-soft">목록</a>
        <button type="submit" class="btn btn-solid-primary">저장</button>
    </div>
</form>
```

- 폼의 주 섹션은 `form > section.card`로 둔다.
- 섹션 제목은 `h2`를 사용한다. 카드 헤더가 별도로 필요하면 view에서 `card-header`, `card-title` 마크업을 직접 출력한다.
- 일반 입력, 셀렉트, 텍스트 영역처럼 한 행에 주 컨트롤이 하나인 경우 `form-row` 바로 아래에 `label.form-label`을 두고, `form-field` 바로 아래에 컨트롤을 둔다. 이때 `label`의 `for`와 컨트롤의 `id`를 연결하고, 숨김 라벨을 감싼 중첩 `label`은 만들지 않는다.
- 저장 폼에서 상태, 색상 모드, 에디터, 정책, 수수료 방식처럼 2~4개 안팎의 고정된 단일 선택값은 셀렉트보다 `sr_admin_radio_toggle_group_html()` 라디오 토글을 우선 사용한다. 시간대, locale, 레이아웃, 스킨, 메뉴, 회원 그룹, 게시판, 쿠폰, 배너/팝업 대상, 자산 항목처럼 길거나 동적으로 바뀌는 목록은 셀렉트를 유지한다.
- 목록 행에서 즉시 상태를 바꾸는 관리 액션은 `select + 저장/변경` 조합을 쓰지 않고, 가능한 다음 상태를 결과 라벨이 적힌 submit 버튼으로 직접 표시한다. 현재 상태 버튼은 숨기고, 삭제·거절·취소·실패·분석 제외처럼 되돌리기 어렵거나 위험한 결과는 확인 문구를 붙인다. 메모가 필요한 행은 메모 입력을 유지하되 상태 값은 버튼 또는 hidden POST 값으로 전달하고, 저장 action은 기존 allowlist와 권한 검증을 그대로 적용한다.
- 켜기/끄기 성격의 단독 관리자 설정은 `sr_admin_switch_html()` 또는 직접 작성한 `.form-switch`를 사용한다. 여러 값을 동시에 고르는 묶음은 일반 체크박스 목록이나 체크박스 토글 그룹을 사용하고, 단독 on/off를 체크박스 토글 그룹처럼 버튼 하나로 표시하지 않는다. 확인 문구, 삭제 확인, 개인정보 처리 확인처럼 신중한 동의가 필요한 항목은 일반 체크박스를 유지한다. 일반 체크박스나 셀렉트를 JavaScript로 자동 치환해 토글 스타일로 바꾸지 말고, 각 view가 저장 의미에 맞는 컨트롤 helper를 직접 출력한다.
- 입력 항목 라벨에는 필드 이름만 둔다. URL 동작, 허용 형식, 업로드 제한, 자동 처리 같은 설명 문구를 괄호로 라벨에 붙이지 말고 입력 컨트롤 아래 `p.form-help` 또는 `small.form-help`로 분리한다. 설정 source 선택처럼 행 오른쪽에 보조 선택지가 함께 있으면 입력 컨트롤과 `form-help`를 한 묶음처럼 왼쪽에 두고, source 선택지는 그 묶음의 높이 기준으로 오른쪽 끝에 정렬한다. `초`, `bytes`처럼 단순 단위만 안내하는 경우에는 도움말 문구를 만들지 않고 `input-group`과 `input-group-text`로 입력 컨트롤 뒤에 단위를 붙인다.
- 좌측 `form-row-label` 또는 행 제목 역할의 `form-label`은 굵게 표시하되, 우측 `form-field` 안의 `label`은 모두 기본 두께로 둔다. 체크박스, 라디오, 스위치처럼 조작 항목 옆에 반복되는 문구까지 굵게 만들지 않는다.
- 모달 본문 안의 `form-row` 왼쪽 레이블은 항목명 축을 빠르게 훑을 수 있도록 굵게 표시한다. 오른쪽 필드 내부의 보조 라벨은 기본 두께를 유지하고, 모달 제목, 섹션 제목, 필수 표시의 강조도 유지한다. 모달 안 단독 on/off 스위치는 라이트 스위치 색상 기준을 따른다.
- 운영자가 직접 입력하는 `*_key`, `module_key`, `menu_key` 같은 key 입력은 첫 글자 영문 소문자와 이후 소문자, 숫자, `_`만 입력되게 `data-admin-key-input`, `pattern="[a-z][a-z0-9_]{1,59}"`, `inputmode="latin"`, `autocapitalize="none"`, `spellcheck="false"`를 함께 둔다. 허용되지 않는 키를 누르거나 붙여넣으면 공통 관리자 JS가 값을 막거나 정규화하고 "영문, 숫자, 밑줄만 입력 가능합니다." 계열의 validation 문구를 표시한다. 저장 action은 같은 값을 `strtolower(trim(...))`로 정규화한 뒤 각 도메인의 key 검증 함수를 적용한다. 내부 관리용 key처럼 운영자가 반드시 직접 기억할 필요가 낮은 신규 입력은 key 입력칸을 유지하되 `data-admin-key-suggest-source`, 필요 시 `data-admin-key-suggest-fallback`과 `data-admin-key-suggest-scope`를 더해 제목/문항/선택지 입력 후 빈 key에 기본값을 제안한다. 숨겨진 추가 모달이나 반복 입력의 key는 페이지 로드만으로 채우지 말고, 실제 source 입력이 시작된 뒤에만 제안해 사용하지 않은 행이 저장 후보가 되지 않게 한다. slug처럼 하이픈을 허용하는 공개 URL 식별자는 `data-admin-slug-input`과 slug 전용 `pattern`으로 별도 처리한다.
- 회원 로그인 아이디처럼 key는 아니지만 영문 소문자, 숫자, `_`만 허용하는 관리자 입력은 `data-admin-login-id-input`처럼 도메인이 드러나는 속성을 쓰고, 같은 브라우저 속성과 서버 검증을 맞춘다.
- 회원 로그인 아이디 원문은 저장하지 않으므로 회원 수정 화면에서는 기존 로그인 아이디를 표시하거나 변경/해제하는 입력을 두지 않는다. 관리자 회원 생성 화면처럼 새 계정에 처음 부여하는 경우에만 로그인 아이디 입력을 제공한다.
- 필수 입력 항목은 HTML `required` 속성이 아니라 저장 action의 서버 검증을 기준으로 판단한다. 값이 비어 있거나 허용 목록 밖이면 저장이 실패하는 텍스트 입력, 셀렉트, 숫자 입력, 체크박스 묶음은 view 마크업의 해당 label 또는 복합 행의 `span.form-label`에 `<span class="sr-required-label">(필수)</span>`를 직접 넣어 표시한다. 특정 선택값에서만 필수인 항목도 라벨 문구는 짧은 `(필수)`만 사용하고, 조건 변화에 따라 표시와 브라우저 검증을 토글한다. 예를 들어 참조 유형과 참조 ID처럼 둘이 함께 의미를 갖는 짝, 특정 매칭 방식에서만 필요한 subject ID, 정책이 `group`일 때 필요한 회원 그룹 선택, 종결 상태에서 필요한 관리자 메모는 같은 규칙으로 처리한다. 프론트 검증은 서버 검증을 보조하는 층으로 함께 맞추되, 저장 action은 JS 비활성화나 임의 POST에서도 같은 필수 조건을 직접 검증해야 한다. 관리자 shell이나 공통 JavaScript로 필수 표기를 후처리하지 않는다.
- 관리자 등록/수정 폼이나 모달이 브라우저 검증 메시지를 공통 스타일로 보여줘야 하면 view의 해당 `form`에 `data-sr-validate-form`을 직접 붙인다. 필수, 형식, 숫자 범위 같은 컨트롤에는 서버 검증과 같은 의미의 `required`, `pattern`, `min`, `max`와 짧은 `data-validation-message`를 둔다. `modules/admin/assets/admin-shell.js`는 opt-in 폼에 한해 `form-input-invalid`, `form-select-invalid`, `form-textarea-invalid`, `form-choice-invalid`와 `validation-error-note`를 붙이고 `aria-invalid`, `aria-describedby`를 연결한다. 또한 관리자 POST 폼에서 `(필수)` 표시가 보이는 셀렉트/체크박스/라디오 선택 묶음이 비어 있으면 제출 전에 먼저 안내하고 제출을 막는다. 여러 선택 컨트롤 중 하나 이상이면 충분한 복합 행은 `.form-row`, `.admin-setting-unit`, `fieldset`, 또는 직접 필드 wrapper에 `data-admin-required-selection-mode="any"`를 명시한다. 검색 필터, 조회 전용 lookup 폼, 저장 action과 무관한 helper 선택기는 opt-in하지 않는다. 조건부 필수 항목은 view 또는 화면별 스크립트가 표시, disabled, required, `(필수)` 라벨을 함께 토글하고, 공통 검증은 현재 DOM 상태만 표시한다.
- 화면의 셀렉트나 체크박스가 서버로 직접 전송되지 않고 hidden input 배열을 만들어 내는 경우, 서버 검증 기준은 화면 컨트롤이 아니라 최종 POST 결과다. 예를 들어 권한 선택 UI가 `permission_keys[]`를 생성한다면 신규 부여 action은 빈 배열과 허용되지 않은 토큰을 모두 거절해야 한다.
- 선택지 설명이나 판정 기준처럼 본문에 두기에는 긴 도움말은 항목명 앞의 `?` 아이콘으로 열리는 설명 모달에 둔다. 도움말 문구는 일반 운영자가 이해할 수 있는 쉬운 말로 쓰고, `fallback`, `canonical`, `locale`, `key`, `slug` 같은 개발자 용어는 꼭 필요할 때만 쉬운 설명과 함께 사용한다. 라벨 버튼은 `sr_admin_form_label_help_html()`, 설명 모달 껍데기는 `sr_admin_help_modal_html()`을 사용해 관리자 화면 간 위치와 접근성 속성을 맞춘다. 도움말이 기존 모달 안에서 열릴 수 있으면 부모 모달을 닫지 않도록 스택 모달로 처리한다. 필수 입력의 라벨 도움말에는 `sr_admin_form_label_help_html()`의 필수 표시 인자를 사용해 `(필수)` 표기를 유지한다.
- 모달 푸터에서 닫기/취소처럼 저장하지 않고 모달을 닫는 버튼은 `modal-action`과 `data-overlay`를 유지하고 `btn-solid-light` 같은 보조 버튼 스타일을 사용한다. 저장/적용 같은 주요 액션이 함께 있으면 공통 CSS가 닫기 계열 버튼을 왼쪽에 붙이고 주요 액션은 오른쪽에 남긴다. 닫기/취소만 있는 모달은 버튼을 오른쪽에 둔다.
- 활성 회원 그룹 중 여러 항목을 선택하는 설정은 그룹 key 직접 입력, 다중 셀렉트, 체크박스 목록 대신 `sr_admin_member_group_key_badge_select_html()` 셀렉트+뱃지 목록을 사용한다. 저장 처리는 뱃지가 만드는 배열 POST를 우선 받되 기존 문자열 입력도 정규화해 과거 데이터나 수동 요청과 호환되게 둔다.
- 체크박스와 라디오처럼 컨트롤 옆 문구가 실제 레이블인 항목은 좌측 설명 칸과 우측 조작 레이블을 view에 직접 둔다. 좌측에는 전체 문구를 그대로 표시하고, 우측의 시각적 레이블은 `허용`, `사용`, `확인했습니다.`처럼 필요한 부분만 보이게 줄인다. 우측에서 생략된 맥락은 `.sr-only` 텍스트로 남겨 보조기기가 전체 의미를 읽을 수 있게 한다.
- 체크박스, 라디오, 여러 입력이 묶인 복합 행처럼 좌측 문구가 특정 입력 하나에 직접 연결되지 않는 경우 `form-row` 바로 아래에 `span.form-label`을 둔다.
- 하나의 행에서 값 컨트롤과 항목별 복사 범위 라디오를 반복 배치할 때는 각 컨트롤을 `admin-setting-unit`으로 감싸 값과 복사 범위가 한 묶음으로 읽히게 한다.
- 같은 폼 화면 안에서 원본 대상, 예상 처리 건수, 부하 등급처럼 읽기 전용 정보를 보여줄 때는 `dl.admin-meta-list`보다 `form-row`와 정적 텍스트를 사용해 입력 행과 같은 구조로 맞춘다. 값은 input처럼 보이게 만들지 않고 `code`, 도움말 텍스트, alert처럼 실제 의미에 맞는 정적 표현을 사용한다.
- 회원의 포인트, 적립금, 예치금처럼 원장 잔액이 바뀌는 기능은 운영자 화면에서 `자산`을 기본 표현으로 쓰지 않는다. 목록/필터/폼 라벨은 맥락에 따라 `포인트/금액`, `처리 항목`, `잔액`, `지급 금액`, `차감 금액`, `환전 항목`처럼 실제 작업을 설명하는 말로 쓴다.
- 기술 값이나 개발 계약이 `asset`, `ledger_asset`, `asset_module`이어도 화면 문구는 운영자가 보는 의미를 우선한다. 예를 들어 보상 공급자 `ledger_asset`은 `포인트/금액`, 단일 잔액 선택은 `포인트/금액 항목`, 잔액 증감 내역은 `거래 내역` 또는 `잔액 변경 이력`으로 표시한다.
- `자산` 표현은 회원 자산 모듈 분류, 감사 로그 이벤트명, 내부 helper/DB 필드, 파일/이미지 asset처럼 기술 범주를 설명해야 할 때만 사용한다. 사용자 입력 라벨이나 도움말에서 불가피하게 `자산`을 쓰면 바로 뒤에 `포인트, 적립금, 예치금 같은 포인트/금액 항목`처럼 쉬운 설명을 붙인다.
- 지급/차감/환전처럼 방향이 중요한 화면은 `포인트/금액`만 단독으로 쓰지 말고 `지급할 포인트/금액`, `차감할 포인트/금액`, `출금 항목`, `입금 항목`처럼 잔액 변화 방향이 드러나게 쓴다. 금액이 음수로 표시되는 요약에는 부호를 함께 보여준다.
- 자산 선택과 사용/과금 체크가 한 항목 안에 함께 있을 때는 자산 선택 영역에 `data-admin-asset-enable-target="#사용_또는_과금_체크박스_ID"`를 둔다. 운영자가 체크박스를 끈 상태에서 자산을 새로 선택하면 공통 관리자 JS가 확인을 받은 뒤에만 사용/과금 체크를 켜고, 취소하면 방금 선택한 자산을 되돌린다. 해당 묶음을 조작한 뒤 사용/과금은 꺼져 있고 자산은 선택된 상태로 저장하려 하면, 저장 시 자산 선택이 실제 동작에 적용되지 않는다는 안내를 확인한 경우에만 제출한다. 과거 저장값이나 기본값도 같은 기준으로 항상 확인해야 하는 화면은 `data-admin-asset-enable-submit-check="always"`를 함께 둔다.
- 복합 차감 자산은 자산 체크박스와 해당 금액 입력을 같은 `input-group` 묶음으로 표시한다. 차감 순서 안내 문구는 `차감 우선순위: 포인트, 적립금, 예치금`처럼 실제 우선순위를 설명하고, `선택한 자산마다 금액을 입력하세요` 같은 중복 입력 지시 문장은 쓰지 않는다.
- 게시판/콘텐츠 편집 화면의 적용 범위 라디오는 화면에는 `단독`, `그룹`, `전체`처럼 짧게 보이고 `적용` 문구는 `sr-only`로 보조한다. `그룹`이나 `전체`를 선택하면 저장 전 의미를 즉시 알 수 있도록 토스트로 같은 그룹/전체 대상에 적용된다는 안내를 보여준다.
- 좌측 레이블이 필요 없는 안내 문장은 입력항목 행으로 만들지 말고 일반 텍스트 문단으로 둔다.
- 저장, 생성, 변경 버튼은 `form-sticky-actions` 안에 두고 문서 흐름 안에서 출력한다. 이 영역은 관리자 화면 하단에 스티키로 붙어 긴 등록/수정/설정 화면에서도 제출 버튼을 유지하되, 상단 바보다 낮은 레이어에 두고 단단한 배경이나 상단 테두리 대신 하단 그라데이션으로 본문이 이어진다는 단서를 남긴다.
- 업데이트 적용처럼 페이지 전체에 영향을 주는 실행 작업은 목록 카드 내부 액션처럼 두지 않고, 대상 목록은 카드에 남기며 확인 체크와 실행 버튼은 카드 밖 `form-sticky-actions`에 함께 우측 배치한다. 확인 체크가 필요한 작업은 같은 POST에서 서버도 확인 값을 다시 검증한다.
- 설정/편집 form 안에서 Enter 기본 제출은 저장 액션이 되게 한다. 초기화, 삭제, 재계산, 만료 처리처럼 작성 중인 값과 별개로 실행되는 보조/위험 작업은 별도 form으로 분리하거나 명시 버튼 흐름으로 처리해 저장 form의 기본 submit 후보가 되지 않게 한다.
- 새 항목 작성 전용 화면은 생성 성공 후 목록 화면으로 redirect하고, 수정 저장은 현재 편집 화면에 남긴다. 생성 성공 안내는 목록 화면의 공통 토스트로 표시한다.
- 저장, 삭제, 적용 같은 POST 처리는 성공 화면을 직접 렌더링하지 않고 `sr_admin_action_result()`와 `sr_admin_redirect_with_result()` 또는 같은 세션 flash 흐름으로 GET 화면에 redirect한다. 새로고침으로 같은 POST가 다시 실행되지 않게 하고, redirect 뒤 GET 화면에서 `sr_admin_pop_flash_result()`와 `sr_admin_feedback_toasts($notice, $errors)`로 짧은 결과를 토스트로 표시한다.
- 검증 오류로 입력 화면에 되돌아갈 때는 오류와 입력값을 세션에 보관한 뒤 GET으로 redirect한다. view는 보관된 입력값을 우선 채우고, 서버 검증 오류는 입력 맥락 가까이에 남기되 단순 성공 안내는 토스트로만 처리한다.
- 모달 안의 확인 문구 입력처럼 제출 전에 정확한 문자열을 요구하는 필드는 `setCustomValidity()`와 `validation-field`, `form-input-invalid`, `validation-error-note` 조합으로 모달 내부에 즉시 오류를 보여준다. 클라이언트 검증은 제출 전 안내만 맡고, 같은 확인값은 서버 POST에서 다시 검증한다.
- 한 action에서 여러 작업을 처리하면 `intent` hidden 값을 allowlist로 분기하고, 알 수 없는 intent는 저장하지 않고 오류 결과로 redirect한다. 모든 intent는 로그인, 권한, CSRF, 서버 필수값 검증을 먼저 통과해야 한다.
- 목록 화면의 검색 폼, 테이블 행의 상태 변경/삭제 폼, 툴바 폼은 페이지 폼 레이아웃을 적용하지 않는다.

체크박스 행 예시는 다음처럼 작성한다.

```php
<div class="form-grid">
    <div class="form-row">
        <span class="form-label">공개 회원가입 허용</span>
        <div class="form-field">
            <label class="form-check form-label">
                <input type="checkbox" name="allow_registration" value="1" class="form-checkbox">
                <?php echo sr_admin_choice_label_html('공개 회원가입 허용'); ?>
            </label>
        </div>
    </div>
</div>
```

회원 설정의 선택 프로필 항목은 기본 고정 항목과 운영자 정의 추가 항목을 같은 목록에서 관리한다. 모든 항목은 위/아래 버튼으로 순서를 조정할 수 있고, 기본 고정 항목은 모달에서 사용 여부와 필수 여부만 수정하며 삭제 버튼을 제공하지 않는다. 운영자 정의 추가 프로필 항목은 같은 목록의 모달에서 Key, 유형, 프로필 화면/관리자 수정 화면 표시 여부, 개인정보 정책을 관리하고, 서버 검증을 통과한 정규화 값과 순서만 저장한다.

## 목록 화면

목록형 화면은 공통 관리자 목록 원형을 기준으로 맞춘다.

- 사이드 메뉴에 없는 하위 흐름이나 필터성 이동 링크는 `admin-local-nav-wrap`과 `admin-local-nav`를 사용한다.
- 목록 상단에 전체/상태별 건수 요약이 필요하면 `admin-local-nav-wrap` 오른쪽에 `admin-summary-stats`와 `admin-summary-meta`를 둔다.
- 목록의 표시 제한, 정렬 기준, 관리 범위가 제목·필터·표 헤더만으로 충분히 드러나면 별도 페이지 안내를 두지 않는다. 운영자가 오해하면 위험한 제약만 `$adminPageSubtitle`로 짧게 둔다.
- 페이지 안내가 여러 문장이면 먼저 줄일 수 있는 문장을 삭제한다. 꼭 필요한 제약이 둘 이상이면 `$adminPageSubtitle`에 문장 배열을 넘겨 문장 단위로 줄바꿈되게 하고, 한 문장 안에서 임의 `<br>`을 넣거나 `admin-notice` 박스로 줄바꿈만 처리하지 않는다.
- 페이지 사용법, 상태 코드 설명, 운영 기준처럼 즉시 조치가 필요하지 않은 안내는 본문 상단에 고정하지 않는다. 오류, 저장 결과, 조건 미충족, 차단 사유처럼 상태성 피드백이나 경고는 `alert` 또는 `admin-notice`를 유지한다.
- 목록 섹션은 `section.card.admin-list-card`를 기본으로 하고, 카드 제목은 `card-header` 안의 `card-title`, 등록/내보내기 같은 화면 액션은 `card-actions`에 둔다.
- 목록 테이블은 목록 섹션 안에서 `table-wrapper`와 `table table-list`를 사용한다. 화면별 추가 class가 필요하면 모듈 범위가 드러나는 이름을 더하되 공통 테이블 원형을 대체하지 않는다. `thead`에는 별도 헤더 전용 class를 붙이지 않고 `table-list`가 헤더와 셀 간격, 보더, 마지막 행 처리를 소유한다.
- 테이블 전체를 감싸는 일괄 작업 폼은 `admin-list-form`을 사용한다.
- 선택 기반 일괄 작업을 제공하는 목록은 첫 열을 선택 체크박스로 두고 폭을 고정한다. 헤더 체크박스는 현재 페이지의 렌더링된 행만 선택/해제하며, 검색 결과 전체 선택은 별도 UI와 서버 계약이 준비된 작업에서만 제공한다. 체크박스의 `value`에는 행 ID를 둘 수 있지만 내부 DB ID를 본문 텍스트로 노출하지 않는다.
- 선택 열을 추가하거나 제거할 때는 `nth-child()` 기반 폭 지정이 제목/주요 정보 열에 잘못 걸리지 않는지 함께 조정한다. 선택, 상태, 날짜, 관리 열은 내용 폭으로 줄이고 제목/이름/메시지 열이 남는 폭을 우선 사용하게 한다.
- 선택된 행이 1개 이상일 때는 목록 상단 또는 테이블 바로 위에 일괄 작업 바를 보여준다. 작업 바에는 선택 수, 허용된 작업 선택 또는 실행 버튼, 실행 전 확인, 선택 해제 버튼을 두고, 선택 수와 실행 가능 상태는 JavaScript로 보강하되 서버 검증을 대체하지 않는다.
- pagination summary 없이 버튼형 일괄 작업만 `admin-list-summary-row`에 둘 때는 화면 전용 class로 `padding-block`과 줄바꿈을 보정해 테이블과 버튼이 붙어 보이지 않게 한다.
- 선택 기반 일괄 작업 POST는 CSRF, 권한, `intent` allowlist, 허용된 `operation_key`, 대상 ID 배열, 대상 존재 여부, 현재 처리 가능 상태를 서버에서 다시 검증한다. 클라이언트가 보낸 선택 수, 진행 수, cursor 값은 표시 보조값으로만 사용하고 완료 판정과 감사 로그는 서버가 재조회하거나 저장한 값으로 남긴다.
- 상태 변경 일괄 작업은 `status = :expected_from` 같은 조건부 상태 전이를 기본으로 하고, 요청 대상 수와 실제 처리 수를 대조해 이미 변경된 행을 건너뜀이나 부분 실패로 표시한다. 삭제, 발송, 승인/반려, 원장 처리처럼 영향이 큰 작업은 처리 대상 수, 성공/실패 수, 실패 사유, 감사 로그 metadata 기준을 작업 사양에 포함한다.
- 선택 기반 작업의 실행 모델은 `즉시 제한형` 또는 `작업 테이블형` 중 하나로 정한다. 고부하 가능 작업이나 검색 결과 전체 선택 작업은 고부하 작업 기준에 따라 진행 모달, 작업 상태 조회, 재시도/중단 가능 여부를 먼저 설계한다. 회원 그룹 재평가처럼 대상 수에 비례해 연산량이 커지는 작업은 바로 실행형 버튼으로 붙이지 않는다.
- 행 단위 관리 버튼은 `admin-table-actions-cell`과 `admin-row-actions` 안에 둔다.
- 행 단위 반복 작업은 가능하면 `btn btn-icon`과 `sr_icon()` 또는 `sr_material_icon_html()` 아이콘으로 표시하고, `aria-label`과 `title`에는 `수정`, `삭제`, `보기`처럼 실제 작업명을 남긴다.
- 행 액션 순서는 빠른 스캔이 가능하도록 읽기/미리보기, 관리/처리, 수정, 복사, 상태 변경, 삭제 순서를 기본으로 한다. 삭제처럼 되돌리기 어렵거나 데이터 정리에 영향을 주는 작업은 danger 계열과 확인 절차를 사용하고, 숨김/미사용/비활성화 같은 상태 전환은 삭제와 구분되는 secondary 또는 warning 계열을 우선 검토한다.
- 새 항목 추가는 행 액션에 섞지 않고 목록 카드 헤더나 화면 상단의 `card-actions`에 둔다. 특정 행을 복제하는 작업은 행 액션에 둘 수 있지만 아이콘, `aria-label`, `title`, 확인 흐름을 다른 목록과 맞춘다.
- 목록 테이블은 공통적으로 `width: 100%`와 `min-width: 100%` 기준으로 카드 너비 안에 먼저 맞춘다. 날짜, 상태, 수량, 액션처럼 짧은 열은 `width: 1%`와 nowrap 기반으로 내용만큼만 차지하게 하고, 제목, 메시지, 설명처럼 긴 열이 남은 폭을 사용하게 한다. 테이블 전체에 임의의 `min-width`를 두면 빈 목록이나 짧은 목록에도 불필요한 가로 스크롤이 생기므로, 폼 컨트롤이나 행 액션처럼 실제 최소 폭이 필요한 요소에만 최소 폭을 둔다. 컬럼을 제거하거나 추가하면 폭 지정과 빈 상태 `colspan`을 같이 조정한다.
- 테이블형 목록의 데이터가 없을 때도 `thead`는 유지하고, `tbody` 안에 전체 열을 덮는 `td.admin-empty-state[colspan]` 빈 행을 출력한다. 카드형 목록, 설정 폼 내부 보조 목록, 공개/회원 화면처럼 테이블 구조가 아닌 영역은 문장형 빈 상태를 사용할 수 있다.
- 폼 내부 선택지 입력 매트릭스처럼 목록 원형이 아닌 보조 테이블은 `table-list` 적용 대상에서 제외한다. 현재 `modules/quiz/actions/admin-quiz.php`의 문항 선택지 입력 테이블 2개는 이 예외에 해당한다. 관리자 화면이 아닌 회원/공개 화면의 테이블도 이 기준의 전수조사 대상이 아니다.
- `/admin/ui-kit`와 `/ui-kit`의 테이블 샘플은 `filtering-plain`, `filtering-card` 필터 예제와 실제 관리자 회원 목록에 가까운 실사용형 테이블 예제를 함께 보여준다. 테이블 샘플은 `table-card`, `table-list`, `table-summary`, `table-sort-*`, `table-status`, `table-actions-cell`을 사용하고, `sample-table-*`, `admin-table-*`, `ui-table-*`처럼 샘플/관리자/화면 접두어가 붙은 테이블 컴포넌트 이름은 쓰지 않는다.
- 목록에서 내부 DB ID를 별도 칼럼이나 `#123` 같은 본문 텍스트로 직접 노출하지 않는다. 행 관리에 필요한 ID는 hidden input, URL 파라미터, `data-*` 속성처럼 동작 식별자로만 사용하고, 운영자가 보는 목록에는 key, 제목, 회원 공개 이름/이메일, 날짜, 상태처럼 업무 식별에 필요한 값을 우선 표시한다. 포인트, 적립금, 예치금 같은 회원 자산 잔액/거래 목록은 회원 공개 해시를 본문에 직접 표시하지 않고 회원 공개 이름/이메일로 식별한다. 잔액 목록의 회원 정보 바로가기는 별도 컬럼으로 두지 않고 관리 컬럼의 회원 정보 아이콘으로 회원 상세를 연결한다.
- 관리자 목록, 검색 결과, 행 요약, 조정 모달의 선택 회원 요약처럼 운영자가 회원을 식별하는 화면 조각은 회원 목록과 같은 표시 기준을 쓴다. 공개 이름은 `sr_admin_member_display_name_preview()`, 이메일은 `sr_admin_member_email_display()`로 출력하고, 입력/수정 폼이나 본인 계정 화면처럼 원문 확인이 필요한 곳만 실제 이름 값을 그대로 보여준다.
- 관리자 작업 로그처럼 표 셀에 긴 JSON, 메타데이터, 원문 로그를 제공해야 하는 경우에는 테이블 안에서 펼치지 않고 `btn` 계열 보기 버튼으로 모달을 열어 확인하게 한다. 테이블 셀은 스캔 가능한 짧은 액션만 남기고, 긴 내용은 모달 본문에서 줄바꿈과 스크롤을 처리한다.
- 목록별 필수/보조/관리용 컬럼 기준은 [관리자 목록 컬럼 기준](admin-list-columns.md)을 따른다. 좁은 화면에서는 이 기준의 핵심 컬럼을 남기고 보조 컬럼을 숨길 수 있다.

목록 위 필터는 테이블 카드 안에서 임의의 문단으로 붙이지 않고, 접힘 상세 조건이 없으면 `form.filtering-form.filtering.filtering-plain.ui-form-theme > .filtering-fields > .filtering-field` 구조를 사용한다. 검색 조건 수가 3개 이상이거나 상세 조건을 접고 펼쳐야 하는 필터는 `form.filtering-form.filtering.filtering-card.ui-form-theme`와 `data-filtering` 구조를 사용하고, 접힘 본문은 `.filtering-body[data-filtering-body]`에 둔다. 상세검색 안의 select는 같은 GET 이름을 유지하는 라디오 토글로 표시한다. 짧은 조건이 여러 개이고 검색어만 남은 폭을 써야 하면 `.filtering-fields-fit`을 기본 조건 묶음에, `.filtering-body-fit`을 상세 조건 묶음에 더해 조건별 fit-content 줄바꿈을 사용한다. 제목이나 초기화 버튼이 필요하면 `.filtering-header`를 먼저 둔다. 필터 초기화는 현재 조회 조건을 비워 전체 조회 상태로 되돌리는 동작이며, 텍스트/날짜 입력은 비우고 select는 첫 옵션으로 돌리며 체크 토글은 `전체` 상태로 맞춘다. 필터가 목록 범위를 바꾸는 조건일 때는 목록 위에 두고, 화면 전체 범위를 바꾸는 조건일 때는 목록 섹션 바깥에 둔다.

CSV 내보내기처럼 현재 필터 조건에 붙는 보조 작업은 검색 submit과 같은 주 액션으로 보이지 않게 별도 보조 액션 묶음으로 분리한다. 필터 초기화 후 기본 선택값을 자동 적용하는 화면은 페이지 subtitle이나 요약 카드 문구로 자동 선택 기준을 알려준다.

필터에서 상태처럼 값이 적고 빠르게 범위를 바꾸는 조건은 `전체`를 포함한 체크 토글 그룹으로 제공할 수 있다. 체크 토글에서 `전체`를 선택하면 나머지 선택은 해제하고, `전체`를 제외한 모든 항목이 선택되면 다시 `전체` 선택으로 환원한다. 체크 토글 helper는 wrapper id와 전체/개별 input id를 서로 다르게 만들어 label 클릭이 항상 실제 input에 연결되게 한다. 검색 대상 선택은 `검색조건` select로 제공하고 `전체`, `해시아이디`, `아이디`, `이름`처럼 검색어의 해석 범위를 고르게 한다. 여러 값을 동시에 조합해 보는 의미가 분명한 조건만 체크박스 목록이나 숨김 체크박스 기반 `btn-group` 선택 버튼으로 제공한다. 체크박스 토글과 라디오 토글 버튼은 일반 버튼보다 좌우 패딩을 좁게 두어 필터 항목이 과하게 넓어지지 않게 한다. 필터 필드명은 `label.filtering-label`로 두고, 한 줄 필터는 데스크톱에서 `fit-content` 또는 `max-content` 기반으로 필요한 폭만 차지하게 하며, 검색어처럼 긴 값만 남은 폭을 사용하게 한다.

관리자 작업 로그처럼 조건 수가 많은 필터는 화면 전용 class로 폭을 조정한다. 기본 검색 조건 select와 검색어 input은 같은 줄에 두고, 상세검색 본문의 결과, 처리자 유형, IP, 기간 같은 보조 조건만 한 줄에 하나씩 배치한다.

회원 검색, 관리자 권한, 회원 자산 잔액/조정/거래 내역처럼 회원을 찾는 필터는 같은 검색 조건/검색어 구조를 사용한다. 계정 상태, 권한, 검색 조건처럼 짧은 select는 내용 폭만 차지하게 두고, 검색어 input이 남은 폭을 사용하며, submit은 기본 버튼 폭을 유지한다.

## 모달 화면

목록 행의 짧은 등록/수정, 규칙 추가, 조정, 상세 원문 확인처럼 현재 목록 맥락 안에서 끝나는 작업은 별도 전체 화면보다 모달을 우선 사용한다. 복잡한 장문 편집, 단계형 작업, 별도 탐색이 필요한 업무는 전체 폼 화면으로 분리한다.

목록에서 짧은 복사, 지급, 상태 변경처럼 현재 행이나 현재 목록 맥락 안에서 끝나는 작업은 행 액션 버튼과 모달을 우선 사용한다. 버튼은 `btn-icon` 또는 기존 목록 버튼 스타일을 쓰고, 모달 본문에는 대상 요약과 서버에서 실제 저장할 필드를 둔다. 모달 POST도 `intent` allowlist, CSRF, 권한, 서버 검증을 통과해야 하며 성공 후에는 새로고침 재제출을 막기 위해 GET으로 redirect하고 토스트로 결과를 표시한다.

- 모달 껍데기는 `modal-overlay modal-overlay-fade overlay hidden pointer-events-none opacity-0`, 내부는 `modal-dialog`와 `modal-content ui-form-theme`를 사용하고 `role="dialog"`, `tabindex="-1"`, `aria-labelledby`, `aria-hidden`, `inert`를 view에서 직접 출력한다.
- 모달은 `modal-header`, `modal-title`, `modal-body`, `modal-footer` 구조를 유지한다. 헤더 닫기 버튼은 시각 스타일을 위해 `btn btn-icon btn-ghost-light`를 쓰고, 스크립트 식별용으로 `modal-close`와 `data-overlay`를 함께 둔다. 저장하지 않는 닫기/취소 버튼에는 `modal-action`과 `data-overlay`를 유지한다.
- 등록/수정 모달의 필수 입력, 도움말, 서버 검증 기준은 전체 폼 화면과 같다. 라벨 도움말은 `sr_admin_form_label_help_html()`, 긴 설명 모달은 `sr_admin_help_modal_html()`을 사용한다.
- 모달 본문의 입력 행은 `.modal-body > .form-row` 구조를 쓰며, 여러 입력 행이 이어질 때 페이지 폼처럼 점선 구분선과 행 안쪽 여백으로 항목을 구분한다. 설명 전용 모달이나 원문 확인 모달처럼 폼 입력이 아닌 본문에는 이 구조를 억지로 적용하지 않는다.
- 그룹 선택, 배지 목록, 도움말이 함께 있는 모달처럼 입력 흐름이 넓은 화면을 요구하면 `modal-dialog modal-dialog-lg`를 우선 쓰고, select와 보조 액션 버튼의 정렬은 화면 전용 class로 제한해 보정한다.
- 모달 안에서 도움말이나 검색 모달을 다시 열어야 하면 부모 모달을 닫지 않도록 스택 모달 흐름을 사용한다. 검색 결과 적용 또는 돌아가기 버튼은 원래 작업 모달을 다시 열어 입력 맥락을 잃지 않게 한다.
- 모달 제출도 서버 POST 후 redirect와 토스트를 기본으로 한다. 저장 오류가 있으면 오류와 입력값을 세션에 보관해 돌아온 GET 화면에서 해당 모달을 다시 열 수 있게 한다.

포인트, 적립금, 예치금처럼 잔액 목록에서 대상 회원이 이미 분명한 경우에는 조정 작업을 독립 화면으로 분리하지 않고 잔액 화면의 행 단위 모달에서 처리한다. 잔액 화면에는 기본 조정 모달 버튼을 항상 노출해 운영자가 회원 공개 해시를 직접 입력할 수도 있게 한다. 환불 처리는 잔액 화면이 아니라 거래 내역 화면의 행 단위 모달에서 처리하고, `refund` 거래 유형과 `refund` 참조 유형을 고정하며 참조 ID에는 원거래 ID를 남긴다. 환불은 조정 모달의 일반 거래 유형으로 제공하지 않으며, 차감 원거래를 선택하지 않은 직접 POST도 서버에서 거부한다. 포인트/적립금/예치금 환불은 차감된 거래를 복원하는 용도로만 노출하고, 원거래의 남은 환불 가능 수량/금액을 넘을 수 없다. 적립금 지급 내역을 되돌릴 때는 환불이 아니라 회수를 사용하고, 예치금을 회원에게 내보낼 때는 원장 환불이 아니라 출금 또는 예치금 환불 신청 처리를 사용한다. 적립금 회수도 거래 내역 화면의 행 단위 모달에서 처리하고, 일반 잔액 조정 모달에는 회수 거래 유형이나 회수 참조 유형을 노출하지 않는다. 회수 거래는 환불 대상으로 제공하지 않는다. 조정/환불/회수 모달의 회원, 거래 유형, 수량/금액, 사유, 참조 항목에는 운영자용 도움말을 둔다. 조정 모달에서 회원이나 참조값을 찾아 넣어야 할 때는 UI-KIT 다중 모달처럼 현재 조정 모달을 검색 모달로 전환하고, 검색 결과 적용 또는 돌아가기 버튼으로 조정 모달을 다시 연다. 최종 저장은 기존 자산 거래 action 제출로 처리한다. 거래 저장에 성공하면 flash 메시지를 남긴 뒤 현재 잔액 또는 거래 내역 GET 화면으로 redirect해 새로고침 중복 실행을 막는다. 서버 검증과 원장 저장은 각 모듈 action이 계속 소유하고, 과거 `/admin/*/adjust` 직접 접근은 같은 회원의 잔액 화면으로 되돌린다.

포인트/금액 환전 정책 화면은 정책 목록을 기본 화면으로 두고, 정책 등록/수정은 모달에서 처리한다. 목록 필터는 상태, 출금 항목, 입금 항목처럼 짧은 조건을 체크박스 그룹으로 제공하고 `fit-content` 폭으로 배치하며, 필터 결과가 없는 경우 조건에 맞는 정책이 없다는 빈 상태를 보여준다. 환전 환경설정 화면은 새 정책 등록 모달의 상태, 비율, 반올림, 환전량 제한, 수수료, 정렬순서 기본값만 저장하며, 저장 시 기존 정책 행을 자동 변경하지 않는다. 환경설정 기본값 항목에는 새 정책 등록에만 적용되고 기존 정책에는 소급되지 않는다는 도움말을 둔다. 정책 모달은 등록일 때 환경설정 기본값을 초기값으로 참고하고, 수정이나 저장 오류 복구 때는 해당 정책/입력값을 우선한다. 정책 모달은 출금/입금 항목, 상태, 비율, 반올림, 환전량 제한, 수수료 적용/기준/금액, 정렬순서처럼 저장 결과와 회원 화면 노출에 영향을 주는 항목에 운영자용 도움말을 둔다. 도움말은 환전 실행 시점의 계산 방식, 비활성 항목 포함 정책의 실행 불가 상태, 수수료 원장 기록 방식을 쉬운 말로 설명한다. 비율 입력은 `분자`/`분모` 같은 수학 용어를 라벨로 노출하지 않고 `환전 비율` 한 항목에 `출금 기준량:입금 환산량` 형식으로 입력하게 한다. 수수료 사용 여부와 적용 대상은 별도 행으로 표시하고, 수수료를 사용하지 않으면 적용 대상과 수수료 설정 입력을 숨기며 서버 저장값도 비운다. 수수료를 적용할 때는 `수수료 방식`에서 정률 또는 정액 중 하나만 고르게 하며, 정률 수수료 입력은 기준값 100을 내부 고정한 퍼센트 숫자 하나만 입력하게 한다.

새 콘텐츠 추가/수정처럼 여러 공개 정책을 한 화면에서 다루는 폼은 Slug, 콘텐츠 그룹, 상태, 레이아웃, 유료 열람, 완료 버튼 포인트/금액 처리, 공개 표시, 다운로드 파일 연결 항목에 운영자용 도움말을 둔다. 도움말 문구는 주소 이름, 공개 여부, 환경설정 기본값과 `그룹`/`전체` 적용의 차이, 포인트/금액 차감/지급 시점처럼 일반 운영자가 실제 저장 전에 판단해야 하는 내용을 쉬운 말로 설명한다. 완료 버튼 포인트/금액 처리처럼 지급/차감 방향을 선택하는 화면은 운영자에게 `자산`보다 `처리 항목`, `포인트/금액`, `잔액` 표현을 우선 사용한다. 지급 방향은 처리 항목을 하나만 선택하게 하고 라디오로 표시하며, 차감 방향만 체크박스로 여러 항목을 선택할 수 있게 한다. 회원 그룹별 적용 선택지는 현재 선택한 처리 항목과 맞는 정책만 보여주고, 뱃지 요약도 같은 항목명과 금액을 표시한다. 그룹별 적용 요약은 차감 방향에서 `-2000`처럼 부호를 표시해 실제 회원 잔액 변화 방향이 드러나게 한다. 다운로드 파일은 전용 관리 화면에서 먼저 등록하고, 콘텐츠 생성/수정 화면에서는 사용 상태 파일을 셀렉트+뱃지 형태로 연결만 하도록 설명한다. 파일 제목, 숨김 상태, 다운로드 과금은 다운로드 파일 관리 화면이 소유한다.

새 배너 추가/수정처럼 노출 위치와 노출 조건을 함께 다루는 폼은 기본 정보, 이미지/텍스트, 노출 위치, 배너 설정 섹션으로 나누고, 링크, 이미지 사용 여부, 이미지 URL, 이미지 업로드, 표시 텍스트/대체텍스트, 서비스, 상세, 노출 범위, 노출 대상 번호, 상태, 스킨, 기간, 정렬처럼 저장 결과 판단이 필요한 항목에 운영자용 도움말을 둔다. 노출 위치는 공용 또는 서비스 선택 뒤 상세 위치를 고르는 2단계 흐름으로 표시하고, 저장값은 기존 `module_key|point_key|slot_key` 구조로 정규화한다. 이미지 배너 스위치를 켜면 이미지 URL 또는 이미지 업로드 중 하나를 필수로 받고, 텍스트 입력은 이미지 대체텍스트로 사용한다. 이미지 배너 스위치를 끄면 이미지 입력은 숨기고 텍스트 입력값을 배너에 표시한다. 기본 노출 범위는 선택한 노출 위치 전체이며, 노출 범위와 노출 대상 번호 입력은 특정 대상 선택을 지원하는 노출 위치에서만 표시한다. 노출 대상 번호는 쿠폰 대상 조회와 같은 검색 모달 흐름으로 채운다. 새 배너 작성 화면은 기본 매칭 방식 설정값과 관계없이 전체 위치 노출 상태로 시작하고, 기존 특정 대상 배너를 수정할 때는 저장된 대상 번호 값을 보존한다.

배너/팝업레이어의 특정 대상 검색 모달은 기존 `items` 응답과 호환하면서 `next_cursor`, `has_more`, `limit`, `notice`를 읽을 수 있어야 한다. 커뮤니티 게시글 대상은 게시판 필터, 상태 필터, 검색어, 20개 단위 결과, 더 보기, 로딩, 빈 결과, 짧은 검색어 안내, 선택된 대상 요약을 제공한다. 텍스트 검색은 2자 이상부터 허용하고, 숫자 검색은 게시글 ID 정확 조회로 처리한다. 선택 후 폼에는 ID만 저장하더라도 운영자가 게시판, 제목, 상태, ID를 확인할 수 있는 요약을 함께 보여준다. 저장 POST는 `exact`일 때 대상 ID 필수 검증과 대상 존재 검증을 서버에서 다시 수행하고, 삭제된 대상은 저장하지 않는다.

로고 매니저의 로고 배치 추가 폼은 로고 용도, 공개 레이아웃 사용처, 이름, 사용 여부, 이미지 파일, 대체 텍스트, 링크 URL, 시작/종료 시각, 정렬을 한 모달에서 저장한다. 로고 용도와 이름, 이미지 파일은 필수이며, 공개 레이아웃 로고 용도는 `전체` 또는 개별 레이아웃 제공 모듈의 `상단`/`하단` 사용처를 하나 이상 선택해야 한다. 사용처 체크 묶음은 조건부 필수로 표시하고 브라우저 검증과 server-side POST 검증을 함께 둔다. 시작/종료 시각을 비우면 상시 적용되고, 같은 용도에서 기간이 겹치면 개별 모듈 직접 지정, 전체 지정, 기존 상단 fallback 순으로 사용처 우선순위를 적용한 뒤 현재 시각에 해당하면서 전체 기간이 더 짧은 로고가 우선된다는 도움말을 둔다. 모듈이 제공하는 기존 `logo-positions.php` 후보도 같은 로고 용도 선택지에 표시하되, 실제 출력은 해당 화면이 `sr_logo_manager_render_logo()`를 명시적으로 호출할 때만 일어난다. 기본 공개 계열 레이아웃은 `사용자화면 로고(PC)` 또는 `사용자화면 로고(모바일)` 중 하나만 등록되어 있어도 양쪽 화면에 같은 로고를 사용하며, 둘 다 있을 때는 모바일 폭에서 모바일 로고를 우선한다. `public.app_icon` 용도에서만 `사용자 화면 심볼로 사용` 스위치를 활성화하고, 앱아이콘 외 용도 제출값은 서버에서 저장하지 않는다. 이 스위치는 공개 헤더 fallback 심볼 후보 여부만 정하며 파비콘 head link 출력 조건과 분리한다. 파비콘 등록 폼은 `앱아이콘으로도 사용`, 앱아이콘 등록 폼은 `파비콘으로도 사용` 스위치를 제공하며, 켜면 같은 파일을 별도 저장본으로 다른 용도에 함께 등록한다. 기본 공개 계열 레이아웃은 사용자화면 전용 로고가 없을 때 심볼 후보와 사이트명을 브랜드 fallback으로 함께 호출한다. 별도 레이아웃에서는 `sr_logo_manager_public_symbol_logo()`, `sr_logo_manager_public_symbol_url()`, 또는 `sr_logo_manager_render_public_symbol_logo()`를 명시적으로 호출해야 반영된다. 파비콘 용도 로고 행은 아이콘 세트 모달을 제공하며, 운영자는 생성 크기, contain/cover 맞춤 방식, 투명 또는 hex 배경색, 생성 후 즉시 사용 여부를 선택한다. 로고 배치 목록은 용도별 `현재 적용`과 현재 시각에 적용 가능한 `적용 후보`를 구분해 표시하고, 아이콘만 있는 관리 버튼은 현재 적용 로고와 목록 하단에 설명을 표시한다. `public.favicon` 로고를 중지하면 해당 로고와 그 아이콘 세트는 head link 후보에서 빠지며, 같은 용도의 다른 활성 후보가 있으면 정렬 기준대로 다음 후보가 적용될 수 있다. 다른 활성 후보가 없으면 `icon`과 `apple-touch-icon` link를 모두 출력하지 않고, `/favicon.ico` fallback도 no-store 404로 응답한다. 완전 삭제 POST는 파비콘 저장본과 생성 아이콘 세트만 제거하고 브라우저가 이전 투명 아이콘 캐시를 붙잡지 않도록 origin cache 정리 헤더를 함께 보낸다. 브라우저 탭 아이콘은 캐시나 루트 `/favicon.ico` fallback 영향을 받을 수 있으므로 운영 확인은 HTML head의 `icon`/`apple-touch-icon` 링크와 `/favicon.ico` 응답을 기준으로 안내한다.

새 팝업 추가/수정처럼 노출 위치와 닫기 정책을 함께 다루는 폼은 상태, 팝업 스킨, 서비스, 노출위치, 매칭 방식, 대상 선택, 시작/종료 시각, 닫기 유지일처럼 저장 결과 판단이 필요한 항목에 운영자용 도움말을 둔다. 제목이나 내용처럼 바로 알 수 있는 기본 입력에는 도움말을 붙이지 않고, 공용 팝업레이어와 서비스/노출위치 자동 노출의 차이, 전체/선택 매칭 차이, 방문자가 닫은 뒤 다시 보이는 시점을 쉽게 설명한다. 팝업레이어는 화면 중앙 overlay로 배열되므로 배너의 상세 삽입 위치와 달리 `본문 위`, `댓글 아래` 같은 slot 라벨을 관리자 선택지로 노출하지 않는다. 매칭 방식은 select 대신 전체/선택 라디오 토글로 표시하고, 선택을 고른 경우에만 대상 검색 컨트롤을 보여주며 서버에서도 대상 ID를 필수로 검증한다.

커뮤니티 환경설정과 게시판처럼 레벨, 쪽지 사용 여부와 보내기 권한, 게시판 접근 권한, 첨부, 배너/팝업, 회원 자산 정책을 함께 다루는 폼은 레벨 사용/자동 재계산, 쪽지 사용 여부, 쪽지 보내기 권한, 회원 그룹, 최소 레벨, 게시판 그룹, 상태, 스킨, 첨부 설정, 허용 파일 확장자, 배너/팝업레이어, 회원 자산, 리액션 사용 여부, 커뮤니티 홈 레이아웃, 정렬, 커뮤니티 피드 노출처럼 운영 판단이 필요한 항목에 도움말을 둔다. 제목, 이름, 설명처럼 바로 알 수 있는 기본 입력에는 도움말을 붙이지 않고, 쪽지를 보낼 수 있는 회원과 받는 회원의 차이, 쪽지 사용을 끌 때 회원의 쪽지함 접근과 새 쪽지 작성이 함께 닫히고 기존 내역은 데이터로 유지되는 점, 매니저는 보내기 조건을 항상 통과하고 스탭은 회원 관리 권한이 있을 때 보내기 조건을 통과하는 점, 환경설정 기본값과 `그룹`/`전체` 적용의 차이, 접근 정책과 레벨 조건, 회원 자산 지급/차감 시점, 적립 회수 실패 조건, 리액션을 끌 때 게시글/댓글 위젯과 목록 반응 수 표시가 함께 숨겨지는 점을 쉽게 설명한다. 게시판 설정의 `커뮤니티 피드 노출`은 꺼진 게시판의 글, 댓글, 시리즈를 커뮤니티 메인, 사이드바, 최근글/인기글 같은 게시판 목록 밖 피드 후보에서 제외할 뿐 게시판 목록과 게시글 직접 접근 정책을 바꾸지 않는다. 게시판 설정의 리액션 사용 여부는 커뮤니티 환경설정과 별도로 저장되며, 해당 게시판의 게시글/댓글 위젯과 목록 반응 수 표시를 결정한다. 회원 그룹별 적용 선택지는 현재 선택한 자산과 맞는 정책만 보여주고, 뱃지 요약도 같은 자산명과 금액을 표시한다. 게시판 관리권한과 게시판 카테고리처럼 한 게시판 안에서 여러 행을 관리하는 보조 설정은 목록을 먼저 보여주고, 추가/수정 입력은 모달로 분리한다. 게시판처럼 긴 수정 폼에서 주 도메인 자체를 삭제하는 작업은 별도 위험 작업 섹션과 탭을 두지 않고 하단 sticky 액션의 목록/복사 다음 위치에 `삭제` 아웃라인 위험 버튼을 배치해 확인 모달을 연다. 게시판 그룹과 콘텐츠 그룹은 기본 정보만 다루므로 같은 하단 sticky 액션에서 삭제 확인 모달을 연다.

커뮤니티 첨부파일 관리는 기존 게시글 첨부를 대상으로 하는 목록 화면으로 제공한다. 첨부 상태 변경은 선택 항목을 100건 이하로 제한한 POST 일괄 작업으로 처리하고, 결과는 토스트로 표시한다. 첨부 다운로드 내역은 무료/유료 구분, 게시판, 게시글 ID, 첨부 ID, 회원 공개 해시, 기간, 검색어 필터를 제공하며 파일 원문이나 저장소 경로 대신 게시글 제목과 첨부 원본명 snapshot을 표시한다.

커뮤니티 개인정보 수집 및 이용동의 설정은 커뮤니티 환경설정과 게시판 개별 설정에서 다룬다. 커뮤니티 환경설정은 새 게시판의 기본값이고, 게시판 개별 설정은 다른 게시판 설정처럼 `현재만 저장`, `그룹에 복사`, `전체에 복사` 범위를 따른다. 사용 여부와 동의 적용 대상별 정책 문서(게시글 동의, 댓글 동의, 업로드 동의)를 저장하고, 정책 문서는 직접 key 입력이 아니라 사용 가능한 문서명 셀렉트로 선택하게 한다. 각 셀렉트의 `선택 안 함`은 해당 대상에 동의를 적용하지 않는다는 뜻이며, 첨부 업로드에는 일반 이미지/파일 첨부와 CKEditor 본문 이미지 임시 업로드가 포함된다. 동의 사용 시 서버는 정책 문서가 선택된 대상 하나 이상과 각 대상의 published 정책 문서 snapshot을 검증하고, 공개 제출과 AJAX 업로드는 UI 표시 여부와 무관하게 서버에서 미동의를 거부해야 한다. 제출 성공 시 커뮤니티 모듈은 대상별 정책 문서 key/title/version/body hash snapshot과 IP/UA 해시를 `sr_community_submission_consents`에 남기고, 개인정보 export/cleanup 계약에 포함한다. 게시글/댓글 관리자 목록은 제출 단위 동의 증적 수와 최근 동의 시각을 표시해 운영자가 별도 export 없이도 동의 저장 여부를 빠르게 확인할 수 있게 한다.

비회원 작성 허용과 게시판별 추가 입력 항목을 추가할 때는 게시판 기본 설정 안에서 별도 섹션으로 분리한다. 비회원 작성은 허용 여부, 작성자 표시명 필수, 수정/삭제 검증 방식, 회원 전용 기능 제한을 함께 안내한다. 추가 입력 항목 관리는 게시판별 보조 목록과 추가/수정 모달로 두고, `field_key`는 `data-admin-key-input` 기준으로 검증하며 라벨, 유형, 필수 여부, 공개/관리자 표시, 개인정보 목적, export/cleanup 정책을 서버 검증과 맞춘다. 상세 설계와 후속 범위는 GitHub 이슈에서 추적한다.

회원 그룹별 설정처럼 여러 조건 행을 편집하는 설정은 기본 정보 폼 행 안에 끼우지 않고 별도 목록형 섹션으로 분리한다. 섹션은 다른 목록 화면처럼 `card-header`, `table-wrapper`, `table` 흐름을 따르고, 규칙 추가 버튼은 카드 헤더 오른쪽에 둔다. 회원 그룹, 계산 방식, 자산처럼 규칙 저장에 필요한 열은 헤더에 `(필수)`를 표시하고, 조정값 입력 여부와 계산 방식을 좌우하는 계산 방식 열에는 도움말 버튼으로 상세 설명을 제공한다. 증감 조정은 0 이상의 정수 조정값을 입력하고 지급 맥락에서는 더하며 차감 맥락에서는 빼는 방식으로 설명한다. 커뮤니티 회원 그룹별 설정처럼 레벨 조건을 지원하는 화면은 최소 레벨 열을 함께 표시하고 저장 POST에도 포함한다. 여러 자산을 지원하는 조정 규칙은 한 행에 자산 하나와 조정값 하나만 설정하게 하고, 여러 설정 항목을 한 폼 행 안의 중첩 테이블로 넣지 않는다.

콘텐츠와 커뮤니티의 실제 자산 설정 화면은 사용 여부, 과금 방식, 버튼 문구, 처리 방향처럼 기능 동작을 결정하는 행과 포인트/금액 항목, 자산별 금액, 회원 그룹별 적용처럼 잔액 처리에 필요한 자산 설정 행을 구분해 표시한다. 사용 여부는 자산 선택 여부가 아니라 `유료 열람 사용`, `댓글 차감 사용` 같은 별도 체크박스로 결정한다. 미리 만든 정책은 콘텐츠와 커뮤니티 모두 `회원 그룹별 적용`으로 표시하고 셀렉트에서 하나 이상 선택하게 한다. 도움말에는 선택한 적용이 회원의 그룹, 커뮤니티 레벨, 선택한 포인트/금액 항목에 맞는 실제 금액을 계산한다는 점을 안내한다. 그룹별 계산 방식, 자산, 조정값 편집은 전용 금액 조정 관리 화면에서만 수행하고, 콘텐츠나 게시판 설정 화면에서는 직접 조정값을 편집하지 않는다.

콘텐츠와 커뮤니티의 실제 자산 설정 화면에서 회원 그룹별 적용을 셀렉트로 선택하면 하단에 선택 뱃지를 추가한다. 뱃지에는 적용 이름과 그룹별 포인트/금액 실제 적용값 요약을 함께 보여주고, 제거 버튼으로 선택을 제거할 수 있다. 이미 추가한 적용은 셀렉트 선택지에서 제외하고 제거하면 다시 선택할 수 있게 한다. 자산 설정 행에서 단독 그룹, 금액 입력 묶음, 그룹/전체 복사 같은 범위 선택을 함께 표시할 때는 각 묶음을 가운데가 아니라 행 상단 기준으로 정렬한다.

도움말 모달 본문은 `sr_admin_help_modal_html()`의 `.admin-help-modal-body` 공통 스타일을 사용한다. 문단, 목록, 문단과 목록 사이 간격은 화면별 CSS가 아니라 공통 스타일에서 맞춘다.

저장, 삭제, 적용 같은 짧은 결과 안내는 `sr_admin_feedback_toasts($notice, $errors)`를 사용해 토스트로 출력한다. 화면 본문에 영구적으로 남아야 하는 설명과 작업 결과 피드백을 섞지 않는다.

## 대시보드

관리자 대시보드는 사이트 기본 정보, 복구 주의 항목, 활성 모듈의 `dashboard.php` 계약 섹션을 표시한다. 대시보드 콘텐츠 제공은 회원, 포인트, 적립금, 예치금, 쿠폰·이용권, 환전, 콘텐츠, 커뮤니티, 퀴즈, 설문처럼 회원/서비스 흐름을 직접 보여주는 모듈을 기준으로 한다. 설치 보호, 인증 런타임, 고위험 설정, 전체 모듈 목록, 배너, 팝업레이어, 알림처럼 운영 진단이나 출력 보조 성격이 강한 항목은 대시보드 기본 섹션으로 노출하지 않는다. 각 섹션은 `data-admin-dashboard-section` 값을 가진 배치 단위로 두고, 사용자가 드래그 앤 드롭으로 순서를 바꾸면 브라우저 `localStorage`에 표시 순서가 저장된다. 사이트 정보와 복구 주의 항목처럼 admin 모듈이 직접 소유한 기본 섹션은 카드 UI를 유지하지만, 모듈 제공 섹션의 카드 여부와 내부 시각 표현은 해당 모듈 view가 결정한다.

모듈별 대시보드 섹션의 내부 구성은 모듈이 소유한다. 모듈은 `dashboard.php`에서 `view`로 모듈 내부 `views/*.php` 파일을 지정할 수 있고, admin 모듈은 검증된 view를 대시보드 섹션 안에 include한다. admin 모듈은 외곽 wrapper, 이동 핸들, 정렬 저장을 맡고, 모듈 view는 제목 배치, 지표 구성, 링크, 도메인별 리듬을 직접 렌더링한다. 필요한 스타일은 각 모듈의 `admin.stylesheets`로 선언한 모듈 내부 CSS에서 소유한다. admin wrapper는 최소 drop target과 drop line 기준을 제공하고, 핸들 주변의 시각적 안전 여백은 모듈 view/CSS가 확보한다. 따라서 모듈 view는 카드형 UI를 선택할 수도 있고 카드가 아닌 자체 레이아웃을 렌더링할 수도 있다.

모듈은 `dashboard.php`에서 `default_visible`로 기본 노출 여부를 선언할 수 있다. 대시보드의 섹션 관리 UI는 페이지 타이틀 오른쪽의 `설정` 버튼으로 `대시보드 설정` 큰 모달을 열어 제공하고, 모달 상단에 드래그 배치와 표시 토글 안내를 표시한다. 모달의 각 섹션 항목은 단독 스위치 토글로 표시/숨김을 바꾸고, 항목 핸들을 드래그 앤 드롭하면 대시보드 본문 이동과 같은 위치 판정, drop line, span 보정 방식으로 순서를 저장한다. `변경취소`는 모달을 연 뒤 변경사항이 있을 때만 표시하고 모달을 열었던 시점의 상태로 되돌린다. 표시/숨김과 순서는 브라우저 `localStorage`에 저장하며, 사용자가 저장한 값이 있으면 모듈 기본값보다 우선한다. 섹션을 숨기면 배치 배열에서 제외해 대시보드에 표시하지 않고 모달 하단의 비활성 항목으로 이동시키며, 남은 grid 행은 즉시 재계산한다. 한 행에 남은 섹션 수에 따라 4열, 3열, 2열, 1열 전체 폭으로 보정한다. 숨긴 섹션을 다시 표시하면 대시보드 맨 마지막에 한 줄 전체를 차지하는 섹션으로 추가한다.

`view`가 없는 기존 모듈은 admin 모듈의 fallback layout으로 렌더링한다. fallback의 `table`은 기존 `항목 / 주요 수치 / 상세` 표이고, `stats`는 지표 카드형 요약이다. 알 수 없는 layout, state, emphasis 값은 admin 모듈이 안전한 기본값으로 되돌린다. fallback row의 라벨과 상세 접두/접미 문구는 `sr_t()`로 제공하고, SQL은 숫자나 코드 값만 반환하게 둔다.

대시보드 섹션은 화면 구조를 바꾸는 관리 도구가 아니라 개인 작업 배치에 가깝다. 따라서 드래그 순서는 DB에 저장하지 않고, 저장된 브라우저 배치가 없을 때의 기본 표시 순서는 관리자 사이드메뉴 출력 순서를 따른다. 초기 배치는 각 섹션을 한 줄 전체 폭으로 표시한다. 섹션 이동은 섹션 전체가 아니라 섹션의 아이콘 핸들에서만 시작하며, 관리자 대시보드 본문은 최대 4열 그리드로 배치한다. 다른 섹션의 좌우에 놓을 때는 수직 고스트 라인을 표시하고 일반 폭으로 배치하며, 1열 전체 폭 섹션의 좌우에 놓으면 기존 1열 섹션과 이동 섹션을 같은 행의 2열로 전환한다. 위아래에 놓을 때는 행 사이 슬롯 중앙에 수평 고스트 라인 하나만 표시하고 한 줄 전체를 차지하는 1열 섹션으로 배치할 수 있다. 드롭 결과는 드롭 순간의 마우스 위치를 다시 해석하지 않고 마지막으로 표시된 고스트 라인의 위치를 기준으로 결정한다. 고스트 라인은 드래그 중 카드 배치를 밀지 않는 overlay로만 표시한다. 기본 배치 기준이나 저장 구조가 바뀌는 경우에는 정렬 저장 key를 변경해 기존 브라우저 정렬 배열을 초기화한다.
