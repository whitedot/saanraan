# 관리자 UI 작성 기준

관리자 화면은 G5 Codex 계열의 관리자 UI 톤을 기준으로 맞춘다. public/admin 양쪽에서 쓰는 이름이 같은 기본 UI class라도 런타임 CSS 파일은 분리한다. 관리자 반복 UI 원형은 admin 모듈의 `modules/admin/assets/common.css`, 공개 반복 UI 원형은 `assets/common.css`가 각각 소유하고, 관리자 shell과 관리자 전용 배치는 `modules/admin/assets/admin.css`로 제한한다. `ui-kit.css` 파일명은 UI-KIT 조회 화면의 보조 스타일처럼 UI-KIT 자체에 남긴다. 모듈 도메인 고유 보정은 각 모듈의 `admin.stylesheets`로 분리한다.

- `assets/tokens.css`: 사이트 전반에서 재사용할 `--color-*`, `--spacing`, 타이포그래피, 반경, 그림자 토큰을 둔다.
- `assets/icons.css`: 프로젝트 기본 아이콘셋인 self-hosted Google Material Symbols Outlined 폰트와 `.sr-icon` 표시 규칙을 둔다.
- `modules/admin/assets/common.css`: 관리자 런타임의 reset/base와 `btn`, `card`, `table`, `badge`, `form-*`, `dropdown-*`, `modal-*`, `tab-*`, icon sizing 같은 반복 UI 원형을 둔다.
- `assets/common.css`: 공개 런타임의 reset/base와 반복 UI 원형을 둔다. 관리자 런타임과 파일을 공유하지 않는다.
- `assets/admin-ui.css`: `.admin-ui-scope` 안의 반복 가능한 관리자 작업 조합만 둔다.
- `modules/admin/assets/admin.css`: 관리자 shell, 사이드바, 상단바, 관리자 콘텐츠 폭, 목록/폼 배치 같은 admin 모듈의 실제 화면 구조를 둔다. 공용 UI 원형이나 모듈 도메인 class를 이 파일에 다시 넣지 않는다.
- `modules/{module_key}/assets/*.css`: 모듈 관리자 본문에서만 쓰는 도메인 고유 스타일을 둔다. 모듈은 `module.php`의 `admin.stylesheets`에 자기 `assets/` 아래 CSS만 선언하고, admin shell은 활성 모듈의 선언을 검증해 공통 관리자 CSS 뒤에 출력한다.

공개 화면 런타임은 `assets/tokens.css`, `assets/icons.css`, `assets/common.css`, `assets/saanraan.css`, `assets/public-ui.css`, 활성 모듈의 `public.stylesheets`, layout context의 `stylesheets` 순서로 호출한다. `assets/icons.css`가 공용 아이콘 폰트를 맡고, `assets/common.css`가 공개 런타임 원형을 맡고, `assets/saanraan.css`가 공개 화면의 `--sr-*` 토큰과 기본 문서 스타일을 맡으며, `assets/public-ui.css`는 공개/회원 화면의 반복 UI 조합을 맡는다. 일반 공개 런타임과 `/ui-kit` 공개 UI-KIT 모두 관리자 런타임 CSS인 `modules/admin/assets/common.css`, `assets/admin-ui.css`, `modules/admin/assets/admin.css`를 직접 호출하지 않는다.

관리자/공개 런타임 CSS 호출은 PHP helper가 실제 파일의 `filemtime()` 값을 `?v=` query string으로 붙여 캐시를 갱신한다.

관리자 런타임 CSS 호출 순서는 `Pretendard`, `assets/tokens.css`, `assets/icons.css`, `modules/admin/assets/common.css`, `assets/admin-ui.css`, `modules/admin/assets/admin.css`, 활성 모듈의 `admin.stylesheets` 선언 순서다. 활성 모듈 stylesheet는 `/modules/{module_key}/assets/*.css` 안쪽 파일만 허용하고 외부 URL, `..` 경로, 모듈 폴더 밖 파일은 무시한다. 공개 모듈 stylesheet도 같은 경로 규칙을 적용하며 `module.php`의 `public.stylesheets`에 선언한다.

프로젝트 기본 아이콘셋은 Google Material Symbols Outlined다. 아이콘 폰트는 `assets/fonts/material-symbols-outlined.ttf`로 self-hosting하고 public/admin 런타임에서 preload한다. `sr_material_icon_html()`로 출력한 `.sr-icon[data-sr-material-icon]`은 `sr_material_icon_bootstrap_script()`가 폰트 준비 완료 class를 붙이기 전까지 투명하게 유지하므로, `home`, `settings` 같은 Material ligature 글자가 로딩 중 잠깐 보이지 않는다.

Material Symbols는 메뉴, 툴바, 본문 액션처럼 페이지에서 독립 아이콘을 표시할 때 사용한다. 체크박스 체크 표시, 드롭다운 caret, 스위치 thumb처럼 컴포넌트 내부 상태나 조작 힌트는 해당 컴포넌트 CSS가 소유하며 Material 아이콘으로 대체하지 않는다. 드롭다운 caret처럼 여러 위치에서 재사용하는 방향 화살표는 `sr_ui_arrow_icon_html()` helper로 출력한다.

드롭다운, 오버레이/모달, 탭처럼 관리자와 공개 화면에서 함께 쓸 수 있는 기본 상호작용은 `assets/common-ui.js`에 둔다. 관리자 UI-KIT과 public 런타임 미리보기도 이 파일을 호출해 같은 동작 원형을 확인한다.

관리자 디자인 책임은 admin 모듈에 둔다. 각 모듈의 관리자 view는 본문 마크업과 도메인 출력만 맡고, 관리자 shell, 사이드바, 상단바, 공통 관리자 asset, 관리자 콘텐츠 컨테이너는 admin skin이 맡는다. 현재 관리자 skin은 `admin_skin_key`로 선택하며, 등록된 key가 없거나 파일이 없으면 `basic`으로 fallback한다.

관리자 shell은 화면을 편하게 그리기 위한 목적으로 view 출력 뒤 `DOMDocument`로 HTML을 다시 해석하거나 class를 주입하지 않는다. 폼 행, 카드 헤더, 테이블 wrapper, 체크박스 보조 텍스트처럼 의미가 있는 구조는 view가 최종 마크업으로 직접 출력해야 한다.

렌더 후 DOM 처리가 필요한 경우는 별도 예외로 다룬다. 예를 들어 보안 정화, 외부 HTML의 제한적 변환, 편집기 콘텐츠 처리처럼 입력 자체가 HTML이고 변환 목적이 명확한 경우에는 호출 위치와 책임 모듈을 드러내고, 허용 범위와 테스트를 함께 둔다. 단순히 관리자 화면의 반복 마크업을 줄이거나 class를 자동 보정하기 위한 후처리는 사용하지 않는다.

CSS class는 범위를 드러내는 접두어를 사용한다.

- 관리자 반복 UI는 `btn`, `card`, `table`, `badge`, `form-*`처럼 `modules/admin/assets/common.css`에 직접 둔다. 공개 런타임의 같은 이름 class는 `assets/common.css`에서 따로 소유한다.
- 관리자 shell과 관리자 전용 배치는 `admin-*` 접두어를 사용하고 `modules/admin/assets/admin.css`에 둔다.
- 모듈별 관리자 본문에서 도메인 고유 스타일이 필요하면 `{module_key}-admin-*` 또는 `sr-{module_key}-admin-*` 형식을 사용하고, 해당 CSS는 모듈의 `assets/` 아래 stylesheet에서 소유한다.
- 관리자 view는 전역 `body`, `a`, `.container`, `.btn` 같은 넓은 선택자를 직접 재정의하지 않는다.
- 탭처럼 공통 CSS에 이미 정의된 반복 UI는 `tab-nav-*`, `tab-trigger-*` 같은 기존 시맨틱 클래스를 먼저 사용한다. 탭 전환 화면은 초기 HTML에서 `role="tablist"`, `role="tab"`, `role="tabpanel"`, `aria-controls`, `aria-labelledby`, `aria-selected`를 직접 출력하고, `data-admin-tabs` 스크립트는 선택 상태와 키보드 이동만 보강한다. 알림은 관리자와 공개 UI-KIT 모두 `alert alert-{status}` 또는 `alert alert-solid alert-{status}`를 기준 class로 사용한다. 토스트는 기존 관리자 메시지 클래스인 `admin-flash-message-*`에 `alert` 상태 class와 `data-admin-toast` 동작 속성을 더해 사용하고, 위치와 닫기 버튼 배치는 `data-admin-toast-*` 속성 선택자로 처리한다.
- UI-KIT 조회 화면의 배치와 예시 상태 표시처럼 실제 컴포넌트 원형이 아닌 표현은 `ui-kit-*` 접두어로만 둔다. `ui-bg-*`, `ui-text-*`, `ui-grid`, `ui-flex`, `ui-gap-*` 같은 Tailwind식 범용 utility 표현은 관리자/공개 UI-KIT 샘플에 사용하지 않는다.
- 공통 UI를 변경하거나 새 관리자 화면에서 UI 조합을 확인할 때는 `/admin/ui-kit` 관리자 조회 화면과 `/ui-kit` public 런타임 UI-KIT에서 런타임별 결과를 확인한다.

버튼, 배지, 탭, 드롭다운, 모달처럼 UI-KIT 샘플과 실제 화면이 함께 쓰는 컴포넌트는 기본, hover, focus-visible, disabled 상태를 관리자에서는 `modules/admin/assets/common.css`, 공개에서는 `assets/common.css`가 각각 소유한다. `btn-solid-*`, `btn-outline-*`, `btn-soft-*`, `btn-ghost-*`, `btn-gradient-*`, `btn-icon`, `badge`, `badge-list`, `tab-trigger-*`, `dropdown-*`, `modal-*` 계열을 새로 쓰면 `/admin/ui-kit`과 `/ui-kit`에서 상태별 표현을 함께 확인한다.

UI-KIT 샘플은 외부 아이콘 런타임에 의존하지 않는다. 실제 아이콘 자리에는 self-hosted Material Symbols helper인 `sr_material_icon_html()`을 사용하고, 샘플 설명이나 임시 보조 표식처럼 아이콘 자체가 목적이 아닌 위치에만 텍스트 보조 라벨을 둔다.

## 화면 내 이동 링크

사이드 메뉴에 이미 등록된 관리자 화면 이동 링크는 본문 상단에 다시 만들지 않는다. 예를 들어 회원 그룹의 그룹 목록과 회원 그룹 규칙처럼 사이드 메뉴가 1차 탐색을 제공하는 화면은 각 view 안에서 `admin-local-nav` 버튼 묶음을 반복하지 않는다.

`admin-local-nav`는 사이드 메뉴에 없는 같은 화면 안의 하위 흐름이나 필터성 이동에만 사용한다. 상태 요약 링크처럼 현재 목록의 조건을 빠르게 바꾸는 경우가 여기에 해당한다. 잔액/조정/거래 내역, 알림 목록/이메일 발송 작업처럼 독립 운영 화면으로 자주 오가는 흐름은 사이드 메뉴 항목으로 둔다. 새 항목 추가, 수정, 목록 복귀처럼 현재 작업의 직접 액션은 카드 헤더나 폼 액션 영역에 둔다.

포인트처럼 운영자가 표시명을 바꿀 수 있는 회원 자산 모듈은 사이드 메뉴의 모듈명과 잔액/거래 내역 항목에도 같은 표시명을 사용한다.

사이드 메뉴와 같은 전체 화면 이동 버튼을 본문 상단에 반복하지는 않지만, 현재 데이터 컨텍스트를 유지하는 액션 링크는 본문에 둘 수 있다. 예를 들어 회원 자산 화면에서 특정 회원을 조회한 뒤 같은 `account_identifier`로 잔액, 조정, 거래 내역을 오가는 버튼이나 잔액 목록 행의 `조정`, `거래 내역` 버튼은 허용한다. 회원 그룹 수동 배정과 배정 이력처럼 특정 그룹을 대상으로 하는 작업은 별도 사이드 메뉴 화면으로 두지 않고 그룹 목록 행의 관리 모달에서 처리한다.

## 텍스트 색상과 크기

관리자 화면의 기본 글자색은 `html`, `body`, `.admin-content`에서 상속되는 `--color-body-color`를 기준으로 한다. 기본 글자 크기는 `--text-body`인 `16px`를 기준으로 한다. 일반 본문, 테이블 셀, 테이블 헤더, 폼 라벨, 필터 라벨, 카드 제목에는 별도 회색 색상이나 작은 글자 크기를 반복해서 지정하지 않는다.

타이포그래피는 primitive 크기 토큰(`--text-xs`부터 `--text-2xl`) 위에 역할 기반 alias를 얇게 둔다. 런타임에서 반복되는 제목과 본문 위계는 가능하면 `.type-display`, `.type-display-fluid`, `.type-page-title`, `.type-page-title-fluid`, `.type-section-title`, `.type-card-title`, `.type-body`, `.type-small`, `.type-meta`, `.type-caption`, `.type-code` 또는 같은 역할 토큰을 사용한다. 관리자와 공개 런타임은 각각 자기 `common.css`에 같은 역할 class를 가지고, 실제 값은 `assets/tokens.css`의 `--type-*-size`, `--type-*-line-height`, `--text-strong`, `--text-muted`, `--text-subtle`에서 가져온다.

관리자 실제 화면의 구조 ID나 화면 고유 클래스는 배치, 간격, 경계선, 상태 예외를 맡고, 텍스트 크기와 줄 높이는 가능한 한 `type-*` 역할 클래스로 표시한다. 예를 들어 현재 페이지 제목은 `#container_title.type-page-title`, 페이지 보조문구는 `#container_subtitle.type-small`로 출력한다. `#container_title`과 `#container_subtitle` 스타일은 제목 아래 선, 여백, 색상 예외처럼 shell에 묶인 표현만 담당한다. `admin-form-help`, `admin-summary-meta`, `admin-list-summary`, `admin-filter-label`, `admin-dashboard-meta` 같은 재사용 `admin-*` 클래스는 제거하지 않고 UI kit 토큰을 소비하는 컴포넌트 규칙으로 유지한다. 이때 색상은 `--text-muted/subtle`, `--color-*`, 배경은 `--color-default-*`나 `color-mix()` 기반 토큰 조합, 포커스 링은 `--color-primary-ring`처럼 light/dark 값을 가진 토큰, 타이포는 `--type-meta-*` 또는 `--type-caption-*` 같은 역할 토큰을 기준으로 한다. 설정 화면처럼 페이지 제목을 더 작게 쓰는 관리자 전용 예외는 `--admin-page-title-compact-*`처럼 admin 레이어의 명시 토큰으로 둔다.

UI-KIT CSS는 토큰 표와 샘플의 배치만 담당한다. 실제 관리자/공개 화면에서 쓰는 제목, 본문, 도움말, 메타, 캡션 값은 `admin.css`, `saanraan.css`, `public-ui.css` 같은 런타임 스타일시트가 역할 토큰을 소비해 정의한다.

역할별 기준은 다음과 같다. `Display`는 공개 hero나 UI-KIT 미리보기처럼 매우 큰 강조가 필요한 위치에만 사용한다. 유동형 hero 제목처럼 화면 폭에 따라 커지는 값은 직접 최대값을 쓰지 말고 `--type-display-max-size` 같은 역할 토큰을 함께 사용한다. `Page title`은 화면이나 콘텐츠의 주 제목, `Section title`은 카드 묶음이나 설정 구역 제목, `Card title`은 카드 내부 제목에 사용한다. 공개 화면의 유동형 페이지 제목 최대값은 `--type-page-title-fluid-max-size`를 사용한다. `Body`는 기본 본문, `Small`은 폼 도움말과 보조 설명, `Meta`는 날짜·상태·식별자 같은 낮은 우선순위 정보, `Caption`은 표·이미지·예시의 캡션, `Code`는 key, 토큰명, 짧은 식별자에 사용한다.

`--font-weight-medium`은 `500`, `--font-weight-semibold`는 `600`, `--font-weight-bold`는 `700`으로 구분한다. 과거처럼 medium과 semibold가 같은 값이면 카드 제목, 라벨, 강조 텍스트의 위계가 UI-KIT에서 구분되지 않으므로 새 기준에서는 분리한다.

`/admin/ui-kit`와 `/ui-kit`의 Typography 섹션은 한글, 영문, 숫자, 긴 식별자, 테이블 내부 텍스트를 함께 보여준다. 공통 타이포그래피를 바꿀 때는 두 화면에서 관리자 조밀 화면 기준과 공개 콘텐츠 기준을 같이 확인한다.

옅은 색상은 의미가 분명한 보조 텍스트에만 사용한다. 예를 들어 `hint-text`, `muted-text`, `admin-form-help`, `admin-card-subtitle`, 빈 상태 안내, 보조 메타 정보처럼 본문보다 우선순위가 낮은 텍스트가 여기에 해당한다. 성공, 경고, 위험, 링크, 활성 탭처럼 상태나 상호작용을 나타내는 색상은 예외로 유지한다.

`var(--text-xs)`, `var(--text-sm)`, `.8125rem`처럼 기본보다 작은 크기는 화면 밀도나 정보 계층이 분명한 경우에만 사용한다. 도움말, 힌트, 보조 메타, 상태 배지, 사이드바 보조 메뉴, 테이블 안의 부가 정보가 여기에 해당한다. 새 관리자 화면의 본문과 입력 라벨은 먼저 16px 상속을 기준으로 확인한다.

관리자 런타임의 기본 입력 필드는 class 없는 `input`, `select`, `textarea`가 아니라 `.form-input`, `.form-select`, `.form-textarea`, `.form-checkbox`, `.form-radio` 시맨틱 클래스를 사용한다. 이 클래스들은 다른 요소의 자식 선택자가 아니라 `admin.css`의 직접 선택자로 스타일을 소유한다. class 없는 필드 선택자는 기존 화면 호환을 위한 fallback으로만 남긴다.

## 사이드바

사이드바는 모듈의 `admin.category_label`을 라벨로만 표시하고, 실제 조작 가능한 메뉴는 `모듈 그룹 > 메뉴 항목`의 2단계 구조를 기본으로 유지한다. 단, 하위 메뉴 항목이 하나뿐인 모듈 그룹은 그룹 자체를 해당 항목 링크로 렌더링해 바로 화면을 연다. 라벨은 시스템, 회원, 사이트, 커뮤니티, 운영처럼 메뉴를 구분하는 시각적 구획일 뿐이며 접기/펼치기 버튼이나 링크로 사용하지 않는다. 루트 경로에서 열 초기화면 후보 선택은 사이트 설정의 `화면` 섹션에서 다룬다.

시스템 분류의 관리자 기본 메뉴 순서는 대시보드, 환경설정, 모듈·플러그인 관리, 업데이트, 데이터 정리, 관리자 메뉴, 관리자 권한, 관리자 작업 로그 순서로 둔다.

관리자 권한 화면은 필터 검색 목록이 아니라 이미 소유자 또는 메뉴 권한이 부여된 회원 목록을 기본으로 보여준다. 목록은 공개 해시와 이메일을 한 셀에 줄바꿈으로 표시하고, 해당 셀의 정렬은 이메일 기준으로 처리한다. 목록의 현재 권한 칸은 UI kit의 `badge-list` 패턴으로 소유자 또는 대표 메뉴를 먼저 보여주고, 조회/수정/삭제 개수는 보조 요약으로 표시한다. `/admin` 진입 권한은 소유자이거나 메뉴별 관리자 권한을 하나 이상 가진 상태를 뜻하고, 실제 하위 메뉴 접근은 각 메뉴의 개별 조회/수정/삭제 권한으로 다시 제한한다. 목록 조회와 상세 조회는 `view`, 새로 만들기/수정 폼 진입과 저장·상태 변경은 `edit`, 명시적 삭제는 `delete` 권한으로 보호한다. 소유자가 아닌 제한 관리자가 `/admin`에 접근하면 대시보드를 보여주지 않고 첫 번째 허용 메뉴로 이동시키며, 관리자 대시보드와 UI-KIT은 소유자 전용으로 둔다. 새 권한 부여는 `권한 추가` 모달에서 시작하고, 회원 검색은 별도 스택 모달로 열어 선택 후 권한 추가 모달로 돌아온다. 전체 및 목록 관리 셀의 `권한 추가` 버튼은 보조 아웃라인 버튼으로 표시하고, 목록 관리 셀의 변경/추가/회수 액션은 아이콘 버튼으로 제공한다. 목록 관리 셀의 `권한 추가` 버튼은 해당 회원을 미리 지정한 상태로 권한 추가 모달을 연다. 권한 추가와 권한 변경 모달은 별도 패널 카드 구조가 아니라 일반 관리자 모달처럼 `admin-form-row`와 `admin-form-field` 흐름으로 정돈한다. 신규 권한은 1차 분류, 2차 메뉴, 조회/수정/삭제 권한을 선택한 현재 상태를 저장 시점에 적용하며, 이미 메뉴 권한을 가진 회원에게는 기존 메뉴 권한을 유지한 채 새 선택 권한을 더한다. 수정이나 삭제 권한은 조회 권한을 전제로 하므로 프론트에서는 수정/삭제 선택 시 조회를 자동 선택하고, 서버 저장 시에도 조회 권한을 함께 정규화한다. 권한 변경 모달은 대상 회원 요약, 소유자 권한, 선택된 메뉴 권한 영역을 분리하고, 기존에 존재하는 메뉴별 행에서 조회/수정/삭제 체크박스와 제거만 조정한다. 1차 분류나 2차 메뉴에서 `전체`를 선택하면 해당 범위의 모든 메뉴에 같은 작업 권한을 적용한다. 새 관리자 권한은 정상 상태 회원에게만 부여할 수 있고, 비활성 회원의 기존 권한 회수와 축소는 허용한다. 소유자는 모든 관리자 권한을 가진 상태이므로 메뉴별 권한을 함께 저장하지 않으며, 기존 소유자에게 메뉴별 권한을 추가하려는 요청은 서버에서 거부한다. 소유자가 2명 이상이면 목록에서 소유자 행에도 `권한 회수` 버튼을 표시할 수 있지만, 저장 시점에는 소유자 행을 잠근 뒤 다시 확인해 마지막 소유자나 마지막 활성 소유자가 사라지는 동시 실행을 서버에서 막는다. 권한 추가와 권한 변경 모달의 항목 도움말은 부모 모달을 유지한 채 위에 스택 모달로 연다. 권한 추가 요청은 소유자 권한 또는 하나 이상의 메뉴 권한이 있어야 하며, 프론트 검증과 서버 POST 검증이 모두 같은 조건을 막아야 한다. 목록의 소유자가 아닌 회원은 행 단위 `권한 회수` 버튼으로 메뉴 권한을 즉시 비울 수 있다.

관리자 사이드바 브랜드는 로고 매니저의 `관리자 사이드바 로고`가 있으면 확장 상태에서 이를 우선 표시한다. 사이드바가 접히면 `파비콘/앱 아이콘 후보`를 사용하고, 해당 자산도 없으면 검정 사각형 안에 사이트명 첫 글자를 표시한다. 관리자 사이드바 로고가 없을 때는 확장 상태에서도 파비콘/앱 아이콘 후보와 사이트명 텍스트를 함께 표시한다. 공개 기본 레이아웃의 브랜드 로고는 `공개 헤더 로고(PC)`를 기본으로 표시하고, `공개 헤더 로고(모바일)`이 있으면 모바일 폭에서 이를 대신 표시한다.

접힌 사이드바에서 하위 메뉴가 없는 메인 메뉴는 메뉴명 커스텀 툴팁을 제공한다. 하위 메뉴가 있는 메인 메뉴는 툴팁 대신 팝업 서브메뉴 목록 맨 위에 메인 메뉴명을 굵은 텍스트로 표시해 현재 목록의 맥락을 드러낸다.

접힌 사이드바의 현재 활성 메인 메뉴는 아이콘만 보이더라도 위치를 인지할 수 있도록 회색 계열 배경과 라운드 처리를 적용한다.

저장된 접힘 상태를 페이지 진입 시 복원할 때는 초기 복원 중 transition을 비활성화해 콘텐츠 영역이 오른쪽에서 왼쪽으로 자라나는 것처럼 보이지 않게 한다. 사용자가 접기/펼치기 버튼을 직접 누를 때의 전환 효과는 유지한다.

`/admin/menu`에서 저장한 순서와 숨김 설정은 기본 선언보다 우선한다. 이 화면은 회원별 권한으로 위임하지 않고 소유자만 접근할 수 있게 둔다. 다만 번들 메뉴 구조를 정리하면서 생긴 과거 기본 순서값은 사용자 커스텀 순서로 보지 않고 새 기본 순서를 따른다. 시스템 분류, 관리자 모듈 그룹, 관리자 메뉴 항목은 운영자가 관리자 메뉴 설정 화면에 다시 접근할 수 있어야 하므로 숨김 처리할 수 없다.

관리자 메뉴 표시 설정 화면은 페이지 제목 아래에 드래그, 위아래 이동, 접기/펼치기, 저장 방식을 짧게 안내한다. 표 상단 카드 헤더에는 제목 문구 대신 좌측에 보조 아웃라인 토글 버튼 하나를 두고, 접힌 항목이 있으면 `전체 펼치기`, 없으면 `전체 접기`로 동작하게 한다. 분류, 모듈 그룹, 항목은 같은 저장 표에서 다루되 대상 칸에서 들여쓰기와 보조 경로로 위계를 드러낸다. 순서 드래그는 같은 범위와 같은 부모 안에서만 가능하게 유지하고, 이동 가능한 위치에만 고스트 라인을 표시한다. 이동 버튼도 드래그 가능한 같은 범위 안에서만 위아래로 움직이며, 분류나 모듈 그룹을 움직이면 하위 행도 같은 블록으로 함께 이동한다. 하위 항목 접기/펼치기는 긴 목록 편집을 돕는 브라우저별 화면 상태이며, 각 행의 접기/펼치기 버튼도 보조 아웃라인 버튼으로 표시한다. 전체 펼치기/접기 버튼은 이 화면 상태를 한 번에 바꾼다. 저장 데이터나 하위 행의 포함 관계는 바꾸지 않는다. 저장 후 다시 펼칠 때도 부모와 자식 행이 분리되지 않아야 한다. 저장/초기화 POST 결과는 새로고침 때 같은 토스트나 저장 작업이 반복되지 않도록 flash 결과와 redirect로 한 번만 표시한다. 저장/초기화 액션 영역은 긴 메뉴 목록에서도 접근 가능하도록 화면 하단에 sticky로 유지한다. 기본값 초기화는 메뉴 표시 순서와 숨김 설정이 모두 초기화된다는 안내에 동의한 경우에만 처리한다.

관리자 UI-KIT은 일반 메뉴 트리에 넣지 않고 사이드바 메뉴 목록 아래쪽의 보조 링크로 노출한다. UI-KIT은 실제 업무 메뉴가 아니라 관리자 UI 점검 화면이므로 메뉴 순서/숨김 오버라이드 대상과 분리한다.

관리자 메뉴 아이콘의 표현 선택권은 각 모듈에 둔다. 모듈은 `module.php`의 `admin.icon`에 관리자 shell이 제공하는 심볼 이름을 선언하거나, 자기 모듈의 `assets/` 아래에 둔 `png`, `webp` 파일을 선언할 수 있다. 관리자 shell은 선언을 그대로 믿지 않고 허용된 심볼과 모듈 내부 자산 경로만 렌더링하며, 선언이 없거나 유효하지 않으면 카테고리 기본 아이콘으로 fallback한다.

관리자 메뉴 심볼 이름 목록과 Material Symbols 매핑은 `modules/admin/helpers/icons.php`의 공통 계약이 소유한다. admin skin은 심볼 이름을 직접 해석하지 않고 helper 매핑을 통해 Material 아이콘 이름으로 변환한다. 기본 표현은 Material Symbols다.

모듈 메뉴에 허용되는 심볼 이름은 `settings`, `admin-mode`, `users`, `user`, `content`, `stats`, `home`, `folder`, `image`, `layers`, `search`, `menu-list`, `bell`, `shield`, `coins`, `wallet`, `gift`, `message-circle`, `service`이다. 툴바 전용 심볼은 같은 helper 계약에 있더라도 `module_menu`가 꺼져 있으면 모듈 선언에서 사용할 수 없다.

아이콘은 메뉴 텍스트를 보조하는 장식 요소로 취급한다. 접근 가능한 이름은 메뉴 label이 맡고, 이미지 아이콘도 `alt=""`와 `aria-hidden="true"`로 출력한다.

모듈 전역 설정은 가능하면 목록 화면 안에 배치하지 않고 독립 설정 화면으로 둔다. 이때 독립 도메인 관리 화면을 가진 모듈은 `admin-menu.php`에 `목록`, `설정`처럼 같은 모듈 그룹 안의 별도 항목을 선언하고, 설정 저장 POST는 설정 화면의 action이 소유하게 한다. CKEditor처럼 다른 화면의 입력 경험을 보강하는 플러그인은 도메인 정책을 자기 설정에 두지 않고, 필요한 경우 `플러그인` 같은 별도 분류에서 런타임 자산 설정 화면만 제공한다.

## 사이트/모듈 설정 화면

사이트 설정 화면은 사이트 이름, 시간대, locale, 운영 상태, 공개 화면 레이아웃, 초기화면, 관리자 UI 색상 모드, 관리자 스킨, 관리자 화면 에디터처럼 운영자가 의미를 이해하고 직접 변경할 수 있는 전용 필드만 다룬다. 시간대와 locale은 자유 입력이 아니라 사용 가능한 값 중에서 선택하는 select로 제공하고, 지원 언어 목록은 최소 한 개 이상 선택하도록 프론트엔드와 서버에서 함께 검증한다. 공개 레이아웃과 초기화면은 `화면` 섹션에 두고, 관리자 UI 색상 모드, 관리자 스킨, 관리자 화면 에디터는 `관리자 화면` 섹션에 둔다. 화면 상단의 다크 모드 전환 버튼은 admin 모듈 설정의 `admin_color_scheme` 값을 라이트/다크로 즉시 저장하는 빠른 전환이며, 공개 화면 색상에는 영향을 주지 않는다. `sr_site_settings`는 내부 저장소로 유지하지만, 임의 key/value를 추가·삭제하는 범용 관리자 화면은 제공하지 않는다.

코어나 모듈이 사이트 범위 설정을 추가로 필요로 하면 먼저 명확한 전용 관리자 화면, 설정 schema, 업데이트 스크립트, 또는 소유 모듈의 설정 화면을 만든다. 기능 플래그나 외부 연동 옵션처럼 운영자가 판단해야 하는 값도 의미, 허용 값, 검증 규칙이 드러나는 전용 UI로 승격한다. 범용 key/value UI로 전용 화면 책임을 우회하지 않는다.

전용 UI로 승격할 기준:

- 설정 의미를 설명해야 한다.
- 값 형식이 복잡하다.
- 실수하면 운영 장애가 난다.
- 여러 필드가 함께 저장되어야 한다.
- 모듈 도메인 정책과 연결된다.

전용 UI가 있는 모듈 전역 설정은 목록/검색/행 관리 화면에 섞지 않고 별도 설정 화면과 저장 action으로 분리한다. 예를 들어 배너 기본 스킨은 `/admin/banners/settings`, 팝업레이어 기본 스킨은 `/admin/popup-layers/settings`가 소유한다.

## 페이지 타이틀

관리자 페이지의 주 타이틀은 admin skin이 출력하는 `#container_title` 하나를 기준으로 스타일링한다. 각 모듈 view는 `$adminPageTitle` 값만 지정하고, 페이지 최상단 `h1`을 별도로 반복하지 않는다.

## 폼 화면

등록, 수정, 설정처럼 화면의 주 작업이 폼인 페이지는 실제 view 마크업에서 다음 구조를 사용한다.

```php
<form method="post" action="..." class="admin-form ui-form-theme">
    <?php echo sr_csrf_field(); ?>

    <section class="admin-card card">
        <h2>기본 정보</h2>
        <div class="admin-form-row">
            <label class="form-label" for="example_title">이름</label>
            <div class="admin-form-field">
                <input id="example_title" type="text" name="title" required>
            </div>
        </div>
    </section>

    <div class="admin-form-sticky-actions admin-form-actions admin-form-actions-split">
        <a href="..." class="btn btn-surface-default-soft">목록</a>
        <button type="submit" class="btn btn-solid-primary">저장</button>
    </div>
</form>
```

- 폼의 주 섹션은 `form > section.admin-card.card`로 둔다.
- 섹션 제목은 `h2`를 사용한다. 카드 헤더가 별도로 필요하면 view에서 `card-header`, `card-title` 마크업을 직접 출력한다.
- 일반 입력, 셀렉트, 텍스트 영역처럼 한 행에 주 컨트롤이 하나인 경우 `admin-form-row` 바로 아래에 `label.form-label`을 두고, `admin-form-field` 바로 아래에 컨트롤을 둔다. 이때 `label`의 `for`와 컨트롤의 `id`를 연결하고, 숨김 라벨을 감싼 중첩 `label`은 만들지 않는다.
- 입력 항목 라벨에는 필드 이름만 둔다. URL 동작, 허용 형식, 업로드 제한, 자동 처리 같은 설명 문구를 괄호로 라벨에 붙이지 말고 입력 컨트롤 아래 `p.admin-form-help` 또는 `small.admin-form-help`로 분리한다. `초`, `bytes`처럼 단순 단위만 안내하는 경우에는 도움말 문구를 만들지 않고 `input-group`과 `input-group-text`로 입력 컨트롤 뒤에 단위를 붙인다.
- 운영자가 직접 입력하는 `*_key`, `module_key`, `menu_key` 같은 key 입력은 첫 글자 영문 소문자와 이후 소문자, 숫자, `_`만 입력되게 `data-admin-key-input`, `pattern="[a-z][a-z0-9_]{1,59}"`, `inputmode="latin"`, `autocapitalize="none"`, `spellcheck="false"`를 함께 둔다. 저장 action은 같은 값을 `strtolower(trim(...))`로 정규화한 뒤 각 도메인의 key 검증 함수를 적용한다. slug처럼 하이픈을 허용하는 공개 URL 식별자는 `data-admin-slug-input`과 slug 전용 `pattern`으로 별도 처리한다.
- 회원 로그인 아이디처럼 key는 아니지만 영문 소문자, 숫자, `_`만 허용하는 관리자 입력은 `data-admin-login-id-input`처럼 도메인이 드러나는 속성을 쓰고, 같은 브라우저 속성과 서버 검증을 맞춘다.
- 회원 로그인 아이디 원문은 저장하지 않으므로 회원 수정 화면에서는 기존 로그인 아이디를 표시하거나 변경/해제하는 입력을 두지 않는다. 관리자 회원 생성 화면처럼 새 계정에 처음 부여하는 경우에만 로그인 아이디 입력을 제공한다.
- 필수 입력 항목은 HTML `required` 속성이 아니라 저장 action의 서버 검증을 기준으로 판단한다. 값이 비어 있거나 허용 목록 밖이면 저장이 실패하는 텍스트 입력, 셀렉트, 숫자 입력, 체크박스 묶음은 view 마크업의 해당 label 또는 복합 행의 `span.form-label`에 `<span class="sr-required-label">(필수)</span>`를 직접 넣어 표시한다. 특정 선택값에서만 필수인 항목도 라벨 문구는 짧은 `(필수)`만 사용하고, 조건 변화에 따라 표시와 브라우저 검증을 토글한다. 예를 들어 참조 유형과 참조 ID처럼 둘이 함께 의미를 갖는 짝, 특정 매칭 방식에서만 필요한 subject ID, 정책이 `group`일 때 필요한 회원 그룹 선택, 종결 상태에서 필요한 관리자 메모는 같은 규칙으로 처리한다. 프론트 검증은 서버 검증을 보조하는 층으로 함께 맞추되, 저장 action은 JS 비활성화나 임의 POST에서도 같은 필수 조건을 직접 검증해야 한다. 관리자 shell이나 공통 JavaScript로 필수 표기를 후처리하지 않는다.
- 화면의 셀렉트나 체크박스가 서버로 직접 전송되지 않고 hidden input 배열을 만들어 내는 경우, 서버 검증 기준은 화면 컨트롤이 아니라 최종 POST 결과다. 예를 들어 권한 선택 UI가 `permission_keys[]`를 생성한다면 신규 부여 action은 빈 배열과 허용되지 않은 토큰을 모두 거절해야 한다.
- 선택지 설명이나 판정 기준처럼 본문에 두기에는 긴 도움말은 항목명 앞의 `?` 아이콘으로 열리는 설명 모달에 둔다. 도움말 문구는 일반 운영자가 이해할 수 있는 쉬운 말로 쓰고, `fallback`, `canonical`, `locale`, `key`, `slug` 같은 개발자 용어는 꼭 필요할 때만 쉬운 설명과 함께 사용한다. 라벨 버튼은 `sr_admin_form_label_help_html()`, 설명 모달 껍데기는 `sr_admin_help_modal_html()`을 사용해 관리자 화면 간 위치와 접근성 속성을 맞춘다. 도움말이 기존 모달 안에서 열릴 수 있으면 부모 모달을 닫지 않도록 스택 모달로 처리한다. 필수 입력의 라벨 도움말에는 `sr_admin_form_label_help_html()`의 필수 표시 인자를 사용해 `(필수)` 표기를 유지한다.
- 모달 푸터에서 닫기/취소처럼 저장하지 않고 모달을 닫는 버튼은 `modal-action`과 `data-overlay`를 유지하고 `btn-solid-light` 같은 보조 버튼 스타일을 사용한다. 저장/적용 같은 주요 액션이 함께 있으면 공통 CSS가 닫기 계열 버튼을 왼쪽에 붙이고 주요 액션은 오른쪽에 남긴다. 닫기/취소만 있는 모달은 버튼을 오른쪽에 둔다.
- 활성 회원 그룹 중 여러 항목을 선택하는 설정은 그룹 key 직접 입력이나 다중 셀렉트 대신 `sr_admin_member_group_key_select_html()` 체크박스 목록을 사용한다. 저장 처리는 배열 POST를 우선 받되 기존 문자열 입력도 정규화해 과거 데이터나 수동 요청과 호환되게 둔다.
- 체크박스와 라디오처럼 컨트롤 옆 문구가 실제 레이블인 항목은 좌측 설명 칸과 우측 조작 레이블을 view에 직접 둔다. 좌측에는 전체 문구를 그대로 표시하고, 우측의 시각적 레이블은 `허용`, `사용`, `확인했습니다.`처럼 필요한 부분만 보이게 줄인다. 우측에서 생략된 맥락은 `.sr-only` 텍스트로 남겨 보조기기가 전체 의미를 읽을 수 있게 한다.
- 체크박스, 라디오, 여러 입력이 묶인 복합 행처럼 좌측 문구가 특정 입력 하나에 직접 연결되지 않는 경우 `admin-form-row` 바로 아래에 `span.form-label`을 둔다.
- 하나의 행에서 값 컨트롤과 항목별 복사 범위 라디오를 반복 배치할 때는 각 컨트롤을 `admin-setting-unit`으로 감싸 값과 복사 범위가 한 묶음으로 읽히게 한다.
- 자산 선택과 사용/과금 체크가 한 항목 안에 함께 있을 때는 자산 선택 영역에 `data-admin-asset-enable-target="#사용_또는_과금_체크박스_ID"`를 둔다. 운영자가 체크박스를 끈 상태에서 자산을 새로 선택하면 공통 관리자 JS가 확인을 받은 뒤에만 사용/과금 체크를 켜고, 취소하면 방금 선택한 자산을 되돌린다. 해당 묶음을 조작한 뒤 사용/과금은 꺼져 있고 자산은 선택된 상태로 저장하려 하면, 저장 시 자산 선택이 실제 동작에 적용되지 않는다는 안내를 확인한 경우에만 제출한다. 과거 저장값이나 기본값도 같은 기준으로 항상 확인해야 하는 화면은 `data-admin-asset-enable-submit-check="always"`를 함께 둔다.
- 복합 차감 자산은 자산 체크박스와 해당 금액 입력을 같은 `input-group` 묶음으로 표시한다. 차감 순서 안내 문구는 `차감 우선순위: 포인트, 적립금, 예치금`처럼 실제 우선순위를 설명하고, `선택한 자산마다 금액을 입력하세요` 같은 중복 입력 지시 문장은 쓰지 않는다.
- 게시판/콘텐츠 편집 화면의 적용 범위 라디오는 화면에는 `단독`, `그룹`, `전체`처럼 짧게 보이고 `적용` 문구는 `sr-only`로 보조한다. `그룹`이나 `전체`를 선택하면 저장 전 의미를 즉시 알 수 있도록 토스트로 같은 그룹/전체 대상에 적용된다는 안내를 보여준다.
- 좌측 레이블이 필요 없는 안내 문장은 입력항목 행으로 만들지 말고 일반 텍스트 문단으로 둔다.
- 저장, 생성, 변경 버튼은 `admin-form-sticky-actions` 안에 두고 문서 흐름 안에서 출력한다. 이 영역은 관리자 화면 하단에 스티키로 붙어 긴 등록/수정/설정 화면에서도 제출 버튼을 유지한다.
- 새 항목 작성 전용 화면은 생성 성공 후 목록 화면으로 redirect하고, 수정 저장은 현재 편집 화면에 남긴다. 생성 성공 안내는 목록 화면의 공통 토스트로 표시한다.
- 목록 화면의 검색 폼, 테이블 행의 상태 변경/삭제 폼, 툴바 폼은 페이지 폼 레이아웃을 적용하지 않는다.

체크박스 행 예시는 다음처럼 작성한다.

```php
<div class="admin-form-grid">
    <div class="admin-form-row">
        <span class="form-label">공개 회원가입 허용</span>
        <div class="admin-form-field">
            <label class="admin-form-check form-label">
                <input type="checkbox" name="allow_registration" value="1" class="form-checkbox">
                <?php echo sr_admin_choice_label_html('공개 회원가입 허용'); ?>
            </label>
        </div>
    </div>
</div>
```

회원 설정의 선택 프로필 항목처럼 한 항목에 여러 불리언 옵션이 붙는 경우에도 같은 `admin-form-row` 안에 체크박스를 나란히 둔다. 선택 프로필은 항목별로 `보이기`와 `필수입력`을 제공하며, `필수입력`을 켜면 `보이기`도 함께 켜지고 `보이기`를 끄면 `필수입력`도 꺼진다.

## 목록 화면

목록형 화면은 공통 관리자 목록 원형을 기준으로 맞춘다.

- 사이드 메뉴에 없는 하위 흐름이나 필터성 이동 링크는 `admin-local-nav-wrap`과 `admin-local-nav`를 사용한다.
- 목록 상단에 전체/상태별 건수 요약이 필요하면 `admin-local-nav-wrap` 오른쪽에 `admin-summary-stats`와 `admin-summary-meta`를 둔다.
- 목록 테이블은 `admin-card card` 또는 `admin-list-card admin-card card` 섹션 안에서 `table-wrapper`와 `table`을 사용한다.
- 테이블 전체를 감싸는 일괄 작업 폼은 `admin-list-form`을 사용한다.
- 행 단위 관리 버튼은 `admin-table-actions-cell`과 `admin-row-actions` 안에 둔다.
- 목록 테이블은 공통적으로 `width: 100%`와 `min-width: 100%` 기준으로 카드 너비 안에 먼저 맞춘다. 날짜, 상태, 수량, 액션처럼 짧은 열은 `width: 1%`와 nowrap 기반으로 내용만큼만 차지하게 하고, 제목, 메시지, 설명처럼 긴 열이 남은 폭을 사용하게 한다. 테이블 전체에 임의의 `min-width`를 두면 빈 목록이나 짧은 목록에도 불필요한 가로 스크롤이 생기므로, 폼 컨트롤이나 행 액션처럼 실제 최소 폭이 필요한 요소에만 최소 폭을 둔다. 컬럼을 제거하거나 추가하면 폭 지정과 빈 상태 `colspan`을 같이 조정한다.
- 테이블형 목록의 데이터가 없을 때도 `thead`는 유지하고, `tbody` 안에 전체 열을 덮는 `td.admin-empty-state[colspan]` 빈 행을 출력한다. 카드형 목록, 설정 폼 내부 보조 목록, 공개/회원 화면처럼 테이블 구조가 아닌 영역은 문장형 빈 상태를 사용할 수 있다.
- `/admin/ui-kit`와 `/ui-kit`의 테이블 샘플은 여러 변형을 나열하지 않고 실제 관리자 회원 목록에 가까운 실사용형 예제 하나를 기준으로 보여준다. 샘플은 `table-card`, `table-list`, `table-summary`, `table-sort-*`, `table-status`, `table-actions-cell`을 사용하고, `sample-table-*`, `admin-table-*`, `ui-table-*`처럼 샘플/관리자/화면 접두어가 붙은 테이블 컴포넌트 이름은 쓰지 않는다.
- 목록에서 내부 DB ID를 별도 칼럼이나 `#123` 같은 본문 텍스트로 직접 노출하지 않는다. 행 관리에 필요한 ID는 hidden input, URL 파라미터, `data-*` 속성처럼 동작 식별자로만 사용하고, 운영자가 보는 목록에는 key, 제목, 회원 공개 이름/이메일, 날짜, 상태처럼 업무 식별에 필요한 값을 우선 표시한다. 포인트, 적립금, 예치금 같은 회원 자산 잔액/거래 목록은 회원 공개 해시를 본문에 직접 표시하지 않고 회원 정보 바로가기와 회원 공개 이름/이메일로 식별한다.
- 관리자 목록, 검색 결과, 행 요약, 조정 모달의 선택 회원 요약처럼 운영자가 회원을 식별하는 화면 조각은 회원 목록과 같은 표시 기준을 쓴다. 공개 이름은 `sr_admin_member_display_name_preview()`, 이메일은 `sr_admin_member_email_display()`로 출력하고, 입력/수정 폼이나 본인 계정 화면처럼 원문 확인이 필요한 곳만 실제 이름 값을 그대로 보여준다.
- 관리자 작업 로그처럼 표 셀에 긴 JSON, 메타데이터, 원문 로그를 제공해야 하는 경우에는 테이블 안에서 펼치지 않고 `btn` 계열 보기 버튼으로 모달을 열어 확인하게 한다. 테이블 셀은 스캔 가능한 짧은 액션만 남기고, 긴 내용은 모달 본문에서 줄바꿈과 스크롤을 처리한다.
- 목록별 필수/보조/관리용 컬럼 기준은 [관리자 목록 컬럼 기준](admin-list-columns.md)을 따른다. 좁은 화면에서는 이 기준의 핵심 컬럼을 남기고 보조 컬럼을 숨길 수 있다.

목록 위 필터는 테이블 카드 안에서 임의의 문단으로 붙이지 않고 `form.admin-filter.ui-form-theme > .admin-filter-grid > .admin-filter-field` 구조를 사용한다. 제목이나 초기화 버튼이 필요하면 `.admin-filter-header`를 먼저 둔다. 필터가 목록 범위를 바꾸는 조건일 때는 목록 위에 두고, 화면 전체 범위를 바꾸는 조건일 때는 목록 섹션 바깥에 둔다.

관리자 작업 로그처럼 조건 수가 많은 필터는 화면 전용 grid class로 폭을 조정한다. 이벤트 유형, 대상 유형, 처리자 계정처럼 같은 방식으로 검색할 수 있는 조건은 검색 조건 select와 검색어 input으로 묶고, 결과와 처리자 유형처럼 값이 적고 여러 값을 함께 고를 수 있는 조건은 체크박스 그룹으로 제공한다. 날짜, IP, 버튼처럼 짧은 조건은 필요한 폭만 사용한다.

회원 검색, 관리자 권한, 회원 자산 잔액/조정/거래 내역처럼 회원을 찾는 필터는 같은 검색 조건/검색어 구조를 사용한다. 계정 상태, 권한, 검색 조건처럼 짧은 select는 내용 폭만 차지하게 두고, 검색어 input이 남은 폭을 사용하며, submit은 기본 버튼 폭을 유지한다.

포인트, 적립금, 예치금처럼 잔액 목록에서 대상 회원이 이미 분명한 경우에는 조정 작업을 독립 화면으로 분리하지 않고 잔액 화면의 행 단위 모달에서 처리한다. 잔액 화면에는 기본 조정 모달 버튼을 항상 노출해 운영자가 회원 공개 해시를 직접 입력할 수도 있게 한다. 환불 처리는 잔액 화면이 아니라 거래 내역 화면의 행 단위 모달에서 처리하고, `refund` 거래 유형과 `refund` 참조 유형을 고정하며 참조 ID에는 원거래 ID를 남긴다. 환불은 조정 모달의 일반 거래 유형으로 제공하지 않으며, 차감 원거래를 선택하지 않은 직접 POST도 서버에서 거부한다. 포인트/적립금/예치금 환불은 차감된 거래를 복원하는 용도로만 노출하고, 원거래의 남은 환불 가능 수량/금액을 넘을 수 없다. 적립금 지급 내역을 되돌릴 때는 환불이 아니라 회수를 사용하고, 예치금을 회원에게 내보낼 때는 원장 환불이 아니라 출금 또는 예치금 환불 신청 처리를 사용한다. 적립금 회수도 거래 내역 화면의 행 단위 모달에서 처리하고, 일반 잔액 조정 모달에는 회수 거래 유형이나 회수 참조 유형을 노출하지 않는다. 회수 거래는 환불 대상으로 제공하지 않는다. 조정/환불/회수 모달의 회원, 거래 유형, 수량/금액, 사유, 참조 항목에는 운영자용 도움말을 둔다. 조정 모달에서 회원이나 참조값을 찾아 넣어야 할 때는 UI-KIT 다중 모달처럼 현재 조정 모달을 검색 모달로 전환하고, 검색 결과 적용 또는 돌아가기 버튼으로 조정 모달을 다시 연다. 최종 저장은 기존 자산 거래 action 제출로 처리한다. 거래 저장에 성공하면 flash 메시지를 남긴 뒤 현재 잔액 또는 거래 내역 GET 화면으로 redirect해 새로고침 중복 실행을 막는다. 서버 검증과 원장 저장은 각 모듈 action이 계속 소유하고, 과거 `/admin/*/adjust` 직접 접근은 같은 회원의 잔액 화면으로 되돌린다.

포인트/금액 환전 정책 화면은 정책 목록을 기본 화면으로 두고, 정책 등록/수정은 모달에서 처리한다. 목록 필터는 상태, 출금 항목, 입금 항목처럼 짧은 조건을 체크박스 그룹으로 제공하고 `fit-content` 폭으로 배치하며, 필터 결과가 없는 경우 조건에 맞는 정책이 없다는 빈 상태를 보여준다. 환전 환경설정 화면은 새 정책 등록 모달의 상태, 비율, 반올림, 환전량 제한, 수수료, 정렬순서 기본값만 저장하며, 저장 시 기존 정책 행을 자동 변경하지 않는다. 환경설정 기본값 항목에는 새 정책 등록에만 적용되고 기존 정책에는 소급되지 않는다는 도움말을 둔다. 정책 모달은 등록일 때 환경설정 기본값을 초기값으로 참고하고, 수정이나 저장 오류 복구 때는 해당 정책/입력값을 우선한다. 정책 모달은 출금/입금 항목, 상태, 비율, 반올림, 환전량 제한, 수수료 적용/기준/금액, 정렬순서처럼 저장 결과와 회원 화면 노출에 영향을 주는 항목에 운영자용 도움말을 둔다. 도움말은 환전 실행 시점의 계산 방식, 비활성 항목 포함 정책의 실행 불가 상태, 수수료 원장 기록 방식을 쉬운 말로 설명한다. 비율 입력은 `분자`/`분모` 같은 수학 용어를 라벨로 노출하지 않고 `환전 비율` 한 항목에 `출금 기준량:입금 환산량` 형식으로 입력하게 한다. 수수료 적용이 `사용 안 함`이면 수수료 설정 입력을 숨기고 서버 저장값도 비운다. 수수료를 적용할 때는 `수수료 방식`에서 정률 또는 정액 중 하나만 고르게 하며, 정률 수수료 입력은 기준값 100을 내부 고정한 퍼센트 숫자 하나만 입력하게 한다.

새 콘텐츠 추가/수정처럼 여러 공개 정책을 한 화면에서 다루는 폼은 Slug, 콘텐츠 그룹, 상태, 레이아웃, 유료 열람, 완료 버튼 포인트/금액 처리, 공개 표시, 다운로드 파일 연결 항목에 운영자용 도움말을 둔다. 도움말 문구는 주소 이름, 공개 여부, 전역/그룹 기본값과 `그룹`/`전체` 적용의 차이, 포인트/금액 차감/지급 시점처럼 일반 운영자가 실제 저장 전에 판단해야 하는 내용을 쉬운 말로 설명한다. 완료 버튼 포인트/금액 처리처럼 지급/차감 방향을 선택하는 화면은 운영자에게 `자산`보다 `처리 항목`, `포인트/금액`, `잔액` 표현을 우선 사용한다. 지급 방향은 처리 항목을 하나만 선택하게 하고 라디오로 표시하며, 차감 방향만 체크박스로 여러 항목을 선택할 수 있게 한다. 회원 그룹별 적용 선택지는 현재 선택한 처리 항목과 맞는 정책만 보여주고, 뱃지 요약도 같은 항목명과 금액을 표시한다. 그룹별 적용 요약은 차감 방향에서 `-2000`처럼 부호를 표시해 실제 회원 잔액 변화 방향이 드러나게 한다. 다운로드 파일은 전용 관리 화면에서 먼저 등록하고, 콘텐츠 생성/수정 화면에서는 사용 상태 파일을 셀렉트+뱃지 형태로 연결만 하도록 설명한다. 파일 제목, 숨김 상태, 다운로드 과금은 다운로드 파일 관리 화면이 소유한다.

새 배너 추가/수정처럼 출력 위치와 노출 조건을 함께 다루는 폼은 링크, 이미지, 출력 위치, 매칭 방식, 특정 subject ID, 상태, 스킨, 기간, 정렬처럼 저장 결과 판단이 필요한 항목에 운영자용 도움말을 둔다. 도움말 문구는 어느 화면에 표시되는지, 공용 배너와 슬롯 대상의 차이, 특정 항목에만 보이는 조건, 이미지 URL과 업로드 우선순위처럼 저장 결과를 예측하는 데 필요한 내용을 쉽게 설명한다.

새 팝업 추가/수정처럼 노출 대상과 닫기 정책을 함께 다루는 폼은 상태, 팝업 스킨, 노출 대상, 매칭 방식, 특정 subject ID, 시작/종료 시각, 닫기 유지일처럼 저장 결과 판단이 필요한 항목에 운영자용 도움말을 둔다. 제목이나 내용처럼 바로 알 수 있는 기본 입력에는 도움말을 붙이지 않고, 공용 팝업레이어와 모듈 노출 대상의 차이, 특정 항목 조건, 방문자가 닫은 뒤 다시 보이는 시점을 쉽게 설명한다.

커뮤니티 환경설정, 게시판, 게시판 그룹처럼 레벨, 쪽지 보내기 권한, 게시판 접근 권한, 첨부, 배너/팝업, 회원 자산 정책을 함께 다루는 폼은 레벨 사용/자동 재계산, 쪽지 보내기 권한, 회원 그룹, 최소 레벨, 게시판 그룹, 상태, 스킨, 첨부 설정, 허용 파일 확장자, 배너/팝업레이어, 회원 자산, 커뮤니티 홈 레이아웃, 정렬처럼 운영 판단이 필요한 항목에 도움말을 둔다. 제목, 이름, 설명처럼 바로 알 수 있는 기본 입력에는 도움말을 붙이지 않고, 쪽지를 보낼 수 있는 회원과 받는 회원의 차이, 전역/그룹 기본값과 `그룹`/`전체` 적용의 차이, 접근 정책과 레벨 조건, 회원 자산 지급/차감 시점, 적립 회수 실패 조건을 쉽게 설명한다. 멤버 그룹별 적용 선택지는 현재 선택한 자산과 맞는 정책만 보여주고, 뱃지 요약도 같은 자산명과 금액을 표시한다.

회원 그룹별 금액 조정처럼 여러 조건 행을 편집하는 설정은 기본 정보 폼 행 안에 끼우지 않고 별도 목록형 섹션으로 분리한다. 섹션은 다른 목록 화면처럼 `card-header`, `table-wrapper`, `table` 흐름을 따르고, 규칙 추가 버튼은 카드 헤더 오른쪽에 둔다. 회원 그룹, 계산 방식, 자산처럼 규칙 저장에 필요한 열은 헤더에 `(필수)`를 표시하고, 조정값 입력 여부와 계산 방식을 좌우하는 계산 방식 열에는 도움말 버튼으로 상세 설명을 제공한다. 증감 조정은 0 이상의 정수 조정값을 입력하고 지급 맥락에서는 더하며 차감 맥락에서는 빼는 방식으로 설명한다. 커뮤니티 멤버 그룹별 금액 조정처럼 레벨 조건을 지원하는 화면은 최소 레벨 열을 함께 표시하고 저장 POST에도 포함한다. 여러 자산을 지원하는 조정 규칙은 한 행에 자산 하나와 조정값 하나만 설정하게 하고, 여러 설정 항목을 한 폼 행 안의 중첩 테이블로 넣지 않는다.

콘텐츠와 커뮤니티의 실제 자산 설정 화면은 사용 여부, 과금 방식, 버튼 문구, 처리 방향처럼 기능 동작을 결정하는 행과 포인트/금액 항목, 자산별 금액, 회원/멤버 그룹별 적용처럼 잔액 처리에 필요한 자산 설정 행을 구분해 표시한다. 사용 여부는 자산 선택 여부가 아니라 `유료 열람 사용`, `댓글 차감 사용` 같은 별도 체크박스로 결정한다. 미리 만든 정책은 콘텐츠에서는 `회원 그룹별 적용`, 커뮤니티에서는 `멤버 그룹별 적용`으로 표시하고 셀렉트에서 하나 이상 선택하게 한다. 도움말에는 선택한 적용이 회원/멤버의 그룹, 커뮤니티 레벨, 선택한 포인트/금액 항목에 맞는 실제 금액을 계산한다는 점을 안내한다. 그룹별 계산 방식, 자산, 조정값 편집은 전용 금액 조정 관리 화면에서만 수행하고, 콘텐츠나 게시판 설정 화면에서는 직접 조정값을 편집하지 않는다.

콘텐츠와 커뮤니티의 실제 자산 설정 화면에서 회원/멤버 그룹별 적용을 셀렉트로 선택하면 하단에 선택 뱃지를 추가한다. 뱃지에는 적용 이름과 그룹별 포인트/금액 실제 적용값 요약을 함께 보여주고, 제거 버튼으로 선택을 제거할 수 있다. 이미 추가한 적용은 셀렉트 선택지에서 제외하고 제거하면 다시 선택할 수 있게 한다.

도움말 모달 본문은 `sr_admin_help_modal_html()`의 `.admin-help-modal-body` 공통 스타일을 사용한다. 문단, 목록, 문단과 목록 사이 간격은 화면별 CSS가 아니라 공통 스타일에서 맞춘다.

저장, 삭제, 적용 같은 짧은 결과 안내는 `sr_admin_feedback_toasts($notice, $errors)`를 사용해 토스트로 출력한다. 화면 본문에 영구적으로 남아야 하는 설명과 작업 결과 피드백을 섞지 않는다.

## 대시보드

관리자 대시보드는 사이트 기본 정보, 복구 주의 항목, 활성 모듈의 `dashboard.php` 계약 섹션을 표시한다. 설치 보호, 인증 런타임, 고위험 설정, 전체 모듈 목록처럼 운영 진단 성격이 강한 항목은 대시보드 기본 섹션으로 노출하지 않는다. 각 섹션은 `data-admin-dashboard-section` 값을 가진 독립 카드로 두고, 사용자가 드래그 앤 드롭으로 순서를 바꾸면 브라우저 `localStorage`에 표시 순서가 저장된다.

모듈별 대시보드 섹션의 내부 구성은 모듈이 소유한다. 모듈은 `dashboard.php`에서 `view`로 모듈 내부 `views/*.php` 파일을 지정할 수 있고, admin 모듈은 검증된 view를 대시보드 섹션 안에 include한다. admin 모듈은 외곽 wrapper, 이동 핸들, 정렬 저장을 맡고, 모듈 view는 제목 배치, 지표 구성, 링크, 도메인별 리듬을 직접 렌더링한다. 필요한 스타일은 각 모듈의 `admin.stylesheets`로 선언한 모듈 내부 CSS에서 소유한다.

모듈은 `dashboard.php`에서 `default_visible`로 기본 노출 여부를 선언할 수 있다. 대시보드의 섹션 관리 UI는 표시/숨김을 브라우저 `localStorage`에 저장하며, 사용자가 저장한 값이 있으면 모듈 기본값보다 우선한다. 섹션을 숨기면 grid가 즉시 남은 섹션으로 재계산되고, 한 행에 남은 섹션 수에 따라 4열, 3열, 2열, 1열 전체 폭으로 보정한다. 숨긴 섹션을 다시 표시하면 대시보드 맨 마지막에 한 줄 전체를 차지하는 섹션으로 추가한다.

`view`가 없는 기존 모듈은 admin 모듈의 fallback layout으로 렌더링한다. fallback의 `table`은 기존 `항목 / 주요 수치 / 상세` 표이고, `stats`는 지표 카드형 요약이다. 알 수 없는 layout, state, emphasis 값은 admin 모듈이 안전한 기본값으로 되돌린다. fallback row의 라벨과 상세 접두/접미 문구는 `sr_t()`로 제공하고, SQL은 숫자나 코드 값만 반환하게 둔다.

대시보드 섹션은 화면 구조를 바꾸는 관리 도구가 아니라 개인 작업 배치에 가깝다. 따라서 드래그 순서는 DB에 저장하지 않고, 모듈이 제공하는 기본 표시 순서는 `dashboard.php`의 `order` 값으로 유지한다. 섹션 이동은 섹션 전체가 아니라 섹션의 아이콘 핸들에서만 시작하며, 관리자 대시보드 본문은 최대 4열 그리드로 배치한다. 다른 섹션의 좌우에 놓을 때는 수직 고스트 라인을 표시하고 일반 폭으로 배치하며, 1열 전체 폭 섹션의 좌우에 놓으면 기존 1열 섹션과 이동 섹션을 같은 행의 2열로 전환한다. 위아래에 놓을 때는 행 사이 슬롯 중앙에 수평 고스트 라인 하나만 표시하고 한 줄 전체를 차지하는 1열 섹션으로 배치할 수 있다. 드롭 결과는 드롭 순간의 마우스 위치를 다시 해석하지 않고 마지막으로 표시된 고스트 라인의 위치를 기준으로 결정한다. 고스트 라인은 드래그 중 카드 배치를 밀지 않는 overlay로만 표시한다. 4열 기준 도입처럼 저장 구조가 바뀌는 경우에는 정렬 저장 key를 변경해 기존 브라우저 정렬 배열을 초기화한다.
