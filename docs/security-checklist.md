# 보안 체크리스트

산란의 기능을 추가하거나 수정할 때 확인할 최소 보안 체크리스트입니다.

## 공통

- 외부 입력을 직접 신뢰하지 않았는가
- enum 값은 허용 목록으로 검증했는가
- 숫자 ID는 정수 변환과 범위 확인을 했는가
- 출력값은 컨텍스트에 맞게 escape했는가
- 오류 화면에 민감 정보가 노출되지 않는가
- debug 오류 메시지에도 민감한 key/value 또는 Bearer 값이 표시 전 마스킹되는가
- 로그에 비밀번호, 토큰, 개인정보 원문이 남지 않는가
- 사용자 입력 HTML은 sanitizer 정책 없이 저장하거나 출력하지 않는가
- 보안 판단이 애매한 경우 [산란 보안 모델](security-model.md)의 참고 기준과 충돌하지 않는가

## 공개 소스 전제

- 공격자가 산란 사용 여부, 기본 디렉터리 구조, 기본 테이블명, 요청 path를 알고 있어도 방어되는가
- 관리자 URL, 파일명, 테이블 prefix, 모듈 key를 숨기는 방식에 보안을 의존하지 않는가
- `sr_` table prefix를 보안 장치가 아니라 네임스페이스로만 취급했는가
- 설치/업데이트 경로가 알려져도 설치 상태, 인증, 권한 검증으로 실행이 차단되는가
- 직접 action 파일 위치가 알려져도 허용된 method/path, 활성 모듈, 권한, CSRF 검증 없이는 상태 변경이 불가능한가

## DB

- 동적 값은 `PDO::prepare()`와 placeholder로 바인딩했는가
- `PDO::query()`는 외부 값이 섞이지 않는 고정 SQL에만 사용했는가
- 설치/업데이트 SQL 파일은 코어 SQL 실행 helper로만 실행했는가
- 테이블명, 컬럼명, 정렬 방향은 허용 목록으로 제한했는가
- SQL 오류를 운영 화면에 그대로 출력하지 않는가
- 설치/업데이트 SQL은 재실행 위험을 검토했는가

## 인증과 세션

- 비밀번호는 `password_hash()`로 저장하는가
- 비밀번호 검증은 `password_verify()`를 사용하는가
- 새 비밀번호 입력은 원문이 허용 길이를 넘을 때 잘라 저장하지 않고 거부하는가
- 설치 최초 관리자 비밀번호 입력도 원문이 허용 길이를 넘을 때 잘라 저장하지 않고 거부하는가
- 관리자 회원 직접 추가 비밀번호도 원문이 허용 길이를 넘을 때 잘라 저장하지 않고 거부하는가
- 로그인 성공 후 `session_regenerate_id(true)`를 실행하는가
- 세션 쿠키에 `HttpOnly`, `Secure`, `SameSite`를 적용하는가
- 장기 로그인이나 재설정 토큰은 hash만 저장하는가
- GET token 입력은 원문이 허용 길이를 넘을 때 잘라 검증하지 않고 거부하는가
- 회원가입 이메일은 원문이 허용 길이를 넘을 때 잘라 저장하지 않고 거부하는가
- 회원가입 로그인 아이디는 원문이 허용 길이를 넘을 때 잘라 저장하지 않고 거부하는가
- 비밀번호 재설정 요청 이메일은 원문이 허용 길이를 넘을 때 잘라 조회하지 않고 거부하는가
- 로그인 식별자는 원문이 허용 길이를 넘을 때 잘라 계정을 조회하지 않는가
- 이메일 로그인은 항상 `email_hash` 기준으로 조회하는가
- 로그인 아이디가 있는 계정은 같은 로그인 폼에서 `login_id_hash` 기준으로도 조회되는가
- 로그인 아이디 중복은 `login_id_hash`와 호환용 `account_identifier_hash` 기준으로 함께 차단하는가
- 회원-facing 개인정보 요청 티켓 등록은 기본 비활성이고, 직접 POST는 요청 row를 만들지 않는가
- 관리자 개인정보 대응 기록 추가는 계정 ID 또는 요청자 식별값 중 하나와 요청 내용을 서버에서 필수 검증하는가
- 운영자가 기록한 개인정보 대응 내용은 원문이 허용 길이를 넘을 때 잘라 저장하지 않고 거부하는가
- 커뮤니티 게시판별 필수 checkbox 추가 입력은 서버에서도 체크된 값만 유효하게 처리하는가
- 커뮤니티 게시판별 추가 입력값은 원문이 허용 길이를 넘거나 배열 payload이면 잘라 저장하지 않고 거부하는가
- 커뮤니티 게시판 그룹 추가 입력 저장 시 그룹 값을 상속하는 게시판의 정의 테이블도 동기화되는가
- 커뮤니티 관리자 게시글 추가 입력 검색은 `show_in_admin` 표시 대상 값만 검색하는가
- 커뮤니티 게시판별 추가 입력 개인정보 사본 제공은 값 테이블과 snapshot 모두에서 export 포함 정책인 값만 내보내는가
- 커뮤니티 게시판별 추가 입력 cleanup은 값 테이블과 snapshot 모두에서 `retain`이 아닌 값만 비우고 legacy 정책 값은 익명화하는가
- 로그아웃 시 관련 세션/토큰을 폐기하는가
- 개인정보 처리 요청 목록은 관리자 메모를 전체 원문 대신 축약해 표시하는가
- 회원 개인정보 사본 내려받기는 현재 비밀번호 재인증을 요구하는가
- 모듈별 개인정보 사본 수집 실패가 전체 내려받기를 중단하지 않는가
- 모듈별 개인정보 사본 수집은 배열 반환값만 포함하는가
- 모듈별 개인정보 사본 수집에서 내부 hash/token/secret-like 필드를 제거하는가
- 회원별 접근/차감 로그를 추가한 모듈은 `privacy-export.php`에 해당 기록을 포함하는가
- 회원별 게시글/시리즈 스크랩처럼 `account_id`에 직접 연결된 개인 보관 데이터가 개인정보 사본 제공에 포함되는가
- 개인정보 사본 JSON은 다운로드 헤더 전 안전하게 인코딩되고 비정상 UTF-8을 치환하는가
- 회원 아바타는 URL 직접 입력이 아니라 업로드로만 처리하고, 확장자 allowlist, MIME 검사, 이미지 재인코딩을 모두 통과하는가
- 공개 댓글 멘션 후보 API는 로그인 회원에게만 공개 이름과 public account hash prefix 같은 최소 식별 정보만 반환하고, 이메일, 내부 계정 ID, 가입일을 노출하지 않으며, rate limit을 적용하는가

## CSRF

- 모든 상태 변경 요청에 CSRF 검증이 있는가
- 자산 차감, 쿠폰 소진, 환전, 원장 조정처럼 금전성 상태 변경이 GET 요청에서 발생하지 않는가
- 로그인 후 CSRF 토큰을 갱신하는가
- 관리자 화면의 POST 요청도 CSRF 검증을 하는가
- SameSite 쿠키 설정을 CSRF 토큰 대체 수단으로 취급하지 않는가

## 자산/원장

- 여러 자산을 한 번의 사용자 동작으로 차감/적립할 때 전체 작업이 하나의 DB 트랜잭션으로 묶이는가
- dedupe placeholder, 원장 transaction, 접근권/entitlement 부여가 서로 다른 커밋 경계로 갈라져 무료 통과나 부분 과금을 만들지 않는가
- 반복 과금 정책은 프리페치, 링크 미리보기, 자동 재요청으로 실행되지 않도록 명시 확인 흐름을 거치며, 성공 후 POST 재전송이 같은 과금을 반복하지 않도록 redirect 또는 1회성 접근권으로 끊는가
- 환전 정책의 반올림/수수료/양방향 조합이 반복 환전으로 가치를 만들지 않는가
- 금전성 관리자 수동 조정에는 운영 상한, 감사 로그, 필요 시 대액 승인 흐름이 있는가

## 요청 contract

- action 파일이 `exit` 또는 `die`를 직접 호출하지 않는가
- action 파일이 `header('Location: ...')`를 직접 호출하지 않고 `sr_redirect()`를 사용하는가
- 다운로드처럼 응답을 직접 끝내는 action은 `sr_finish_response()`를 사용하는가
- 관리자 action은 `sr_member_require_login()`과 `sr_admin_require_permission()` 또는 `sr_admin_require_owner()`를 모두 호출하는가
- action 파일이 low-level contract mark 함수를 직접 호출하지 않는가
- 권한 판단에 `$_SERVER['REQUEST_URI']` 원문 대신 정규화된 `sr_request_path()` 기준을 사용하는가

## 권한

- 로그인 확인과 권한 확인을 분리했는가
- 화면 버튼 숨김에만 의존하지 않고 서버에서 권한을 검증하는가
- 관리자 action 파일 진입 시 권한을 먼저 확인하는가

## 모듈과 include

- 모듈 키는 허용된 형식으로 검증했는가
- action 파일 경로는 모듈 디렉터리 내부 상대 경로로 제한했는가
- 활성화되지 않은 모듈 action 파일을 include하지 않는가
- 설정값을 검증 없이 파일 경로로 조립하지 않는가
- 읽기 참조 계약은 활성 모듈의 선언된 계약 파일만 읽고, helper 경로를 모듈 폴더 안으로 제한했는가
- 공유 대상 삭제/비활성화/사이트명 변경 같은 민감 POST는 최신 읽기 참조 결과와 계약 오류를 서버에서 다시 확인하는가
- 읽기 참조 결과를 근거로 소비 모듈 정책 테이블을 자동 update/delete/치환하지 않는가

## 관리자

- 관리자 화면은 기본 `noindex`인가
- 설정 변경, 모듈 변경, 회원 상태 변경은 감사 로그에 남는가
- 관리자 POST 대상 ID는 순수 양의 정수 문자열만 허용하는가
- 회원 관리자 throttle 숫자 설정은 원문이 허용 범위 안의 정수일 때만 저장되는가
- 보관 기간 정리 일수는 원문이 1-3650 범위 정수일 때만 실행되는가
- 데이터 정리 화면은 보관 기간 저장과 삭제 실행을 분리하고, 삭제 실행은 모달 확인 후 처리하는가
- 데이터 정리 화면의 보관일 도움말은 대상별 정리 기준 시각과 수동/자동 정리 실행 시점을 함께 안내하는가
- 요청 기반 자동 정리는 정리 대상 맥락별로 실행하며, 실행 간격과 배치 크기를 제한하고 lock으로 중복 실행을 막는가
- 관리자 작업 로그 필터는 허용된 검색 조건과 결과 값만 적용하고, 이벤트/대상 검색어 형식을 제한하는가
- 관리자 작업 로그 날짜 필터는 실제 존재하는 날짜만 적용하는가
- 관리자 작업 로그 metadata에 민감 키가 섞여도 목록 표시에서 마스킹되는가
- 기존 관리자 작업 로그 message와 metadata 문자열에 민감한 key/value 또는 Bearer 값이 남아 있어도 표시 전 마스킹되는가
- 감사 로그 metadata에 민감 키가 섞이면 저장 전부터 마스킹되는가
- 감사 로그 message와 metadata 문자열에 민감한 key/value 또는 Bearer 값이 섞이면 저장 전부터 마스킹되는가
- 운영 복구 marker에 민감 키가 섞이면 저장 전부터 마스킹되는가
- 기존 운영 복구 marker message에 민감한 key/value 또는 Bearer 값이 남아 있어도 표시 전 마스킹되는가
- 예외 파일 로그에 민감한 key/value가 섞이면 저장 전부터 마스킹되는가
- 개인정보 조회/처리 작업은 별도 로그를 남기는가
- 개인정보 대응 기록 목록은 요청자와 요청 내용을 전체 원문 대신 축약해 표시하는가
- 개인정보 처리 요청 목록 필터는 상태, 요청 유형, 검색 조건 허용 목록 안에서만 적용하는가
- 종결된 개인정보 처리 요청 상태는 다시 미처리 상태로 변경할 수 없게 막는가
- 종결된 개인정보 대응 기록의 메모만 보강할 때 최초 처리 관리자와 처리 시각을 덮어쓰지 않는가
- 관리자 개인정보 처리 요청 메모는 원문이 허용 길이를 넘을 때 잘라 저장하지 않고 거부하는가
- 관리자 개인정보 처리 요청 목록에서 기존 관리자 메모 원문을 입력칸에 미리 노출하지 않는가
- 관리자 개인정보 처리 자료 다운로드는 현재 관리자 비밀번호 재인증을 요구하는가
- 관리자 개인정보 처리 자료 다운로드는 연결 계정 데이터 실패가 전체 요청 자료 생성을 중단하지 않는가
- 회원/관리자 권한 목록은 이메일과 이름을 전체 원문 대신 축약해 표시하는가
- 관리자 화면에도 XSS 방어가 적용되는가
- 모듈 소스 반영, 파일 버전 동기화 같은 고위험 owner 작업은 재인증을 요구하는가
- 관리자 권한 추가와 owner 권한 변경은 현재 owner 비밀번호 재인증을 요구하는가
- 관리자 권한 화면은 전체 회원을 기본 노출하지 않고 필터 결과 안에서만 권한을 변경하는가
- 모듈 zip 업로드 요청 후 모듈 소스 반영 허용 값이 자동으로 비활성화되는가
- 모듈 zip 업로드 사전 검증 실패도 업로드 실패 감사 로그와 자동 비활성화를 남기는가
- 모듈 소스 반영 실패 메시지에 민감한 key/value 또는 Bearer 값이 섞여도 표시 전 마스킹되는가
- 모듈 zip 압축 해제 후 실제 파일 트리가 작업 디렉터리 안에만 남아 있는지 다시 검증하는가
- 모듈 소스 교체 실패 후 기존 백업 복구가 실패하면 성공처럼 지나가지 않고 실패로 처리되는가
- 보안 경계를 바꾸는 on/off 설정은 문자열이 아니라 bool 타입으로만 저장되는가
- 사이트/모듈 설정은 범용 key/value 편집 UI가 아니라 의미, 허용 값, 검증 규칙이 드러나는 전용 화면에서만 바꾸는가

## 개인정보

- 필수 정보와 선택 정보를 분리했는가
- 동의가 필요한 항목은 버전과 시점을 기록하는가
- 삭제/익명화 정책을 확인했는가
- 서비스 접근권처럼 개인정보처리방침의 보유기간이 거래 증빙과 다른 데이터는 원장/사용 로그와 분리되어 있는가
- 개인정보 요청 이력은 계정 삭제 후에도 보존 가능한가

## 배포 경로

- 루트 `index.php`만 공개 진입점으로 사용하는가
- `config/`, `storage/`, `database/`, `core/`, `modules/`, `docs/`, `examples/` 직접 접근을 운영 환경에서 차단했는가
- `.git/`, `.tools/` 직접 접근을 운영 환경에서 차단했는가
- `AGENTS.md`, `README.md` 같은 루트 보조 파일이 웹에서 열리지 않는가

## 파일 업로드

- 업로드 action이 `sr_upload_validate_file()` 또는 같은 수준의 검증을 통과하는가
- 업로드 파일 크기를 제한했는가
- 확장자 allowlist와 MIME 검증을 함께 적용했는가
- 이미지 업로드는 확장자만 변조한 파일을 막기 위해 실제 이미지 판별과 재인코딩을 통과하는가
- 재인코딩 실패 시 원본 저장을 허용하는 모듈은 MIME/확장자/이미지 크기 검증, 실행 확장자 차단, 공개 action의 nosniff 헤더를 함께 적용하는가
- 실행 가능한 확장자를 차단했는가
- 원본 파일명으로 저장하지 않고 서버 생성 이름을 사용하는가
- 가능하면 웹에서 직접 접근할 수 없는 위치에 저장하는가
- 프로젝트 루트가 웹 접근 루트인 공유호스팅에서는 접근 차단 규칙이 적용된 저장 위치만 사용하는가
- 업로드 원본을 `modules/{module_key}/assets`에 저장하지 않는가
- 다운로드는 직접 경로 노출 대신 action 파일을 통하는가
- 비공개 다운로드는 짧은 만료 token이나 같은 수준의 서버 측 권한 검사를 통과하는가
- 모듈 zip은 크기, checksum, 경로, symlink, 파일/디렉터리 경로 충돌, 단일 모듈 구조, module key, version, downgrade 정책을 통과하는가
- 모듈 zip의 금지 파일 검사는 선택된 모듈 폴더뿐 아니라 압축 해제된 zip 전체에 적용되는가
- 모듈 zip은 서버 설정/비밀 파일, 저장소 메타 파일, 인증 정보 파일/디렉터리, DB/백업 파일, 우회 실행 확장자를 거부하는가
- 모듈 zip은 `module.php` 정적 return 배열, `paths.php` 정적 route map, `updates/YYYY.MM.NNN.sql` 파일명과 module.php version 이하 업데이트 버전을 검증하는가
- 모듈 zip은 공개 `assets/` 아래 PHP 실행 파일과 SQL 파일을 거부하는가

## 캐시

- 로그인 상태나 권한에 따라 달라지는 HTML을 캐시하지 않는가
- CSRF 토큰이 포함된 화면을 캐시하지 않는가
- 개인정보가 포함된 화면을 캐시하지 않는가
- locale처럼 출력에 영향을 주는 값이 캐시 키에 반영되는가
