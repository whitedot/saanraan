# 산란 특장점 소개

이 문서는 `saanraan`의 현재 구현 기능과 특장점을 정리한다. 실제 저장소에 포함된 본체와 번들 모듈을 기준으로 작성하며, 본인확인, 결제, 회원 마이그레이션처럼 계획 단계에 있는 기능은 구현 기능으로 설명하지 않는다.

`saanraan`은 저가형 PHP 웹호스팅에서도 올려 쓸 수 있는 회원 중심 모듈형 웹 솔루션 베이스다. 코어는 설치, 요청 진입, 설정, 보안 helper처럼 실행 기반에 집중하고, 회원·콘텐츠·커뮤니티·퀴즈·설문·공통 리액션·본문 임베드·자산·쿠폰·알림·개인정보·사이트 구성은 모듈이 각자 책임진다. 처음부터 모든 기능을 켜지 않고도 회원 사이트를 시작할 수 있고, 이후 운영 목적에 맞춰 필요한 모듈을 차례로 붙일 수 있다.

## 주요 특장점

산란의 핵심 장점은 “작게 시작하고, 필요한 기능을 모듈 단위로 더해 가는 편의”다. 예를 들어 처음에는 회원, 관리자, 개인정보 모듈만 설치해 기본 계정 사이트로 운영하고, 이후 콘텐츠 판매가 필요해지면 콘텐츠와 포인트/적립금/예치금, 쿠폰 모듈을 추가할 수 있다. 커뮤니티 참여를 열고 싶을 때는 게시판, 댓글, 신고, 레벨, 쪽지, 유료 첨부 기능을 커뮤니티 모듈 안에서 관리한다.

주요 편의는 세 가지다.

- 기능을 모듈 단위로 켜고 끌 수 있어 사이트 규모에 맞춰 시작하기 쉽다.
- 관리자 화면에서 회원, 콘텐츠, 커뮤니티, 자산, 쿠폰, 알림, 개인정보 업무를 한 흐름으로 다룰 수 있다.
- 회원과 관련된 활동, 자산, 쿠폰, 알림, 개인정보 대응 기록이 같은 회원 식별자를 기준으로 이어져 문의 대응과 기록 확인이 쉽다.

## 현재 단계

현재 산란은 1.0 릴리스 전 안정화 단계다. 이미 설치와 운영 화면, 회원, 콘텐츠, 커뮤니티, 퀴즈, 설문, 리액션, 자산, 쿠폰, 알림, 개인정보, 사이트 운영 모듈은 실제 코드와 DB 테이블로 구현되어 있다. 본인확인, 외부 결제, 기존 회원 마이그레이션은 계획 문서 단계이며 구현 완료 범위에 포함하지 않는다.

이 문서는 기능의 사용 가치를 설명하지만, 운영 신뢰성을 보증하는 검증 기록은 아니다. 모듈별 `stable-candidate`/`beta` 등급, 미실행 smoke, 수동 점검 필요 항목은 [모듈 상태 등급](module-status.md)과 [검증 상태와 증거 기준](verification-status.md), [프로젝트 리스크 레지스터](risk-register.md)를 기준으로 판단한다.

## 설치와 기본 관리

설치와 기본 관리는 사이트를 처음 열 때 필요한 영역이다. 산란은 별도 빌드 서버나 상시 worker가 없어도 PHP 파일과 DB를 중심으로 시작할 수 있게 설계되어 있으며, 모듈 설치와 DB 업데이트를 화면에서 확인할 수 있다. 운영 중에는 새 기능 파일을 올리는 일과 DB 스키마를 반영하는 일이 분리되어, 기능 추가와 업데이트 상태를 관리자 화면에서 단계별로 볼 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 웹 설치 | 브라우저에서 설치 흐름에 들어갈 수 있어 SSH나 CLI 접근이 어려운 호스팅에서도 초기 구성을 시작하기 쉽다. | 미설치 상태 자동 설치 흐름, DB 기반 사이트 설정 저장 |
| 모듈 설치와 활성화 | 기능을 모듈 단위로 설치·활성화하므로 당장 쓰지 않는 기능을 숨기고 필요한 때에만 열 수 있다. | 모듈 설치, 활성화, 비활성화, 모듈 상태 관리 |
| 모듈 파일 관리 | 외부 모듈 zip을 업로드하거나 파일 업데이트를 반영할 때 버전, 계약, 소유자 재확인을 함께 제공해 파일 교체 과정을 화면에서 따라갈 수 있다. | 모듈 zip 업로드, 파일 업데이트 반영, 다운그레이드 확인, 소유자 비밀번호 재확인, 계약/호환성 표시 |
| 업데이트 관리 | 본체와 모듈의 schema version을 따로 확인해 어떤 SQL이 적용됐고 무엇이 남았는지 화면에서 판단할 수 있다. | 본체 schema version, 모듈 schema version, update SQL 구조 |
| 배포 친화 구성 | 파일 배치와 DB 업데이트가 분리되어 FTP/SFTP, Git checkout, zip 배포 같은 여러 방식에 맞춰 반영할 수 있다. | PHP 파일 기반 배포, Apache 호환 보호 기준, `sr_` DB prefix |
| 저장소 기반 | 로컬 파일과 S3 호환 저장소를 공통 helper로 다룰 수 있어 업로드 기능이 모듈마다 제각각 흩어지는 일을 줄인다. | 로컬 파일 저장 helper, S3 호환 저장소 helper |

## 관리자 대시보드와 관리 도구

관리자 영역은 사이트 운영 업무의 기본 창구다. 산란은 대시보드, 메뉴, 권한, 감사 로그, 보존 정책, UI 기준을 한곳에 묶어 사이트 상태와 운영 작업을 빠르게 살필 수 있게 한다. 여러 사람이 함께 관리하는 사이트에서는 역할별 권한을 나눠 필요한 화면만 맡길 수 있고, 변경 이력은 감사 로그에서 이어서 확인할 수 있다.

## 회원 관리

회원 관리는 산란의 중심 축이다. 다른 모듈에서 포인트를 지급하거나 쿠폰을 사용하거나 알림을 보내더라도, 기준은 회원 계정과 회원 그룹으로 이어진다. 회원 상태, 이메일 인증, 닉네임, 그룹, 프로필 항목을 함께 관리해 권한과 혜택 기준을 일관되게 둘 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 회원가입과 로그인 | 이메일 계정과 로그인 ID를 기반으로 회원 기능을 바로 시작할 수 있고, 다른 모듈은 이 계정 흐름 위에 자연스럽게 올라간다. | 회원가입, 로그인, 로그아웃, 이메일 기반 계정, 로그인 ID |
| 이메일 인증 | 이메일 인증 상태를 계정 데이터와 분리해 보관하므로 인증 전후 정책을 화면과 기능에서 구분하기 쉽다. | 이메일 인증, 이메일 인증 여부 관리 |
| 비밀번호 관리 | 비밀번호 재설정과 변경 흐름이 준비되어 있어 계정 문제를 별도 문의 처리로만 떠안지 않아도 된다. | 비밀번호 재설정, 비밀번호 변경 |

## 콘텐츠 관리

콘텐츠 모듈은 공지, 안내문, 자료 페이지, 연재 글, 유료 자료처럼 사이트가 직접 발행하는 글을 관리한다. 단순히 글을 저장하는 데서 끝나지 않고 예약 발행, 작성자 제출, 커버 이미지, SEO 문구, 시리즈, 댓글, 배너/팝업 연결까지 운영 흐름에 필요한 항목을 함께 제공한다. 콘텐츠 화면 안에서 공개 상태, 주소, 공개 시각, 과금 정책, 댓글, 메뉴와 sitemap 노출을 한 번에 다룰 수 있다.

| 콘텐츠 작성 | 단일 콘텐츠를 만들고 주소를 부여해 공개할 수 있어 공지, 안내, 자료 페이지 같은 기본 콘텐츠를 빠르게 구성한다. | 콘텐츠 작성, 수정, 숨김 처리, `/content/{slug}` 공개 URL |

| 예약 발행 | 콘텐츠를 미리 작성해 미래 시각에 공개할 수 있어 캠페인, 공지, 강의 회차처럼 공개 타이밍이 중요한 운영을 수동 작업 없이 준비할 수 있다. | `scheduled` 상태, 예약 발행 시각, 예약 전 공개 제외, 관리자 미리보기, 예약 공개 전환 감사 로그 |

## 콘텐츠 유료 열람과 파일 다운로드

유료 열람과 다운로드 기능은 콘텐츠를 회원 자산, 쿠폰, 접근권 기록과 연결한다. 무료 글과 유료 글을 같은 콘텐츠 관리 흐름에서 다루면서, 최초 1회 과금, 매번 과금, 회원 그룹별 가격, 쿠폰 우선 적용, 환불과 접근권 회수까지 제공한다. 회원은 보유 자산과 쿠폰으로 콘텐츠를 이용하고, 열람과 다운로드 내역은 관리자 화면에서 이어서 확인할 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 유료 열람 | 무료 콘텐츠와 유료 콘텐츠를 함께 다룰 수 있어 멤버십 자료, 프리미엄 글, 유료 자료실 같은 구성을 만들 수 있다. | 포인트, 적립금, 예치금 기반 유료 열람 |

| 차감 정책 | 최초 1회 차감과 매 열람 차감을 고를 수 있어 영구 접근권과 반복 이용 과금을 콘텐츠 성격에 맞게 구분한다. | 최초 1회 차감, 매 열람 차감 |

| 개인정보 연동 | 유료 접근과 다운로드 기록이 개인정보 사본 제공과 탈퇴 정리에 포함되어 회원 데이터 흐름이 따로 누락되지 않는다. | 처리 로그 기록, privacy export, 탈퇴/익명화 cleanup |

## 커뮤니티

커뮤니티 모듈은 회원이 직접 글을 쓰고 대화하는 영역을 담당한다. 게시판, 게시판 그룹, 카테고리, 댓글, 첨부파일, 신고, 스크랩, 쪽지, 시리즈, 레벨과 그룹 규칙이 모두 커뮤니티 안에서 이어진다. 게시판 성격에 맞춰 공개 범위, 글쓰기 권한, 카테고리, 첨부 허용, 유료 다운로드, 게시자 리워드, 신고 처리 흐름을 관리자 화면에서 조정할 수 있다.

| 기능 | 안내 문구 | 주요 기능 |

## 퀴즈

퀴즈 모듈은 콘텐츠나 커뮤니티 글을 읽은 뒤 바로 참여하는 확인형, 이벤트형, 진단형 흐름에 맞다. 문제와 선택지, 공개 기간, 응시 제한, 통과 기준, 결과 문구, 보상 지급 조건을 한 세트로 관리한다. 응시와 보상은 기록으로 남기 때문에 응시 횟수, 보상 지급 결과, 댓글 문의 위치를 관리자 화면에서 추적할 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 퀴즈 세트 | 단일/복수 선택 문제와 결과 규칙을 묶어 학습 확인, 이벤트 참여, 간단한 진단형 콘텐츠를 만들 수 있다. | 퀴즈 생성과 수정, 문제/선택지 관리, 총점/카테고리 결과 규칙 |

| 콘텐츠/커뮤니티 연계 | 콘텐츠나 커뮤니티 화면에서 퀴즈를 시작하고 완료 뒤 원래 맥락으로 돌아갈 수 있어 글 읽기와 참여 흐름을 자연스럽게 이어준다. | source module/type/id 연결, 콘텐츠/커뮤니티 CTA, 모달 우선, 페이지 fallback, 내부 복귀 경로 |

공통 리액션 모듈은 `좋아요`, `슬퍼요`, `재밌어요`처럼 가벼운 반응을 여러 서비스 화면에 같은 방식으로 붙이는 영역이다. 콘텐츠, 커뮤니티, 퀴즈, 설문과 각 댓글은 자기 대상 검증과 공개 가능 여부만 제공하고, 리액션 모듈은 정의, preset, 원장, 사용 중지 key 정리, 개인정보 export/cleanup을 맡는다. 운영자는 사이트 성격에 맞는 반응 종류와 표시 묶음을 만들고, 각 모듈의 전역/그룹/개별 설정에서 어떤 묶음을 쓸지 선택할 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 반응 정의 | 운영자가 반응 key, 표시명, 아이콘, 색상, 설명, 상태, 정렬을 직접 관리해 서비스 말투에 맞는 반응 버튼을 구성한다. | 리액션 정의 관리, 이모지, Material 아이콘 key, JPG/PNG/WebP 이미지 업로드, 상태와 정렬 |
| Preset 관리 | 여러 반응을 공개 버튼 묶음으로 저장해 콘텐츠, 게시판, 퀴즈, 설문 화면에서 같은 세트를 재사용한다. | preset 생성/수정, 공개 노출 key, 기본 표시 6개, 최대 12개 |
| 대상별 적용 | 콘텐츠/댓글, 커뮤니티 게시글/댓글, 퀴즈/댓글, 설문/댓글에 리액션을 붙이고 상위 설정을 개별 화면에서 덮어쓸 수 있다. | `reaction-targets.php`, 콘텐츠 환경설정/그룹/개별, 커뮤니티 환경설정/게시판 그룹/게시판, 퀴즈·설문 환경설정/개별 설정 |
| 단일 선택 | 한 회원이 같은 대상에 하나의 반응만 남기게 해 반응 의미와 집계가 과하게 부풀지 않게 한다. | target/account unique, 같은 key 재클릭 취소, 다른 key 선택 시 교체 |
| 작성자 본인 제한 | 자기 글, 댓글, 콘텐츠, 퀴즈, 설문에는 신규 반응을 남기지 못하게 하고 알림도 만들지 않아 자기 반응으로 집계를 왜곡하지 않게 한다. | 작성자 본인 write 차단, actor/recipient 동일 알림 no-op |
| 알림 연동 | 다른 회원이 내 글이나 댓글에 반응하면 알림 모듈이 있을 때 사이트 알림으로 이어지고, 알림 모듈이 없어도 반응 저장은 실패하지 않는다. | `reaction`/`target.reacted`, notification no-op, 알림 제외 target 계약 |
| 레코드 점검 | 회원, 대상, 반응 key 기준으로 원장을 조회하고 사용 중지한 key의 기존 기록을 보관, 삭제, 병합 중 선택해 정리한다. | 레코드 필터, target 상태 점검, 사용 중지 key 영향 수, 삭제/병합 감사 로그 |
| 개인정보 연동 | 회원 탈퇴나 익명화 시 해당 계정의 리액션 원장을 삭제해 활동 기록 정리 흐름을 다른 모듈과 맞춘다. | privacy export, privacy cleanup, 계정별 reaction record 삭제 |

## 포인트, 적립금, 예치금

포인트, 적립금, 예치금은 회원 활동에 따른 지급, 차감, 환불, 출금/환불 신청을 기록하는 자산 모듈이다. 숫자를 직접 수정하는 방식이 아니라 원장 거래를 쌓아 잔액을 만든다. 현재 잔액뿐 아니라 지급과 차감의 흐름을 거래 내역에서 확인할 수 있다.

| 기능 | 안내 문구 | 주요 기능 |

| 관리자 잔액 관리 | 회원별 잔액을 목록에서 확인하고 필요한 경우 거래 내역으로 바로 이어갈 수 있어 자산 상태 파악이 빠르다. | 관리자 포인트/적립금/예치금 잔액 목록 |
| 거래 내역 | 모든 증감을 원장 형태로 남겨 현재 잔액이 어떤 거래들로 만들어졌는지 추적할 수 있다. | 관리자 포인트/적립금/예치금 거래 내역 |
| 수동 조정 | 지급, 차감, 차감 원거래 환불, 적립금 회수 같은 예외 처리를 사유와 참조 정보와 함께 남겨 단순 숫자 수정이 아닌 기록 기반 조정을 제공한다. | 관리자 수동 지급과 차감, 원거래 기준 환불, 적립금 회수, 감사 로그 metadata |

| 출금과 환불 신청 | 적립금 출금과 예치금 환불 신청을 회원 화면에서 받고, 관리자 화면에서 완료·반려·일괄처리·원장 차감까지 이어 처리한다. | 적립금 출금 신청, 예치금 환불 신청, 신청 상태 관리, 처리 메모, 일괄처리 |
| 출금 대상 제한 | 적립금 출금 신청 사용 여부와 가능 대상을 전체 회원 또는 회원 그룹으로 지정해 현금성 처리에 필요한 자격 조건을 명확히 둘 수 있다. | 출금 신청 사용 설정, 출금 신청 대상 설정, 전체 회원/그룹 기준 서버 검증 |

| 개인정보 연동 | 탈퇴와 개인정보 사본 제공 시 잔액, 거래, 신청 이력을 함께 다뤄 자산 기록 누락을 줄인다. | 회원 탈퇴 시 자산 처리 계약, privacy export |

## 포인트/금액 환전

환전 모듈은 포인트, 적립금, 예치금처럼 성격이 다른 자산 사이의 이동을 정책으로 열어 주는 기능이다. 허용한 자산 조합과 비율, 수수료, 최소·최대 금액을 기준으로 회원 환전을 제공한다. 회원은 보유 자산을 더 유용한 형태로 바꿀 수 있고, 환전 실행 로그로 입금, 출금, 수수료 흐름을 함께 확인할 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 환전 정책 | 허용한 자산 조합만 환전되도록 정책을 저장해 무분별한 자산 이동을 막고 필요한 교환 경로만 열 수 있다. | 포인트, 적립금, 예치금 간 환전 정책 관리 |
| 환전 조건 | 비율, 반올림, 최소·최대 금액, 수수료를 함께 설정해 자산 성격에 맞는 환전 규칙을 만들 수 있다. | 환전 가능 자산, 환전 비율, 수수료 정책 |
| 회원 환전 | 회원이 보유 자산을 직접 환전할 수 있어 잔액 활용도를 높이고 수동 조정 의존도를 줄인다. | 회원 환전 화면, 회원 환전 실행 |
| 환전 로그 | 실행된 환전 묶음을 로그로 확인해 출금, 입금, 수수료가 같은 처리에서 어떻게 연결됐는지 볼 수 있다. | 관리자 환전 로그 조회 |
| 환전 설정 | 새 정책 등록에 사용할 기본값을 따로 관리해 반복 입력을 줄이고 정책 입력 기준을 일정하게 유지한다. | 환전 환경설정 |
| 정합성 보호 | 양방향 환전으로 가치가 불어나는 조합을 막고 DB 잠금 오류를 재시도해 금액성 처리의 안정성을 높인다. | 무수수료 양방향 정책 순환 가치 증가 방지, DB 잠금 오류 재시도 |
| 정정 기반 | 완료된 환전을 직접 삭제하지 않고 관리자 환전 로그 화면에서 반대 원장 거래로 정정할 수 있어 회계성 기록의 흔적이 남는다. | `/admin/asset-exchange/logs` 완료 환전 묶음 정정 |

## 쿠폰과 이용권

쿠폰과 이용권은 특정 회원이나 회원 그룹에 혜택을 지급하고, 콘텐츠나 커뮤니티 이용 시 자산 차감보다 먼저 적용할 수 있는 장치다. 이벤트 보상, 멤버십 혜택, 유료 콘텐츠 무료 열람권, 게시판 접근권 같은 용도로 쿠폰을 만들 수 있다. 지급, 사용, 환불, 접근권 회수 기능이 함께 제공되어 혜택 운영을 한 화면 흐름 안에서 이어갈 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 쿠폰 종류 | 쿠폰 이름, 상태, 사용처를 관리해 단순 할인권부터 콘텐츠·커뮤니티 접근권까지 같은 틀로 다룰 수 있다. | 쿠폰 종류 관리, 쿠폰 상태 관리, 사용처 설정 |
| 쿠폰 지급 | 개별 회원이나 회원 그룹에 쿠폰을 지급할 수 있어 이벤트, 보상, 등급 혜택을 빠르게 적용할 수 있다. | 개별 회원 지급, 회원 그룹 대상 지급, 지급 사유 기록 |
| 지급 내역 | 누가 어떤 쿠폰을 받았는지 남기고 활성 쿠폰 지급을 취소할 수 있어 지급 오류나 혜택 적용 여부를 나중에 확인하기 쉽다. | 쿠폰 지급 내역 조회, 회원/상태/사용처 필터, 지급 취소 |
| 사용 내역 | 쿠폰이 어디에 사용됐는지 확인하고 필요하면 환불 흐름으로 되돌리며, 쿠폰으로 생긴 접근권까지 함께 회수할 수 있어 접근권 운영이 단단해진다. | 쿠폰 사용 내역 조회, 쿠폰 환불 관리, 접근권 회수, 환불 감사 로그 |
| 회원 쿠폰함 | 회원이 보유 쿠폰과 이용권 상태를 직접 확인해 사용할 수 있는 혜택을 놓치지 않게 한다. | 회원 보유 쿠폰·이용권 조회 |
| 콘텐츠 이용권 | 유료 콘텐츠 접근 전에 쿠폰을 먼저 확인해, 쿠폰이 있으면 자산 차감 없이 열람권을 적용할 수 있다. | 콘텐츠 열람권 연동, 쿠폰 우선 사용 |
| 커뮤니티 이용권 | 유료 게시판이나 게시글 접근에도 쿠폰을 연결해 커뮤니티 전용 이용권 구성을 만들 수 있다. | 커뮤니티 게시판 열람권, 커뮤니티 게시글 열람권 |
| 자산 차감 연결 | 사용할 쿠폰이 없을 때만 포인트·적립금·예치금 차감으로 내려가도록 구성해 혜택과 과금을 함께 운영한다. | 쿠폰 없음 시 포인트/적립금/예치금 차감 |
| 개인정보 연동 | 지급, 사용, 환불 기록이 개인정보 사본 제공과 탈퇴 처리에 포함되어 혜택 기록까지 일관되게 정리된다. | 회원 탈퇴 시 쿠폰 상태 처리, privacy export |

## 사이트 운영 모듈

사이트 운영 모듈은 회원이나 콘텐츠 데이터 자체보다 사이트의 외형과 안내 흐름을 맡는다. 메뉴, 로고, SEO, 배너, 팝업레이어는 자주 바뀌는 영역이므로 코드 수정 없이 관리자 화면에서 조정할 수 있게 분리되어 있다. 메뉴와 로고 구성으로 사이트 구조를 잡고, 캠페인이나 공지가 있을 때 배너와 팝업레이어 노출 위치를 화면에서 설정할 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |

## 알림

알림 모듈은 회원에게 보여 줄 사이트 알림과 이메일 발송 대기열을 관리한다. 자산 거래, 쿠폰 상태, 댓글 멘션, 직접 발송 안내처럼 여러 모듈에서 발생하는 이벤트를 한 흐름으로 모을 수 있다. 이메일은 즉시 성공을 전제로 하지 않고 queue에 남기므로, 외부 발송 서비스 장애가 있어도 원래 업무 처리가 함께 무너지지 않게 한다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 사이트 알림 | 개별 회원 또는 전체 회원에게 전달할 알림을 사이트 안에 남길 수 있어 이메일 없이도 중요한 처리 결과를 계정 영역에서 확인하게 한다. | 사이트 알림 생성, 개별/전체 회원 대상, 관리자 알림 목록 |
| 회원 알림함 | 회원이 자신에게 온 알림을 모아 보고 읽음 처리할 수 있어 신청, 자산, 쿠폰 상태 변화가 흩어지지 않는다. | 공개 상단 알림 드롭다운, 회원 알림 목록, 알림 읽음 처리 |
| 이메일 발송 대기열 | 이메일 발송을 queue 형태로 기록해 즉시 발송 실패가 전체 처리 흐름을 망가뜨리지 않게 한다. | 이메일 delivery queue, 알림 발송 작업 상태 조회 |
| 이메일 발송 방식 | SMTP 또는 HTTP API 방식의 이메일 발송 정보를 저장해 호스팅 환경이나 외부 발송 서비스에 맞춰 전달 채널을 선택한다. | 이메일 채널 사용 설정, SMTP 설정, HTTP API endpoint, Bearer token, timeout |
| 발송 상태 관리 | 외부 발송 작업의 수신자, provider message ID, 오류 메시지, 상태를 조회하고 실패/취소된 작업은 재시도 대기열로 되돌리거나 대기/실패 작업을 취소해 발송 지연을 추적한다. | 발송 작업 필터, queued/sent/failed/canceled 상태, 재시도/취소 전이, provider ID, 오류 메시지 |
| 이벤트 템플릿 | 자산 거래나 쿠폰 상태 변경처럼 반복되는 이벤트 알림을 템플릿으로 관리해 문구와 채널 기준을 통일한다. | 알림 이벤트 템플릿 |
| 알림 설정 | 알림 모듈의 동작 기준을 별도 화면에서 관리해 사이트 알림과 외부 채널 사용 범위를 조정한다. | 알림 설정 관리 |
| 알림 요약 | 전체 알림과 발송 대기 상태를 알림 관리자 화면에서 확인해 전달 지연이나 실패 가능성을 빠르게 감지한다. | 알림 목록, 발송 대기 상태 |
| 개인정보 연동 | 회원 알림 기록을 개인정보 사본 제공에 포함해 계정 관련 통지 이력도 함께 확인할 수 있다. | privacy export 연동 |

## 개인정보 대응

개인정보 모듈은 회원의 사본 제공, 계정 관련 처리 요청, 탈퇴와 익명화 흐름을 운영 기록으로 남기는 영역이다. 산란은 모듈별 export와 cleanup 계약을 사용해 회원, 자산, 쿠폰, 알림, 콘텐츠/커뮤니티 활동 같은 데이터를 한 회원 기준으로 모아 보거나 정리할 수 있게 한다. 회원은 재인증 뒤 자기 개인정보 사본을 직접 내려받고, 정정/삭제/처리 제한처럼 검토가 필요한 요청은 운영자 문의로 접수한 뒤 관리자가 대응 기록으로 남긴다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 회원 안내 | 회원 계정 영역에서 사본 직접 내려받기와 운영자 문의 안내를 제공하되, 기본 설치는 회원 요청 티켓을 만들지 않는다. | `/account/privacy-export`, 개인정보 문의 안내 |
| 관리자 대응 기록 | 운영자 문의로 접수한 건의 요청 유형, 요청자 식별값, 처리 상태와 처리 결과를 남길 수 있어 개인정보 대응 과정이 화면 안에서 기록된다. | 관리자 개인정보 대응 기록 |
| 사본 제공 | 모듈별 export 계약을 모아 회원에게 제공할 개인정보 사본을 구성하므로 확장 모듈 데이터도 함께 포함된다. | 개인정보 사본 제공, `privacy-export.php` 계약 |
| 회원 직접 내보내기 | 회원이 재인증 뒤 자기 개인정보 사본을 직접 내려받을 수 있어 단순 열람 요청을 관리자 처리에만 의존하지 않는다. | `/account/privacy-export`, 비밀번호 재확인, 다운로드 감사 로그 |
| 관리자 재확인 | 처리 자료 다운로드나 민감 요청 처리에는 관리자 비밀번호와 확인 항목을 요구해 개인정보 자료 접근을 한 번 더 통제한다. | 관리자 비밀번호 재확인, 본인확인 체크, export/action 확인 |
| 탈퇴와 정리 | 회원 탈퇴나 익명화 시 각 모듈의 cleanup 흐름을 연결해 계정 삭제 이후에도 필요한 운영 기록과 개인정보를 구분한다. | cleanup 흐름, 회원 탈퇴와 익명화 연결 |
| export action | 대응 기록별 사본 제공 파일을 내려받는 action 기반을 제공해 처리 결과 전달 방식을 일정하게 유지한다. | 개인정보 대응 기록 export action |
| 운영 화면 | 개인정보 대응 기록을 별도 화면에서 확인해 계정, 자산, 커뮤니티 기록처럼 여러 모듈 데이터가 얽힌 요청도 추적할 수 있다. | 개인정보 대응 운영 화면 |

## 본문 임베드와 참조 점검

본문 임베드는 콘텐츠와 커뮤니티 글 안에서 다른 콘텐츠, 게시글, 퀴즈, 설문을 연결하는 기능이다. 작성 중 검색해 넣으면 공개 화면에서는 카드나 링크로 보이고, 내부에는 어떤 대상을 참조했는지 기록이 남는다. 대상이 삭제되거나 비공개로 바뀌어도 글을 자동으로 망가뜨리지 않고 참조 상태를 점검할 수 있게 하므로, 연결 오류를 찾아 고칠 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 대상 검색 삽입 | 콘텐츠와 커뮤니티 작성 화면에서 서로의 공개 항목과 퀴즈·설문을 검색해 본문 안에 연결할 수 있어 서비스 화면 사이의 이동 흐름을 코드 수정 없이 만든다. | 콘텐츠, 커뮤니티 게시글, 퀴즈, 설문 검색 삽입 |

| 서버 검증 | 저장 action이 클라이언트 marker를 그대로 믿지 않고 최종 정화 HTML에 남은 참조를 다시 검증해 대상 모듈, 유형, ID, 표시 문구를 동기화한다. | `embed-manager-targets.php`, ref key 검증, 저장 transaction 내 refs 동기화 |

| 복사 정합성 | 콘텐츠 복사와 커뮤니티 게시판 복사 때 본문 marker ref key를 새로 발급하고 새 소유 대상으로 refs를 복사해 원본과 사본의 참조가 섞이지 않게 한다. | ref key 재발급, 본문 rewrite, refs 복사 |

## CKEditor 플러그인

CKEditor 플러그인은 긴 본문을 더 편하게 작성하기 위한 선택 기능이다. 모든 입력창을 무겁게 만들지 않고, 콘텐츠와 커뮤니티처럼 풍부한 본문 편집이 필요한 textarea에만 붙일 수 있다. 관리자 설정에서 직접 호스팅과 CDN 로딩을 고를 수 있고, 본문 이미지 업로드와 임베드 대상 삽입도 화면 소유 모듈의 저장 흐름에 맞춰 사용할 수 있다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 에디터 적용 | 필요한 textarea에만 CKEditor를 붙일 수 있어 단순 입력 화면은 가볍게 두고 본문 작성 화면은 풍부하게 만든다. | CKEditor 5 선택 플러그인, textarea 기반 에디터 강화 |
| 관리자 설정 | 직접 호스팅과 CDN 방식을 선택할 수 있어 배포 환경, CSP, 외부 의존성 정책에 맞게 로딩 방식을 정한다. | 관리자 설정 화면, 직접 호스팅 방식, CDN 방식 |
| 에셋 로딩 | 저장소에 포함된 에셋 또는 CDN을 사용할 수 있어 인터넷 연결 정책과 파일 배포 방식에 맞춰 선택한다. | CKEditor 에셋 로딩, vendor 배포 파일, CDN 버전 |
| 라이선스 기준 | CKEditor 5 v44 이상 라이선스 키 설정을 관리자 화면에서 관리해 GPL 또는 발급 키 기준을 명확히 남긴다. | 라이선스 키, GPL 키, self-hosting/CDN 구분 |
| 저장 fallback | 에디터가 비활성화되거나 로딩에 실패해도 textarea 제출 흐름이 유지되어 본문 저장 자체가 막히지 않는다. | 성공 시 HTML 저장, 비활성/실패 시 plain textarea 저장 |
| 모듈별 저장 정책 | 콘텐츠와 커뮤니티가 각자 본문 저장과 출력 정책을 소유해 에디터는 입력 경험만 제공하고 도메인 정책을 침범하지 않는다. | 화면 소유 모듈 저장 정책, HTML Purifier 우선 서버 측 sanitizer, 내부 fallback |
| 본문 이미지 업로드 | 콘텐츠, 커뮤니티, 팝업레이어처럼 화면을 소유한 모듈이 업로드 endpoint와 저장소 정리 정책을 넘기고 CKEditor는 adapter 연결만 맡는다. | `data-sr-editor-upload-*`, 모듈별 body-file upload, CSRF, 저장소 key |
| 검색 대상 삽입 | CKEditor 본문에는 임베드 매니저 marker를 포함한 허용 HTML을, 일반 textarea에는 제목과 절대 URL 텍스트를 넣어 복사와 외부 붙여넣기에서도 의미가 남게 한다. | 검색 대상 삽입, 허용 HTML 저장, plain text 저장, 절대 URL |
| Plain URL 자동 링크 | 콘텐츠/커뮤니티 환경설정에서 textarea로 저장된 plain text 본문의 http/https URL을 공개 화면 링크로 변환할지 정한다. | 본문 URL 자동 링크, HTML 본문 제외, nofollow 링크 |

## 보안과 안정성

보안과 안정성 기능은 기본 흐름에서 공통 보호 장치를 사용할 수 있게 한다. 산란은 관리자 권한, CSRF 검증, 출력 escape, HTML sanitizer, rate limit, token hash 저장, 감사 로그, 개인정보 export/cleanup 계약을 여러 기능에 나눠 제공한다. 덕분에 회원 관리, 콘텐츠 저장, 댓글 작성, 자산 조정, 개인정보 대응 같은 업무가 공통 보호 기준 위에서 동작한다.

| 기능 | 안내 문구 | 주요 기능 |
| --- | --- | --- |
| 관리자 보호 | 관리자 action은 로그인, 권한, CSRF 검증을 거쳐야 실행되므로 화면 접근과 저장 권한을 따로 통제할 수 있다. | 관리자 로그인, 권한 검사, POST CSRF 검증 |
| 출력 보호 | 변수와 사용자 입력을 escape helper로 출력할 수 있어 화면 출력 기준을 여러 모듈에서 공통으로 사용할 수 있다. | 출력 escape helper |
| 본문 링크 보호 | 본문 HTML은 허용 태그와 안전한 URL 속성만 보존해 링크 삽입 편의와 출력 안전성을 함께 챙긴다. | HTML sanitizer, 안전한 링크 속성 정규화 |
| 요청 제한 | 로그인, 재인증, 이메일 요청처럼 반복 시도가 문제가 되는 흐름에 rate limit 기반을 적용할 수 있다. | rate limit helper |
| 세션과 토큰 | 세션과 토큰 원문 대신 hash 저장 기준을 제공해 인증 기록과 토큰 처리 방식을 일관되게 유지한다. | 세션 관리, 토큰 hash 저장 |
| 기록 기반 확인 | 관리자 작업과 회원 인증 이력을 남겨 문제가 생겼을 때 “무엇이 언제 일어났는지”를 확인할 수 있다. | 관리자 감사 로그, 회원 인증 로그 |
| 개인정보 계약 | 모듈별 개인정보 export와 cleanup 계약을 통해 새 모듈이 추가돼도 사본 제공과 탈퇴 정리 흐름에 들어올 수 있다. | privacy export 계약, privacy cleanup 계약 |
| 기준 문서 | 배포, 보안, DB 접근, 릴리스, 스모크 테스트 기준을 문서로 남겨 코드 변경과 운영 절차가 따로 놀지 않게 한다. | DB 접근 정책, 보안 모델, 보안 체크리스트, 배포 보호 기준 |
