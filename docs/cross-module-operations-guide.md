# 모듈을 가로지르는 운영 흐름 가이드

이 문서는 여러 모듈에 흩어진 운영 맥락을 한 흐름으로 이어야 할 때 따르는 기준이다.

산란의 모듈 경계와 계약은 개발 구조를 지키기 위한 수단이다. 실제 사용자와 운영자가 크게 체감하는 가치는 그 경계 자체보다, 사이트 상태를 한눈에 이해하고 필요한 조치로 자연스럽게 이어지는 경험에서 나온다.

따라서 대부분의 실용적인 관리자 기능에서는 여러 모듈의 상태, 설정, 경고, 바로가기를 한 화면이나 한 작업 흐름 안에 배치해야 하는 필요가 생긴다. 이것은 예외가 아니라 모듈형 시스템을 운영할 때 자연스럽게 생기는 요구다.

목표는 그 필요를 막는 것이 아니다. 목표는 여러 모듈을 가로지르는 운영 흐름을 허용하되, 코어와 admin 모듈이 각 도메인의 소유권과 정책을 흡수하지 않게 만드는 것이다.

## 기본 원칙

### 경계를 숨기지 말고 번역한다

운영자에게 모듈 경계를 완전히 숨기려 하지 않는다. 대신 이 정보가 어느 모듈에서 왔고, 자세한 관리는 어디에서 하는지 명확히 보여준다.

좋은 방향:

- 회원 모듈이 제공한 회원 운영 상태
- 커뮤니티 모듈이 제공한 신고 대기 상태
- 콘텐츠 모듈이 제공한 레이아웃 설정 상태
- 배너 모듈이 제공한 노출 기간 경고

나쁜 방향:

- 코어가 회원, 커뮤니티, 콘텐츠 설정을 직접 해석해 하나의 통합 설정 테이블로 저장
- admin 모듈이 각 도메인 테이블을 직접 읽고 상태 의미를 추론
- 모듈이 공개하지 않은 내부 설정을 통합 화면에서 임의로 노출

### 코어와 admin은 조립자이고 모듈은 소유자다

코어와 admin 모듈은 운영자 화면을 조립할 수 있다. 하지만 도메인 설정, 상태 판단, 저장 정책은 각 모듈이 소유한다.

코어 또는 admin 모듈이 해도 되는 일:

- 설치된 모듈 목록 조회
- 모듈이 명시적으로 공개한 운영 요약 계약 읽기
- 관리자 권한, CSRF, POST/Redirect/Get, flash 처리
- 공통 관리자 UI shell 제공
- 모듈별 패널, 바로가기, 조치 항목을 같은 화면에 배치
- 여러 모듈을 가로지르는 운영 checklist 구성

코어 또는 admin 모듈이 하면 안 되는 일:

- 모듈 테이블을 직접 읽어 도메인 상태 추론
- 모듈 설정값의 의미를 코어에서 판단
- 모듈별 저장 정책을 코어나 admin helper에 내장
- 모든 설정을 하나의 코어 테이블로 흡수
- 기존 모듈 관리자 화면을 대체하는 통합 CRUD 제공

### 자동 추론보다 명시적 계약을 쓴다

설치된 테이블이나 설정 key를 뒤져서 운영 UI를 자동 생성하지 않는다. 각 모듈이 운영자에게 보여줘도 되는 정보와 빠르게 수정해도 되는 값을 명시적으로 제공한다.

예시 계약 파일:

```text
modules/{module_key}/admin-ops.php
modules/{module_key}/admin/ops.php
```

실제 파일명은 구현 시 하나로 정하고, [모듈 작성 가이드](module-guide.md)의 known contract 목록에 shape와 호출 책임을 기록한다. 이 계약은 route registration API가 아니라, 관리자 운영 화면이 설치된 모듈의 명시적 파일을 확인하고 검증한 뒤 include하는 읽기 가능한 계약 파일이어야 한다.

### 읽기와 쓰기를 분리한다

운영 상태 요약은 비교적 넓게 모아 볼 수 있다. 하지만 저장은 좁고 명시적이어야 한다.

- 읽기: 모듈이 제공한 summary, status, action, warning을 조립해 표시한다.
- 쓰기: 모듈이 명시한 field 또는 모듈 소유 save endpoint만 허용한다.

코어가 임의의 `key=value` payload를 받아 저장하는 generic settings editor가 되면 안 된다. 빠른 편집은 모듈이 공개한 항목에만 적용하고, 서버 저장 검증은 해당 모듈이 다시 수행한다.

### 빠른 편집은 전체 설정 관리가 아니다

빠른 편집은 운영자가 자주 확인하는 좁은 항목에만 적용한다.

적합한 예:

- 회원 가입 허용 여부
- 커뮤니티 비회원 글쓰기 허용 여부
- 신고 자동 숨김 사용 여부
- 콘텐츠 기본 레이아웃 선택

주의가 필요한 예:

- 권한 정책 전체 편집
- 결제 정책 변경
- 데이터 삭제 정책 변경
- 도메인별 복잡한 workflow 설정

복잡하거나 위험한 설정은 원래 모듈 관리자 화면으로 이동시킨다.

### 통합 흐름은 허용하되 통합 소유권은 만들지 않는다

운영 화면은 여러 모듈의 정보를 한 흐름으로 엮을 수 있다. 다만 그 흐름이 새로운 도메인 소유권을 코어에 만들면 안 된다.

예를 들어 사이트 오픈 전 점검 화면은 회원 가입, 기본 레이아웃, 게시판 공개 상태, 이메일 발송 설정을 함께 보여줄 수 있다. 하지만 각 항목의 판단과 저장은 해당 모듈의 계약 또는 관리자 endpoint를 통해 이뤄져야 한다.

## 운영 보조 계약 예시

읽기 전용 운영 요약은 다음처럼 모듈이 이미 판단한 상태를 반환한다.

```php
<?php

return [
    'title' => '회원',
    'summary' => [
        'status' => 'enabled',
        'items' => [
            [
                'label' => '가입',
                'value' => '허용',
                'severity' => 'normal',
            ],
            [
                'label' => '이메일 인증',
                'value' => '비활성',
                'severity' => 'warning',
                'message' => '스팸 가입이 많다면 활성화를 권장합니다.',
            ],
        ],
    ],
    'actions' => [
        [
            'label' => '회원 설정',
            'href' => '/admin/member/settings.php',
        ],
    ],
];
```

빠른 편집이 필요하면 읽기 요약과 별도 shape로 분리한다.

```php
'fields' => [
    [
        'key' => 'allow_registration',
        'label' => '가입 허용',
        'type' => 'boolean',
        'value' => true,
        'required' => true,
    ],
]
```

저장은 해당 모듈의 handler 또는 endpoint로 위임한다. 운영 보조 화면이 POST를 받아 공통 처리를 하더라도, 저장 가능한 field allowlist, 값 정규화, 도메인 검증, 감사 로그 payload 구성은 소유 모듈의 helper나 action이 책임져야 한다.

## 허용되는 화면 형태

허용되는 화면은 통합 관리자가 아니라 운영 보조판이다.

가질 수 있는 요소:

- 설치된 모듈의 운영 상태 목록
- 조치 필요 항목 모음
- 모듈별 주요 설정 요약
- 좁은 quick edit
- 기존 관리자 화면 바로가기
- 각 정보의 제공 모듈 표시
- 여러 모듈을 가로지르는 운영 checklist

위험한 요소:

- 모든 모듈 설정을 한 form에서 저장
- 코어가 모듈별 DB schema를 직접 해석
- 도메인 정책을 코어에서 결정
- 기존 모듈 관리자 화면을 대체하는 통합 CRUD
- 모듈이 공개하지 않은 내부 설정 노출

## 판단 기준

운영 편의를 위해 모듈 경계를 넘는 기능이 필요해지면 다음 질문을 먼저 한다.

1. 이 정보는 어느 모듈이 소유하는가?
2. 코어나 admin 모듈이 이 값의 의미를 알아야 하는가, 아니면 표시만 하면 되는가?
3. 저장이 필요한가, 아니면 상세 화면 링크로 충분한가?
4. 이 설정은 quick edit으로 다뤄도 안전한가?
5. 모듈이 명시적으로 공개한 계약으로 구현할 수 있는가?
6. 이 기능이 기존 모듈 관리자 화면을 대체하려고 하지는 않는가?
7. 여러 모듈을 한 흐름에 배치하더라도 각 모듈의 소유권이 드러나는가?

답이 불명확하면 코어에 흡수하지 말고 모듈 계약을 먼저 좁게 정의한다.

## 문서화 기준

모듈을 가로지르는 운영 흐름을 새로 만들거나 운영 보조 계약을 추가하면 같은 작업에서 관련 문서를 갱신한다.

- 새 계약 파일과 반환 shape는 [모듈 작성 가이드](module-guide.md)에 기록한다.
- 관리자 화면 구성과 quick edit UX는 [관리자 UI 작성 기준](admin-ui-guide.md)에 기록한다.
- 코어와 admin 모듈의 책임 경계가 달라지면 [핵심 설계 결정](core-decisions.md)에 반영한다.
- 실제 구현 상태와 검증 결과는 필요한 경우 `docs/records/` 또는 관련 검증 문서에 남긴다.

## 결론

모듈형 시스템에서 운영자가 여러 모듈을 한 흐름으로 다루고 싶어 하는 필요는 대부분 생긴다. 이를 예외로 보거나 막으려 하기보다, 처음부터 인정하고 안전한 운영 보조 계약을 마련하는 편이 좋다.

좋은 운영 보조 기능은 모듈 경계를 부수는 통합 관리자가 아니라, 모듈이 자기 소유권을 유지한 채 운영자에게 필요한 상태와 조치를 제공하도록 조립하는 얇은 운영 계층이다.
