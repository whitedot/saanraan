# 보상 부정 수급 공통 기준 초안

## 범위

퀴즈와 설문처럼 회원 행동에 따라 보상을 자동 지급하는 모듈은 지급 요청, 중복 방지, 재시도, 공급자 재검증 기준을 같은 용어로 기록한다.

## 공통 용어

- `reward_provider`: 실제 보상을 처리하는 공급자. 현재 기준 값은 `ledger_asset`, `coupon`이다.
- `reward_module`: 공급자 안의 세부 모듈 또는 자산 key. `ledger_asset`에서는 `point`, `reward`, `deposit` 같은 회원 자산 모듈 key를 사용한다.
- `reward_code`: 공급자별 세부 보상 식별자. 쿠폰은 쿠폰 정의 ID 문자열을 사용한다.
- `dedupe_scope`: 지급 중복 방지 범위. 기본 값은 `per_quiz` 또는 `per_survey`처럼 대상 단위 1회이며, 출처가 보상 의미에 포함될 때만 `per_source`, 매 제출이 독립 보상일 때만 `per_attempt` 또는 `per_response`를 쓴다.
- `dedupe_key`: 회원, 보상 대상, 정책, 공급자, 모듈, scope별 대상 식별자를 포함한 불변 문자열이다.
- `grant`: 보상 지급 시도와 결과를 담는 모듈 소유 로그 행이다.

## Dedupe 기준

1. `dedupe_key`에는 회원 ID, 대상 ID, 보상 정책 ID, `reward_provider`, `reward_module`을 포함한다.
2. `per_source`는 source module/type/id가 모두 검증된 경우에만 사용한다.
3. 기존 `granted` 행이 있으면 같은 dedupe key의 후속 제출은 새 보상을 지급하지 않는다.
4. 기존 `pending` 또는 `failed` 행은 공급자와 정책을 다시 검증한 뒤 같은 로그 행에서 재시도할 수 있다.
5. 원본 로그 행을 중복 행으로 바꾸지 않는다. 중복 제출은 현재 처리 결과에서만 중복으로 표시한다.

## Retry 기준

재시도는 모듈 소유 grant 로그가 `pending` 또는 `failed`일 때만 허용한다. 재시도 전에는 보상 정책, 자산 거래 함수, 쿠폰 정의 활성 상태, 회원 상태를 다시 확인한다. 성공하면 provider reference와 result snapshot을 기록하고, 실패하면 민감정보를 제거한 오류 메시지와 실패 시각을 기록한다.

## Provider 재검증 기준

`ledger_asset`은 활성 회원 자산 계약과 거래 함수 존재 여부를 실행 직전에 확인한다. `coupon`은 쿠폰 모듈 활성화, 쿠폰 테이블 가용성, 쿠폰 정의의 활성/유효 기간을 실행 직전에 확인한다. 공급자를 알 수 없거나 재검증에 실패하면 지급하지 않고 실패 로그를 남긴다.

## 모듈 적용

퀴즈는 `sr_quiz_reward_grants`를 기준 로그로 사용한다. 설문은 `sr_survey_reward_grants`를 기준 로그로 사용한다.

쿠폰은 `sr_coupon_redemptions.dedupe_key`와 쿠폰 사용 환불 시 소비자 접근권 회수 계약으로 중복 사용과 환불 후 접근 유지를 막는다.

포인트, 적립금, 예치금은 각 거래 원장에 `reference_type`과 `reference_id`를 저장하고, 환불/회수/출금성 처리는 원거래 또는 신청 행을 참조한다. 포인트 환불은 원 차감 거래를 잠그고 이미 환불된 누적 수량을 빼서 남은 범위를 넘는 환불을 거부한다. 적립금과 예치금의 출금/환불 신청은 대기 신청액과 회원 그룹 조건을 처리 직전에 다시 확인한다.

자산 환전은 `exchange_group_id`를 기준으로 환전 묶음을 잠그고, 같은 환전 요청이 중복 실행되지 않게 한다. 출금, 입금, 수수료 원장은 같은 참조 묶음으로 연결한다.

콘텐츠와 커뮤니티는 접근권, 자산 차감 로그, 쿠폰 접근 이력을 분리해 저장하고 `dedupe_key`로 같은 대상의 중복 차감과 중복 접근권 부여를 막는다. 유료 다운로드와 첨부 다운로드는 파일 전달 전에 자산 처리를 먼저 완료하고, 환불 시 접근권 회수 helper를 호출한다.
