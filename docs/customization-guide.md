# 커스터마이징과 업데이트 충돌 가이드

이 문서는 배포판에 포함된 코어와 기본 모듈을 운영 환경에서 커스터마이징할 때 업데이트 병합 비용이 어디에서 생기는지 설명한다. 산란은 배포판 파일을 직접 수정하는 운영 방식을 막지 않는다. 다만 수정 위치와 방식에 따라 이후 배포판 업데이트 때 같은 파일, 같은 줄, 같은 테이블 구조를 병합해야 할 가능성이 달라진다.

## 기본 원칙

배포판이 소유하는 테이블, 컬럼, helper, view는 배포판 업데이트의 기준선이다. 사이트별 커스터마이징은 가능하지만, 배포판 파일이나 공식 테이블을 직접 고치면 다음 업데이트에서 같은 이름의 컬럼, 저장 helper 변경, view 교체, update SQL과 충돌할 수 있다.

커스터마이징이 DB를 필요로 하면 기존 공식 테이블을 직접 넓히는 방식과 별도 모듈 또는 별도 확장 테이블로 분리하는 방식의 업데이트 비용을 먼저 비교한다. 공식 테이블에 컬럼을 더하면 구현은 빠를 수 있지만, 공식 update SQL과 이름이 겹치거나 공식 저장 helper가 모르는 필수 컬럼 때문에 등록/수정 흐름이 깨질 수 있다. 별도 테이블이나 별도 모듈은 초기 작업이 늘 수 있지만, 배포판 업데이트와 충돌하는 지점을 줄인다.

## 변경 방식별 업데이트 비용

업데이트 충돌은 금지 목록으로 관리하기보다 변경 방식별 비용으로 판단한다.

충돌 가능성이 낮은 편인 변경:

- 관리자 설정값, 모듈 설정값, 테마 설정값 변경
- 별도 layout/theme/skin 추가
- 별도 모듈이나 플러그인 추가
- 배포판 테이블과 안정 식별자로 연결되는 별도 확장 테이블 추가
- 기존 계약 파일 구조에 맞춘 사이트 전용 계약 파일 제공
- 화면 소유 모듈이 이미 제공하는 출력 슬롯이나 선택지 계약 사용

충돌 가능성이 중간인 변경:

- 기본 모듈 view 파일 직접 수정
- 관리자 목록 컬럼을 기본 view에 직접 추가
- 기본 CSS, 언어 파일, theme/skin 파일 직접 수정
- 공식 helper의 반환값을 전제로 한 사이트 전용 layout 작성

충돌 가능성이 높은 편인 변경:

- `core/*` helper, 설치/update 흐름, 요청 흐름 직접 수정
- 기본 모듈 action이나 저장 helper 직접 수정
- `sr_content_items`, `sr_community_posts`, `sr_member_accounts`처럼 공식 모듈이 소유하는 테이블에 사이트 전용 필수 컬럼 추가
- 배포판 update SQL 파일을 운영 사이트에 맞게 수정
- 권한, CSRF, 인증, 개인정보 cleanup/export 흐름 직접 변경

충돌 가능성이 높다는 말은 해당 변경이 잘못이라는 뜻이 아니다. 같은 영역을 배포판 업데이트도 자주 바꿀 수 있으므로, 변경 이유와 수정 파일, 관련 SQL, 수동 병합 절차를 기록해두면 업데이트 비용을 줄일 수 있다는 뜻이다.

## 분리 구조

사이트 전용 기능은 다음 중 하나로 분리할 수 있다.

- 별도 모듈: 관리자 화면, public route, 권한, 설치/update SQL이 필요한 경우.
- 플러그인: 기존 모듈의 계약 파일, 출력 슬롯, 에디터, 결제 수단처럼 특정 확장점에 붙는 경우.
- 별도 확장 테이블: 기존 공식 레코드에 사이트 전용 표시값만 붙이고, 별도 화면이나 helper가 그 값을 읽는 경우.
- 별도 layout/theme/skin: 표시 구조만 바꾸고 공식 저장 정책을 그대로 쓰는 경우.

테이블 이름은 프로젝트 prefix와 소유 영역을 드러내야 한다.

```text
sr_site_content_main_sections
sr_custom_content_cards
sr_clientname_content_layout_settings
```

공식 레코드와 연결할 때는 안정 식별자를 사용한다.

```text
content_id
content_group_id
board_id
post_id
account_id
```

## 계약 기반 운영 조합

장기 운영과 업데이트 병합을 고려하면, 배포판 파일을 고치더라도 화면 소유 모듈에는 작고 반복 가능한 계약 읽기 지점만 두고 실제 도메인 요약, 정책, 쿼리는 제공 모듈의 계약 파일에 두는 방식이 병합 비용을 줄인다.

기본 구조:

```text
소비 모듈은 자신이 읽을 수 있는 계약 구조를 문서화한다.
제공 모듈은 module.php의 contracts.provides에 해당 계약 파일을 선언한다.
실행 시점에는 소비 모듈이 활성 모듈 중 해당 계약 파일을 제공하는 모듈만 읽는다.
```

예를 들어 회원 목록에 커뮤니티 글 수, 콘텐츠 발행 수, 콘텐츠 조회수를 표시하려면 회원 모듈이 커뮤니티와 콘텐츠 테이블을 직접 조회하는 대신 `admin-member-summary.php` 같은 계약을 읽는 흐름을 둘 수 있다.

```text
member 모듈
- 회원 목록 화면을 소유한다.
- 현재 페이지의 account_id[]를 확보한다.
- 활성 모듈의 admin-member-summary.php 계약을 수집한다.
- 반환된 요약값을 회원 목록이나 회원 상세 패널에 표시한다.

community/content 모듈
- admin-member-summary.php를 provides에 선언한다.
- 자기 테이블을 배치 조회해 account_id별 요약값을 반환한다.
- 라벨, 표시값, 관리자 이동 URL, 권한 판단을 책임진다.
```

이 방식은 회원 모듈이 특정 서비스 모듈을 필수 의존하지 않게 하고, 서비스 모듈의 테이블 구조가 바뀔 때 수정 지점이 해당 서비스 모듈 안에 남게 한다. 회원 목록 view를 직접 고쳐 도메인별 SQL을 붙이는 방식보다 초기 구조는 조금 길 수 있지만, 배포판 업데이트 때 병합해야 할 지점은 작아진다.

목록형 계약은 행마다 호출하지 않고 배치 조회를 기본으로 둔다.

```php
batch_function(PDO $pdo, array $accountIds, array $context): array
```

계약 파일이 많아질 때의 주요 성능 비용은 파일 존재 확인이나 include보다 계약 함수가 실행하는 쿼리에서 생긴다. 회원 30명 목록에서 모듈별로 30회씩 쿼리하지 않고, 모듈별 `batch_function` 1회가 `account_id[]` 전체를 처리하게 한다. 계약 파일 탐색 결과는 요청 단위 메모리 캐시로 재사용하고, 복잡한 파일 캐시는 운영 환경의 `storage/cache` 접근 차단 기준을 확인한 뒤 별도 판단한다.

읽기 요약 계약은 관리자 화면을 보강하는 기능이므로 계약 오류가 나면 해당 확장만 제외하거나 오류 상태를 표시하고 기본 목록은 계속 렌더링하는 편이 운영에 유리하다. 삭제, 비활성화, key 변경처럼 위험한 작업 전 영향 확인에 쓰는 읽기 참조 계약은 별도 성격의 계약이며, 그런 POST에서는 계약 오류를 차단 사유로 다룰 수 있다.

## 콘텐츠 메인 화면 예시

콘텐츠 메인 페이지에 추천 문구, 히어로 이미지, 카드 섹션, 운영자가 고른 콘텐츠 묶음을 표시하고 싶다면 공식 `sr_content_items`나 `sr_content_groups`에 컬럼을 추가하는 방식과 별도 사이트 전용 테이블을 두는 방식을 비교한다. 별도 테이블을 쓰면 배포판 콘텐츠 모듈 업데이트와 충돌하는 지점을 줄일 수 있다.

분리 흐름:

1. 사이트 전용 테이블을 만든다.
2. 필요한 경우 사이트 전용 관리자 화면에서 값을 저장한다.
3. 배포판 `content.basic` 파일을 직접 수정하거나 별도 콘텐츠 layout/theme/skin 후보를 만들 때의 병합 비용을 비교한다.
4. 새 layout/theme/skin은 공식 콘텐츠 helper로 배포판 데이터를 읽고, 사이트 전용 helper로 확장 테이블 데이터를 함께 읽는다.
5. 콘텐츠 삭제, 그룹 삭제, 복사, sitemap, 개인정보 처리에 확장 데이터가 영향을 주는지 별도로 판단한다.

예시 테이블:

```sql
CREATE TABLE IF NOT EXISTS sr_site_content_main_sections (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    section_key VARCHAR(60) NOT NULL,
    title VARCHAR(160) NOT NULL,
    description TEXT NULL,
    linked_content_id BIGINT UNSIGNED NULL,
    image_storage_key VARCHAR(255) NOT NULL DEFAULT '',
    status VARCHAR(30) NOT NULL DEFAULT 'enabled',
    sort_order INT NOT NULL DEFAULT 0,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    PRIMARY KEY (id),
    UNIQUE KEY uq_sr_site_content_main_sections_key (section_key),
    KEY idx_sr_site_content_main_sections_status_sort (status, sort_order, id),
    KEY idx_sr_site_content_main_sections_content (linked_content_id)
);
```

이 테이블은 배포판 콘텐츠 모듈이 아니라 사이트 전용 기능이 소유한다. 배포판 콘텐츠 업데이트는 이 테이블을 설치하거나 수정하거나 정리하지 않으므로, 사이트 전용 기능은 자기 update SQL과 검증 절차를 따로 유지한다.

## 레이아웃과 DB 책임

레이아웃은 기본적으로 표시 책임만 가진다. `<head>`, 공통 header/footer, 메뉴 위치, 전체 폭 같은 문서 골격은 public layout이 맡고, 콘텐츠 목록/상세 같은 도메인 표시는 콘텐츠 모듈의 theme/skin 또는 별도 확장 모듈이 맡는다. 콘텐츠/커뮤니티/퀴즈/설문 환경설정에서 교차 모듈 레이아웃을 선택하려면 해당 layout option이 대상 모듈의 필수 화면 target 전체를 명시적으로 지원해야 한다.

레이아웃 때문에 DB 테이블이나 컬럼이 필요해지는 순간, 그 기능은 단순 레이아웃이 아니라 저장 정책을 가진 확장 기능이다. 이때는 저장 helper, update SQL, 삭제/복사/권한/개인정보 정책을 함께 둘 위치를 정한다. 배포판 모듈 파일에 직접 넣으면 구현 흐름은 짧지만 업데이트 병합 대상이 되고, 별도 모듈 또는 명시적인 확장 테이블로 분리하면 관리 파일은 늘지만 배포판과의 충돌 가능성은 줄어든다.

## 업데이트 영향

배포판 업데이트는 공식 테이블과 공식 파일을 기준으로 동작한다. 사이트 커스터마이징이 공식 테이블을 직접 변경하면 다음 문제가 생길 수 있다.

- 공식 update SQL이 같은 컬럼명을 추가하면서 충돌한다.
- 공식 저장 helper가 모르는 `NOT NULL` 컬럼 때문에 등록/수정이 실패한다.
- 콘텐츠 복사, 삭제, 리비전 생성, 개인정보 export/cleanup에서 사이트 전용 데이터가 누락된다.
- 배포판 view 교체 후 직접 수정한 화면 코드가 사라진다.
- 공식 helper의 반환 필드나 출력 순서가 바뀌어 커스텀 layout이 깨진다.

별도 확장 테이블을 쓰면 배포판 업데이트는 공식 스키마만 갱신하고, 사이트 전용 기능은 자기 update SQL과 검증 절차로 따로 관리할 수 있다. 공식 테이블을 직접 바꾸는 선택을 했다면 업데이트 전에 배포판 update SQL, 저장 helper, 삭제/복사/개인정보 흐름을 함께 확인한다.

## 삭제, 복사, 개인정보 처리

확장 데이터가 공식 레코드와 연결되면 다음 정책을 반드시 정한다.

- 공식 레코드가 삭제될 때 확장 데이터도 삭제할지, 보관할지.
- 공식 레코드를 복사할 때 확장 데이터도 복사할지, 새 기본값을 쓸지.
- 회원 계정과 연결되는 데이터가 개인정보 사본 제공에 포함되는지.
- 회원 탈퇴/익명화 시 account 연결을 제거해야 하는지.
- 저장소 파일을 쓰면 삭제 실패 기록과 재시도 흐름을 둘지.

이 정책이 없으면 화면은 동작해도 운영 중 고아 데이터나 개인정보 처리 누락이 생긴다.

## 구현 체크리스트

- [ ] 변경한 배포판 파일과 변경 이유를 기록했는가.
- [ ] 다음 배포판 업데이트 때 같은 파일이나 같은 테이블 구조를 병합해야 할 가능성을 확인했는가.
- [ ] 커스터마이징 데이터가 공식 모듈 테이블에 직접 들어가는지, 별도 테이블로 분리되는지 선택 이유를 남겼는가.
- [ ] 별도 테이블을 쓴다면 이름에 소유 영역과 site/module prefix가 드러나는가.
- [ ] 공식 레코드와 안정 식별자로 연결하는가.
- [ ] 별도 설치/update SQL이 있고, 반복 실행되어도 안전한가.
- [ ] 삭제, 복사, 개인정보 export/cleanup, 저장소 정리 정책을 정했는가.
- [ ] 관리자 목록이나 상세 화면을 확장할 때 화면 소유 모듈에 도메인별 SQL을 직접 붙이는 방식과 계약 파일로 분리하는 방식의 병합 비용을 비교했는가.
- [ ] 목록형 계약이나 요약 조회는 행별 호출이 아니라 배치 조회로 처리하는가.
- [ ] 배포판 view 파일을 직접 수정한다면 업데이트 병합 절차를 기록했고, 분리한다면 별도 layout/theme/skin 또는 출력 슬롯을 쓰는가.
- [ ] 배포판 업데이트 후 `/admin/updates`와 관련 public/admin 화면을 스모크 테스트하는가.
